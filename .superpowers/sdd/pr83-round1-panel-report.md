## Findings, what/where/test evidence

### 1. Uptime i18n (CodeRabbit + Codex dupe)

- **Where**: `panel/src/features/equipment/AgentCard.tsx` — `formatUptime`.
- **What**: `formatUptime` built the raw string `` `uptime ${hours} h ${minutes} m` `` itself, never going through `react-i18next` — a RU-locale operator saw an English fragment mid-meta-line. Changed the function to `formatUptime(t: TFunction, uptimeSeconds: number)`, returning `t("equipmentUptime", { hours, minutes })`; the caller now passes its own `t` from `useTranslation()`. Added `equipmentUptime` to both `en.json` (`"uptime {{hours}} h {{minutes}} m"`, byte-identical to the old hardcoded string) and `ru.json` (`"работает {{hours}} ч {{minutes}} мин"`).
- **Test evidence**: `AgentCard.test.tsx` — new test switches `i18n` to `"ru"` (via `i18n.changeLanguage`, same pattern as `LanguageSwitcher.test.tsx`) and asserts the meta line reads `/работает 3 ч 12 мин/` with no `/uptime/` fragment surviving. Ran red first (failed: RU render still showed `"uptime 3 h 12 m"`), then green after the fix. `keyParity.test.ts` re-run to confirm EN/RU key parity.

### 2. Manual port validation (CodeRabbit + Codex dupe)

- **Where**: `panel/src/features/equipment/PrinterWizard.tsx` — `handleManualSubmit` + the manual-add form's Save button.
- **What**: The guard used `Number.isFinite(port)`, which lets an emptied field (`""` → `0` via `Number("")`) and an out-of-range port (e.g. `70000`) straight through to `POST /printers/add`. Added `manualPortValid = Number.isInteger(manualPortNumber) && manualPortNumber >= 1 && manualPortNumber <= 65535` (mirrors the Save-step address form's existing `addressValid` rule exactly), used it both in the in-handler defense-in-depth check and to disable the Save button proactively.
- **Test evidence**: `PrinterWizard.test.tsx` — 3 new tests: port `70000` → Save disabled, no POST; emptied port → Save disabled, no POST; port `9100` → Save enabled, POST fires with the correct numeric port. Ran red first (both rejection tests failed: button wasn't disabled), green after.

### 3. No-machine-id affordances

- **Where**: `panel/src/features/equipment/DeviceCard.tsx` (new `disableActions` prop) + `panel/src/features/equipment/EquipmentPage.tsx` (wires it, plus disables the header "+ Add device" trigger).
- **What**: A legacy agent (`connected_legacy`) with no cached identity has `machineId === null`; `openCreateWizard`/`openCreateScannerWizard`/`onSaveUnsaved`'s handlers already silently no-op in that state, but the Set up (header + empty-state) and Save… buttons rendered fully enabled — looking actionable while doing nothing. Added `disableActions?: boolean` to `DeviceCardProps`, applied to all three buttons; `EquipmentPage` passes `disableActions={machineId === null}` to both `DeviceCard` instances and disables the "+ Add device" `Button` trigger the same way. The WHY is already visible via `AgentCard`'s existing `equipmentAgentLegacyHint` — no new copy needed.
- **Test evidence**: `EquipmentPage.test.tsx` — 2 new tests in the `legacy agent (connected_legacy)` describe: no cached identity → Save…/Set up (both columns)/+ Add device all disabled; cached identity → all enabled. Ran red first (all assertions failed: nothing was disabled), green after.

### 4. Readiness invalidation for equipment mutations

- **Where**: `panel/src/features/equipment/hooks.ts` — `useCreateDevice`, `useDeleteDevice`, `useSetDefaultPrinter`, `useMarkTestPassed`.
- **What**: Equipment is an org-level resource with no `eventId`, but its readiness-gated content (device presence / a tested default printer) feeds the backend's EQUIPMENT readiness step for **every** event under the tenant. `READINESS_KEY(eventId)` (`events/hooks.ts`) needs a concrete `eventId` this file doesn't have, so added a local `ALL_EVENTS_READINESS_KEY = ["get", "/api/events/{id}/readiness"] as const` — the `[method, path-template]` **prefix** idiom `ATTENDEES_LIST_KEY` documents (`attendees/hooks.ts`): a queryKey with no third `init` element partial-matches every registered readiness query regardless of which event populated it. Added `queryClient.invalidateQueries({ queryKey: ALL_EVENTS_READINESS_KEY })` to the four readiness-affecting mutations; deliberately **not** added to `usePatchDevice` (rename) or `useUpsertMachine` (reconcile) since neither changes anything the readiness step reads.
- **Test evidence**: `hooks.test.tsx` — new `describe("readiness invalidation …")` mounts TWO genuinely-subscribed `useEventReadiness` observers (`evt-1`, `evt-2`, same `ReadinessObserver` discipline as `AddAttendeeDialog.test.tsx`) sharing one `QueryClient`, and asserts both refetch (readinessGetCount 2→4) after each of the four mutations, plus a fifth test proving `usePatchDevice` does **not** cause a refetch. Ran red first (all four "invalidates every event's readiness query" tests failed: count stayed at 2), green after.

### 5. Retest busy gate (ScannerWizard)

- **Where**: `panel/src/features/equipment/ScannerWizard.tsx` — `handleOpenChange`, `preventDialogDismiss`, the retest footer's Close button.
- **What**: Retest mode's auto-fired `markTestPassed.mutate(...)` (on a confirmed detection) sat outside every busy/dismissal check. Since closing bumps `sessionRef` (the open-keyed reset effect), a late failure's `onError` saw a stale session and silently dropped `testPassedWarning` — the operator never learned the stamp didn't take, and Close stayed clickable mid-flight regardless. Added `markTestPassed.isPending` to both `handleOpenChange`'s and `preventDialogDismiss`'s busy checks (alongside the existing `saving`/`comAdding`), and disabled the retest-mode Close button on the same condition. The existing `testPassedWarning` UI (`equipmentWizardScanTestWarn`) now actually gets to render because the session survives long enough.
- **Test evidence**: `ScannerWizard.test.tsx` — 2 new tests in `retest mode`: Close disabled while `POST /test-passed` is in flight (click is a no-op, `onClose` never called, dialog stays open until settled); a failure (500) surfaces the in-dialog warning even after an attempted close. Ran red first (Close wasn't disabled / `onClose` fired), green after.

### 6. Save inert during test send (PrinterWizard)

- **Where**: `panel/src/features/equipment/PrinterWizard.tsx` — the shared Test/Save footer's Save button.
- **What**: The footer's `disabled` expression checked `saving` but not `printing` — if the operator raced "Yes, looks right" → "Save printer" while the auto-fired test print was still in flight, Save could create the registry row and close the dialog with the physical send unresolved (a late print failure would then be invisible). Added `printing` to the disabled expression.
- **Test evidence**: `PrinterWizard.test.tsx` — new test with `printDelayMs = 120`: after "Yes, looks right" advances to Save while the print is still pending, "Save printer" is disabled; re-enables once the send settles. Ran red first (button wasn't disabled), green after.

### 7. Cancel vs comAdding (ScannerWizard)

- **Where**: `panel/src/features/equipment/ScannerWizard.tsx` — the create-mode footer's explicit Cancel button.
- **What**: `handleOpenChange`/`preventDialogDismiss` already gated on `comAdding`, but the explicit Cancel button checked only `saving` — clicking it during a slow `POST /scanners/add` closed the dialog with no registry row and no warning that the agent-side port might already be open. Added `comAdding` to the button's `disabled` expression.
- **Test evidence**: `ScannerWizard.test.tsx` — new test delays `POST /scanners/add` by 120ms: Cancel is disabled while in flight, clicking it is a no-op (`onClose` not called), and the wizard proceeds into the listen phase once the add settles. Ran red first, green after.

### 8. Precedence pending window (useAgentPrinters)

- **Where**: `panel/src/shared/agent/useAgentPrinters.ts`.
- **What**: For a modern agent (`machineId` known from `useAgentInfo`), `configuredDefault` fell back to `agentConfiguredDefault` as soon as the printers probe settled, even while the registry (`useEquipmentMachine`) query was still pending — a slow registry response could route a print to the old agent default for however long that window lasted, moments before the server-wins read would have overridden it. Added `registryPending = machineId != null && machine.isPending` (the `machineId != null` half is what excludes the legacy/no-identity path, where the query is disabled and stays "pending" forever without ever fetching — that path must keep resolving immediately, unchanged); `configuredDefault` now short-circuits to `null` while `registryPending` is true, ahead of both the registry-default and agent-default fallbacks. `defaultPrinter`'s own "always have SOME preselection" fallback is untouched (it derives from `configuredDefault` but has its own separate first-in-list rule), and every other returned field's semantics are unchanged.
- **Test evidence**: `useAgentPrinters.test.tsx` — 3 new tests under `pending-window safety`: (a) `configuredDefault` is `null` while the registry query is held open via a deferred promise, even though printers/agent-default already resolved, settling to the agent default once released (empty registry); (b) same shape but the registry resolves with a server default — `configuredDefault` transitions `null → "Server_Registry_Printer"`, never exposing the stale agent default in between; (c) legacy path (`machine_id` 404) unaffected — resolves the agent default immediately. Ran red first (both (a) and (b) initially returned the stale agent default instead of `null`), green after. Full `useAgentPrinters.test.tsx` (17 tests) and the six P3.2 consumer suites (`AttendeeDrawer`, `BulkBar`, `LaunchCeremony`, `StationPage`, `RecentScansRail`, `TestPrintDialog` — 134 tests) re-run explicitly and pass untouched.

### 9. Retry on empty registry (useEquipmentMachine)

- **Where**: `panel/src/shared/agent/useEquipmentMachine.ts`.
- **What**: The machine query had no `retry` override, so TanStack's default 3 retries applied to the documented-normal "never registered" 404 (`isEmptyRegistry`), delaying the first reconcile on a fresh machine for no benefit. Added `retry: false` — same house convention `useAgentInfo.ts`/`useAgentPrinters.ts` already follow, with a comment explaining why.
- **Test evidence**: `hooks.test.tsx` (equipment) — new test builds its own `QueryClient` **without** the file's usual `retry: false` client-level default (so the assertion actually exercises the hook's own `retry` setting, not the test harness masking it) and asserts exactly 1 request after a 404 settles (`retryDelay: 0` only removes the backoff wait, not the retry count under test). Ran red first (4 requests: 1 + 3 retries), green after (1 request).
