# BACKEND-SEC — backend/, БЕЗОПАСНОСТЬ

### BACKEND-SEC-01: Fallback на захардкоженный JWT-секрет в middleware валидации токенов
- Файл: backend/internal/middleware/jwt.go:29-33
- Описание: `middleware.JWT()` при отсутствии переменной окружения `JWT_SECRET` тихо подставляет захардкоженную строку `"idento_secret_key_change_me"` в качестве ключа проверки подписи (`secret = "idento_secret_key_change_me"`). При этом сторона выпуска токенов (`generateTokenForTenant` в internal/handler/auth.go:204-208) делает наоборот — жёстко требует `JWT_SECRET` и возвращает ошибку, если он не задан. Т.е. поведение асимметрично: подпись «fail closed», а проверка «fail open» на известную всем (из исходников) строку.
- Влияние: если на каком-либо окружении (dev/staging, неудачный docker-compose/k8s secret, забытая переменная) `JWT_SECRET` не установлен, сервер всё равно запускается (main.go не проверяет наличие `JWT_SECRET` при старте) и начинает принимать как валидные JWT, подписанные HS256 с публично известным ключом `idento_secret_key_change_me`. Любой внешний атакующий может сгенерировать токен с произвольными `user_id`, `tenant_id` и `role` (включая `"admin"` для любого чужого tenant_id) и получить полный доступ к данным любой организации — полный обход аутентификации/авторизации для всей мульти-тенантной системы.
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: убрать дефолт полностью — при пустом `JWT_SECRET` middleware должен возвращать 500/останавливать обработку (аналогично поведению при выпуске токена), а лучше — проверять наличие `JWT_SECRET` при старте приложения в main.go и завершать процесс (`log.Fatal`), если он не задан вне dev-окружения.
- Вердикт: ПОДТВЕРЖДЕНО — middleware/jwt.go:29-33 действительно подставляет хардкод при пустом `JWT_SECRET`, тогда как auth.go:204-208 (`generateTokenForTenant`) явно фейлится; main.go не проверяет наличие `JWT_SECRET` при старте.

### BACKEND-SEC-02: Захардкоженный секрет подписи JWT в обработчике QR-логина персонала
- Файл: backend/internal/handler/qr_auth.go:45-46
- Описание: `LoginWithQR` (маршрут `POST /auth/login-qr`, зарегистрирован в handler.go:29, публичный, без аутентификации) подписывает токен строкой-константой: `token.SignedString([]byte("your-secret-key")) // TODO: use env var`, вместо чтения `JWT_SECRET` из окружения, как это сделано в `generateTokenForTenant`.
- Влияние: (1) Секрет для подписи токена персонала захардкожен в исходном коде (CWE-798) — если этот код когда-либо совпадёт с реальным `JWT_SECRET` (например, при копировании значения из туториала/примера в конфиг) или будет переиспользован, любой, кто знает исходный код, сможет подделать JWT с произвольной ролью/tenant_id. (2) Практически прямо сейчас это, вероятнее всего, ломает сценарий входа персонала по QR: токен, подписанный `"your-secret-key"`, не пройдёт валидацию в `middleware.JWT()`, которая проверяет подпись по `JWT_SECRET` (реальному секрету) — т.е. staff получает 200 OK с токеном, который не работает ни на одном защищённом эндпоинте. QR-логин — это отдельный, специально упомянутый в задании как чувствительный сценарий аутентификации персонала.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: убрать литерал `"your-secret-key"`, использовать ту же функцию генерации токена, что и в auth.go (`generateTokenForTenant`), читающую `JWT_SECRET` из окружения и возвращающую ошибку при его отсутствии.
- Вердикт: ПОДТВЕРЖДЕНО — qr_auth.go:46 действительно подписывает токен литералом `"your-secret-key"` с комментарием `// TODO: use env var`, не используя `JWT_SECRET`/`generateTokenForTenant`.

### BACKEND-SEC-03: Отсутствует проверка принадлежности tenant при получении списка участников и обновлении статуса чек-ина
- Файл: backend/internal/handler/attendees.go:97-110 (GetAttendees), backend/internal/handler/attendees.go:182-246 (UpdateAttendeeHandler)
- Описание: `GetAttendees` (`GET /api/events/:event_id/attendees`) парсит `event_id` из URL и сразу вызывает `h.Store.GetAttendeesByEventID(ctx, eventID)` — нет ни одной проверки, что событие принадлежит tenant'у вызывающего пользователя (в отличие от `CreateAttendee`, `UpdateAttendeeInfo`, `BlockAttendee`, `UnblockAttendee`, `DeleteAttendee` в том же файле, где такая проверка есть). Аналогично `UpdateAttendeeHandler` (`PUT /api/attendees/:id`, используется для чек-ина/расчекина — помечен в коде как «for check-in status») получает участника по ID и обновляет `checkin_status`/`checked_in_at`/`checked_in_by` без какой-либо сверки `attendee.EventID` → `event.TenantID` с tenant'ом из JWT.
- Влияние: Любой аутентифицированный пользователь любой организации (в т.ч. только что самостоятельно зарегистрированный через публичный `POST /auth/register`, для которого не требуется подтверждение почты или модерация) может: (а) получить полный список участников (ФИО, email, компания, должность, `code`, custom_fields) произвольного мероприятия другой организации, зная/подобрав его `event_id`; (б) отметить чек-ин/расчекин произвольного участника чужого мероприятия, зная его `attendee_id`, тем самым портя данные регистрации чужого события. Это прямое нарушение горизонтальной изоляции арендаторов (tenant isolation) для одного из самых чувствительных типов данных и ключевого сценария (чек-ин).
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: в обоих хендлерах получить `event`/`tenant_id` из JWT и явно сверить `event.TenantID == tenantID` перед выполнением запроса/операции, по образцу уже существующих проверок в `BlockAttendee`/`DeleteAttendee` в этом же файле.
- Вердикт: ПОДТВЕРЖДЕНО — `GetAttendees` (attendees.go:97-110) и `UpdateAttendeeHandler` (182-246) действительно не содержат ни одной проверки tenant, в отличие от `BlockAttendee`/`UnblockAttendee`/`DeleteAttendee`/`UpdateAttendeeInfo` в том же файле.

- Серьёзность: High
- Уверенность: высокая
- Рекомендация: добавить проверку `event.TenantID == tenantID` (аналогично `BadgeZPL`) перед генерацией QR.
- Вердикт: ПОДТВЕРЖДЕНО — `GetAttendeeQR` (qr.go:12-38) получает attendee и сразу кодирует `attendee.Code` без единой проверки владения, в отличие от `BadgeZPL` (badge_zpl.go:70-79), которая такую проверку делает.

### BACKEND-SEC-05: ZoneCheckIn не проверяет ни принадлежность zone/event к tenant'у, ни назначение персонала на зону
- Файл: backend/internal/handler/zones.go:327-466
- Описание: `ZoneCheckIn` (`POST /api/zones/checkin`) — единственная проверка на роль/принадлежность отсутствует полностью: обработчик берёт `zone_id` и `attendee_code` из тела запроса, находит zone и attendee, проверяет только бизнес-правила (активна ли зона, время работы, доступ по категории), но ни разу не сверяет `zone`→`event.TenantID` с tenant'ом вызывающего, и не проверяет, назначен ли вызывающий пользователь как staff на эту зону (существующая функциональность `staff_zone_assignments`/`GetAvailableZones` для этого есть, но здесь не используется).
- Влияние: Любой аутентифицированный пользователь (включая только что самостоятельно зарегистрированного в собственной пробной организации) может выполнить чек-ин/провести регистрацию произвольного участника в произвольной зоне произвольного чужого мероприятия, если знает (или подбирает) `zone_id` и короткий `attendee_code` участника. Создаёт поддельные записи `zone_checkins` с `checked_in_by`, ссылающимся на постороннего пользователя, искажает данные о посещаемости чужого мероприятия и данные регистрации (для registration-зон ещё и выставляет `RegisteredAt`/`PacketDelivered`). Это ключевой сценарий чек-ина, полностью лишённый серверной авторизации.
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: в `ZoneCheckIn` после получения `zone` получить `event := GetEventByID(zone.EventID)` и сверить `event.TenantID` с tenant'ом из JWT; дополнительно проверять, что вызывающий пользователь имеет роль admin/manager либо назначен как staff на данную зону (`GetZoneStaffAssignments`/`GetStaffZoneAssignments`).
- Вердикт: ПОДТВЕРЖДЕНО — `ZoneCheckIn` (zones.go:327-466) действительно не сверяет tenant ни разу и не проверяет staff-назначение на зону; единственные проверки — бизнес-правила (активность зоны, время, регистрация, `CheckZoneAccess`).

### BACKEND-SEC-06: Системное отсутствие проверки tenant во всех остальных обработчиках zones.go
- Файл: backend/internal/handler/zones.go (CreateEventZone:18-36, GetEventZones:39-61, GetEventZone:64-76, UpdateEventZone:79-97, DeleteEventZone:100-111, CreateZoneAccessRule:116-134, GetZoneAccessRules:137-149, BulkUpdateZoneAccessRules:152-168, CreateAttendeeZoneAccess:173-191, GetAttendeeZoneAccess:194-206, UpdateAttendeeZoneAccess:209-227, DeleteAttendeeZoneAccess:230-241, AssignStaffToZone:246-273, GetZoneStaff:276-288, RemoveStaffFromZone:291-307, GetUserZoneAssignments:310-322, GetZoneCheckins:469-492, GetAttendeeZoneHistory:495-530)
- Описание: ни один из перечисленных обработчиков не сверяет `tenant_id` вызывающего пользователя с владельцем event/zone/attendee. Хендлеры принимают ID напрямую из URL/тела запроса и передают их в Store без какой-либо фильтрации по tenant (сам слой Store — internal/store/pg_store_zones.go — тоже не принимает tenant_id ни в одном из этих методов, так что фильтрации нет ни на одном уровне).
- Влияние: любой аутентифицированный пользователь любой организации может (при знании/переборе UUID): читать и изменять зоны, правила доступа по категориям, индивидуальные overrides доступа участников, историю перемещений участника по зонам чужого мероприятия; создавать/удалять зоны чужого мероприятия; назначать и снимать произвольных пользователей (`user_id` из тела запроса, тоже без проверки принадлежности) на зоны чужой организации. Это системный пробел авторизации по всей функциональности зон, отдельно от уже выделенного критического `ZoneCheckIn` (BACKEND-SEC-05).
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: добавить единую проверку владения (event/zone → tenant_id) — например, middleware или хелпер, резолвящий zone_id/attendee_id в tenant_id и сравнивающий с claims.TenantID, — и применить её последовательно ко всем перечисленным маршрутам, как это уже сделано в events.go/attendees.go для большинства операций.
- Вердикт: ПОДТВЕРЖДЕНО — проверены все перечисленные хендлеры в zones.go: ни один не сверяет tenant, и pg_store_zones.go подтверждает, что ни один из соответствующих store-методов не принимает/не фильтрует по tenant_id.

### BACKEND-SEC-07: API-ключи можно создавать, просматривать и отзывать для чужих мероприятий
- Файл: backend/internal/handler/api_keys.go:17-91 (CreateAPIKey, GetAPIKeys, RevokeAPIKey)
- Описание: все три обработчика управления API-ключами берут `event_id`/`key_id` из URL и вызывают Store напрямую, не проверяя, что `event_id` принадлежит tenant'у вызывающего пользователя (сравните с `CreateAPIKey`/`BulkCreateAttendees`, где такая проверка есть).
- Влияние: аутентифицированный пользователь любой организации может: создать новый API-ключ для чужого `event_id` и затем использовать публичный `POST /api/public/import` (защищённый только этим ключом, см. `middleware.APIKeyAuth`) для внедрения произвольных «участников» в чужое мероприятие; просмотреть список (имя, превью, срок действия) чужих ключей; отозвать (DoS) действующий API-ключ чужой интеграции, зная только его `key_id`.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: во всех трёх хендлерах получить событие по `event_id` (для Revoke — ключ по `key_id`, затем событие по `key.EventID`) и сверить `event.TenantID` с tenant'ом из JWT перед выполнением операции.
- Вердикт: ПОДТВЕРЖДЕНО — `CreateAPIKey`/`GetAPIKeys`/`RevokeAPIKey` (api_keys.go:17-91) берут `event_id`/`key_id` напрямую и вызывают Store без единой проверки tenant.

### BACKEND-SEC-08: QR-токен для быстрого входа персонала хранится и отдаётся в открытом виде через GET /api/users
- Файл: backend/internal/models/models.go:43 (`QRToken *string json:"qr_token,omitempty"`), backend/internal/store/pg_store.go:226-244 (GetUsersByTenantID выбирает qr_token), backend/internal/handler/users.go:17-36 (GetUsers)
- Описание: `qr_token` — это фактически второй пароль пользователя (см. `LoginWithQR`/qr_auth.go: обладание токеном = полная аутентификация без пароля). В отличие от `PasswordHash`, у которого стоит `json:"-"`, поле `QRToken` сериализуется в JSON и возвращается списком через `GET /api/users`, доступный любому пользователю с ролью `admin` или `manager` в тенанте. Кроме того, в БД токен хранится и сравнивается как открытый текст (`GetUserByQRToken`: `WHERE qr_token = $1`), без хеширования, в отличие от паролей (bcrypt) и API-ключей (bcrypt).
- Влияние: пользователь с ролью `manager` (более низкий уровень доверия, чем `admin`) может получить список пользователей своего тенанта вместе с открытыми QR-токенами администраторов и использовать их для входа под чужой учётной записью (`/auth/login-qr`) — горизонтальная эскалация привилегий manager → admin внутри одного tenant'а. Дополнительно, компрометация БД (бэкап, реплика, SQL-доступ) сразу отдаёт все действующие bearer-токены персонала в открытом виде.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: не включать `qr_token` в ответ списочных/чтения эндпоинтов (`json:"-"`, отдавать только сам факт наличия токена, например `has_qr_token: bool`); хранить в БД хеш токена (как это уже сделано для паролей и API-ключей), а не сам секрет.
- Вердикт: ПОДТВЕРЖДЕНО — `QRToken` в models.go:43 сериализуется (`omitempty`, не `-`), `GetUsersByTenantID` (pg_store.go:226-244) выбирает `qr_token`, `GetUserByQRToken`/`UpdateUserQRToken` хранят и сравнивают токен как открытый текст; `GetUsers` (users.go:17-36) доступен admin и manager.

### BACKEND-SEC-10: Разрешающий CORS для всех источников на API с Bearer-токенами
- Файл: backend/main.go:430-433
- Описание: `middleware.CORSWithConfig` сконфигурирован с `AllowOrigins: []string{"*"}` и разрешает заголовок `Authorization`; сам комментарий в коде отмечает «Configure properly for production».

### BACKEND-SEC-11: CSV-экспорт участников уязвим к формула-инъекции (CSV/Excel injection)
- Файл: backend/internal/handler/attendee_codes.go:116-171 (ExportAttendeesCSV)
- Описание: значения полей (`company`, `position`, произвольные `custom_fields`, пришедшие в том числе из bulk-импорта/внешнего API-импорта) записываются в CSV как есть, без экранирования ведущих символов `=`, `+`, `-`, `@`.
- Влияние: если атакующий (например, через `bulk_import`/`ExternalImport`, где custom_fields принимаются без валидации типов/содержимого) внесёт в поле участника формулу вида `=HYPERLINK(...)`/`=cmd|'/c calc'!A1`, а сотрудник тенанта откроет экспортированный CSV в Excel/Google Sheets, это может привести к выполнению формулы/DDE-инъекции на рабочей станции сотрудника.

### BACKEND-SEC-12: Отсутствие rate limiting на аутентификацию и чек-ин по короткому коду
- Файл: backend/internal/handler/auth.go (Login, LoginWithQR), backend/internal/handler/zones.go (ZoneCheckIn); в internal/middleware/ нет ни одного rate-limit/anti-bruteforce middleware
- Описание: пароль (`Login`), QR-токен (`LoginWithQR`) и код участника для чек-ина (`ZoneCheckIn`, коды генерируются функцией `generateUniqueCode()` — 8 hex-символов от UUID, т.е. ~32 бита энтропии) не защищены никаким ограничением частоты запросов ни на уровне Echo middleware, ни в бизнес-логике.
- Влияние: возможен онлайн-перебор паролей пользователей, перебор attendee_code (32 бита — практически осуществимо при отсутствии лимитов) в сочетании с BACKEND-SEC-05 для несанкционированных чек-инов, а также перебор QR-токенов персонала.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: добавить rate limiting (по IP и/или по email/zone_id) на `/auth/login`, `/auth/login-qr` и `/api/zones/checkin`, например через `echo/middleware.RateLimiter`.
- Вердикт: ПОДТВЕРЖДЕНО — в internal/middleware/ (jwt.go, limits.go, super_admin.go, api_key.go) действительно нет ни одного rate-limit/anti-bruteforce механизма, и main.go не регистрирует такое middleware глобально.

### BACKEND-SEC-13: Пароль передаётся как аргумент командной строки в утилите reset_password
- Файл: backend/cmd/reset_password/main.go:19-24
- Описание: CLI-утилита принимает новый пароль вторым аргументом (`os.Args[2]`) и вызывает `bcrypt.GenerateFromPassword` над ним; пароль в открытом виде проходит через историю шелла, список процессов (`ps`) на время выполнения и, вероятно, попадает в логи CI/bash-историю при использовании на сервере.

- Серьёзность: Low
- Уверенность: высокая
- Рекомендация: читать пароль интерактивно (без эха) или из переменной окружения/stdin вместо позиционного аргумента.
- Вердикт: ПОДТВЕРЖДЕНО — cmd/reset_password/main.go:19-24 действительно читает пароль из `os.Args[2]` без какой-либо альтернативы (env/stdin/интерактивный ввод).
