# WEB-BUG — web/src/, КОРРЕКТНОСТЬ / БАГИ

### WEB-BUG-01: Гонка при polling сканера может привести к повторной обработке одного скана
- Файл: web/src/hooks/useScanner.ts:18-43 (функция `pollScanner`), web/src/hooks/useScanner.ts:54 (`setInterval(pollScanner, 200)`)
- Описание: `pollScanner` — асинхронная функция, вызываемая через `setInterval(pollScanner, 200)`. Таймер не ждёт завершения предыдущего вызова: если один цикл `getLastScan()` + `clearLastScan()` (два отдельных HTTP-запроса к локальному агенту, `web/src/lib/agent.ts:130-148`) занимает больше 200 мс (просадка агента, GC-пауза, задержка сети), запускается новый параллельный вызов `pollScanner` ещё до того, как предыдущий успел вызвать `clearLastScan()`. Оба вызова видят один и тот же ещё не очищенный скан и оба вызывают `setLastScan({...})` с новым объектом — а значит `useEffect` в `CheckinFullscreen.tsx:107-113`, реагирующий на изменение `lastScan`, срабатывает дважды для одного и того же физического скана. Никакого мьютекса/флага "запрос уже выполняется" в `pollScanner` нет.
- Влияние: Один скан бейджа/QR-кода может вызвать двойной вызов `handleSearch`/`handleCheckin` (web/src/pages/CheckinFullscreen.tsx:267-388), что при печати бейджей приводит к двойной печати одного бейджа (см. WEB-BUG-02) и к лишним PUT-запросам обновления посетителя.
- Серьёзность: High
- Уверенность: средняя
- Рекомендация: Добавить `useRef`-флаг "poll in flight" и пропускать очередной тик, пока предыдущий не завершился (или использовать `AbortController`/отмену), либо переключиться на последовательный `while`-цикл с `await` вместо `setInterval`.

### WEB-BUG-02: `isFirstCheckin` вычисляется по устаревшему локальному состоянию, а не по ответу сервера
- Файл: web/src/pages/CheckinFullscreen.tsx:267-325 (`handleCheckin`), строка 282: `const isFirstCheckin = !attendee.checkin_status;`
- Описание: Флаг "первый ли это чек-ин" берётся из локального массива `attendees` (последний раз загруженного через `fetchAttendees`), а не из ответа PUT-запроса. Если тот же посетитель регистрируется дважды подряд быстрее, чем успевает выполниться `fetchAttendees` после первого чек-ина (см. строки 305-308: обновление списка запускается уже ПОСЛЕ показа результата, не блокируя следующий ввод), либо срабатывает двойная обработка одного скана (WEB-BUG-01), второй вызов снова увидит `checkin_status: false` и трактует повторный чек-ин как "первый": перезапишет `checked_in_at`, покажет зелёный "success" вместо жёлтого "уже зарегистрирован" и, при `!manualPrint`, повторно распечатает бейдж (строки 300-303, `printBadge`).
- Влияние: Дублирующая печать бейджей, неверная индикация статуса регистрации персоналу на стойке чек-ина (ключевой сценарий приложения), перезапись времени первого чек-ина.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Определять "первый чек-ин" по значению, которое возвращает backend в ответе на PUT (например, флаг `already_checked_in` из ответа), а не по локально закэшированному `attendee.checkin_status`; либо блокировать повторный ввод, пока предыдущий запрос чек-ина не завершится.

### WEB-BUG-03: Редактор бейджей не предупреждает о потере несохранённых изменений
- Файл: web/src/pages/BadgeTemplateEditorV2.tsx (весь компонент, 37-1135); состояние без отслеживания "грязности": строки 43-63 (`widthMM`, `heightMM`, `dpi`, `elements`); мутации без сохранения: `addTextField`/`addQRCode`/`addBarcode`/`addLine`/`addBox`/`updateElement`/`handleDragEnd`/`deleteElement`, строки 111-206
- Описание: Все изменения макета бейджа (добавление/перемещение/удаление элементов, изменение размеров) хранятся только в локальном state и попадают на сервер лишь по явному нажатию кнопки "Сохранить" (`handleSave`, строки 208-247). В коде нет `window.onbeforeunload`, нет `useBlocker`/`usePrompt` react-router и вообще нет обработчика подтверждения перехода (проверено `grep` по всему `web/src` — совпадений нет). Переход на другую вкладку события, случайное нажатие "назад" в браузере или закрытие вкладки без сохранения полностью уничтожает произведённую работу без предупреждения.
- Влияние: Потеря несохранённого макета бейджа (позиции элементов, размеры, шрифты, QR/штрихкод-привязки) при любой случайной навигации; пользователь узнаёт об этом только вернувшись в редактор и увидев старый шаблон.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Добавить отслеживание "dirty"-состояния (сравнение текущего `elements/widthMM/heightMM/dpi` с последним сохранённым) и подтверждение перед уходом со страницы через `window.onbeforeunload` и/или блокировку роутинга (`useBlocker`/`unstable_usePrompt`).

### WEB-BUG-04: Сохранение шаблона бейджа делает полный overwrite объекта события устаревшим снимком (потеря параллельных изменений)
- Файл: web/src/pages/BadgeTemplateEditorV2.tsx:208-247 (особенно строки 222-228: `await api.put(\`/api/events/${eventId}\`, { ...event, custom_fields: { ...event.custom_fields, badgeTemplate: template } })`); аналогичный паттерн: web/src/pages/event/EventSettings.tsx:90-116 (строки 93-107)
- Описание: `BadgeTemplateEditorV2` держит собственную, независимо загруженную копию `event` (строка 52, заполняется в `loadEventAndTemplate`, строки 78-109), а не общий `event` из `useOutletContext` — из контекста берётся только `reloadEvent`. При сохранении шаблона выполняется PUT всего объекта события, собранного из этого локального снимка (`...event`) плюс новый `badgeTemplate`. Если за время, пока пользователь работал в редакторе бейджей (это может быть долгий процесс — расстановка элементов, шрифтов), другой сотрудник/вкладка изменил другие поля события (название, даты, `attendeeTemplate`, `badgeTypeField` и т.п., как это делает `EventSettings.tsx`), эти изменения будут молча перезаписаны устаревшим локальным `event` в момент сохранения шаблона. Тот же паттерн "полный PUT снимком, взятым при монтировании" повторяется и в `EventSettings.tsx`.
- Влияние: Незаметная потеря параллельно внесённых изменений в настройки события при одновременной работе нескольких организаторов с одним событием (типичный сценарий для команды на мероприятии).
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Сохранять точечно (PATCH только `custom_fields.badgeTemplate` / только изменённых полей формы) вместо полного PUT снимком всего объекта события; либо перед сохранением подтягивать свежую версию события и мёржить именно изменённое поле.

### WEB-BUG-05: Импорт CSV не учитывает кодировку файла — кириллица/спецсимволы могут превратиться в "кракозябры"
- Файл: web/src/components/CSVImportEnhanced.tsx:68-90 (`parseCSV`, вызов `Papa.parse(file, { header: true, skipEmptyLines: true, complete, error })`)
- Описание: В конфиге `Papa.parse` не задан параметр `encoding`, поэтому файл читается как UTF-8 по умолчанию. CSV-файлы, экспортированные из Excel в кириллической локали, часто сохраняются в Windows-1251 (ANSI), а не в UTF-8. При загрузке такого файла все нелатинские символы (имена, компании, кастомные поля) будут декодированы неверно и превратятся в нечитаемые символы — без какой-либо ошибки или предупреждения пользователю, т.к. `Papa.parse` успешно "распарсит" некорректно декодированный текст.
- Влияние: Массовое незаметное искажение данных участников (ФИО, email, кастомные поля) при импорте файлов не в UTF-8, что напрямую влияет на бейджи и чек-ин (неверные имена на распечатанных бейджах).
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Определять/уточнять кодировку (например, через `jschardet`/эвристику по BOM, либо явный выбор кодировки пользователем "UTF-8 / Windows-1251" перед парсингом) и передавать её в `Papa.parse({ encoding })`; либо хотя бы показывать предупреждение, если после парсинга обнаружены символы replacement character (U+FFFD).

### WEB-BUG-06: Гонка состояний при быстрой смене CSV-файла — старый парсинг может перезаписать данные новее выбранного файла
- Файл: web/src/components/CSVImportEnhanced.tsx:60-66 (`handleFileChange`), 68-90 (`parseCSV`), 165-177 (кнопка "Изменить", сброс `file`/`csvData`)
- Описание: `parseCSV` не имеет токена отмены/идентификатора запроса. Пользователь может: выбрать большой файл A (парсинг ещё выполняется асинхронно), затем нажать "Изменить" (сбрасывает `file`, `csvData`, `fieldSchema`, снова показывает `<input type=file>`) и выбрать файл B поменьше. Оба вызова `Papa.parse` работают параллельно; чей `complete`-колбэк завершится позже, тот и запишет финальные `csvData`/`fieldSchema`/`hasCodeColumn` через `setCSVData`/`setFieldSchema`/`setHasCodeColumn` — независимо от того, какой файл сейчас показан как выбранный (`file.name`). Если A (большой) завершится позже B, пользователь увидит имя файла B, но предпросмотр и импортируемые данные будут от файла A.
- Влияние: Импорт данных из "не того" CSV-файла без какой-либо индикации несоответствия — пользователь может импортировать не тот набор участников, который видит выбранным в UI.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Использовать идентификатор/токен текущего запроса парсинга (например, ref со счётчиком) и игнорировать в `complete`/`error` колбэках результаты парсинга, если файл с момента запуска парсинга успел смениться.

### WEB-BUG-07: Нет ограничения размера/строк CSV и парсинг не вынесен в worker — риск зависания вкладки на больших файлах
- Файл: web/src/components/CSVImportEnhanced.tsx:60-66 (`handleFileChange`), 68-90 (`parseCSV` без `worker: true`), 92-118 (`handleImport` — один большой POST без чанкинга/таймаута)
- Описание: `handleFileChange` не проверяет размер файла или количество строк перед вызовом `Papa.parse`. Парсинг выполняется в основном потоке (`worker` не включён), а весь результат целиком копится в `csvData` и затем одним запросом уходит на `/api/events/${eventId}/attendees/bulk` (строка 102-105). Для очень большого CSV (десятки/сотни тысяч строк, что реалистично для крупных мероприятий) это может надолго заблокировать UI-поток при парсинге и породить один огромный JSON-запрос без таймаута/ретрая/прогресса.
- Влияние: Зависание вкладки браузера при импорте больших файлов, неинформативное "зависшее" состояние кнопки "Импортировать" без прогресса, риск обрыва запроса без внятной обратной связи пользователю.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Добавить проверку размера/числа строк с предупреждением, включить `worker: true` в конфиге Papa.parse для больших файлов, и/или разбивать импорт на чанки с прогресс-баром.

### WEB-BUG-08: Форматтеры дат падают с необработанным исключением на невалидной дате, а в приложении нет ErrorBoundary
- Файл: web/src/utils/dateFormat.ts:6-21 (`formatDateTime`), аналогично 26-37 (`formatDate`), 42-54 (`formatTime`), 59-83 (`formatRelativeTime`); используется в web/src/pages/CheckinFullscreen.tsx:751 и web/src/pages/event/EventAttendees.tsx:476
- Описание: Функции проверяют только `!date` (null/undefined/пустая строка), но не проверяют, что `new Date(date)` дал валидный результат. `new Intl.DateTimeFormat(locale, options).format(dateObj)` бросает `RangeError: Invalid time value`, если `dateObj` — `Invalid Date` (например, при повреждённом/нестандартном значении `checked_in_at` или дате из кастомного поля). Проверено по всему `web/src`: ни в `App.tsx`, ни в `main.tsx` не подключён React ErrorBoundary, — необработанное исключение при рендере уронит всё дерево компонентов (белый экран) вместо локальной деградации.
- Влияние: Один некорректный по формату таймстамп (например, из повреждённых данных API/CSV-импорта в кастомное поле, который где-то передан в форматтер) приводит к полному краху страницы чек-ина или списка участников, а не к отображению "-" для одной ячейки.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Проверять `isNaN(dateObj.getTime())` в начале каждого форматтера и возвращать `'-'`/safe-fallback вместо вызова `Intl.DateTimeFormat`; добавить хотя бы один React ErrorBoundary верхнего уровня в `App.tsx`, чтобы падение рендера не сносило всё приложение.

### WEB-BUG-09: Falsy-проверка данных для полей бейджа отбрасывает валидные нулевые/пустые значения
- Файл: web/src/utils/zpl.ts:91 (`generateTextZPL`), 190 (`generateQRCodeZPL`), 219 (`generateBarcodeZPL`) — во всех трёх: `if (element.source && data[element.source]) { ... = String(data[element.source]); }`
- Описание: Проверка `data[element.source]` истинности (truthy), а не `!= null`/`in`, означает, что легитимное значение кастомного поля, равное числу `0` (например, "номер стола: 0", "приоритет: 0"), трактуется как отсутствующее — вместо него используется `element.text || ""` (обычно пустая строка), т.к. `0` — falsy в JS.
- Влияние: Для участников с нулевым значением в поле, привязанном к тексту/QR-коду/штрихкоду на бейдже, соответствующий элемент печатается пустым — бейдж или QR/штрихкод оказывается некорректным без какой-либо ошибки при предпросмотре или печати.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Заменить проверку на `element.source && data[element.source] !== undefined && data[element.source] !== null && data[element.source] !== ''` (или аналогичную явную проверку на nullish), чтобы `0`/`false` не терялись.

### WEB-BUG-10: Таймер тестирования сканера в настройках оборудования не останавливается при размонтировании компонента
- Файл: web/src/pages/EquipmentSettings.tsx:92 (`testIntervalRef`), 358-412 (`startScannerTest`, `setInterval` на строке 389), 414-422 (`closeScannerTest`)
- Описание: `setInterval` в `startScannerTest` останавливается только при успешном совпадении кода, по таймауту (60 попыток × 500 мс) или при явном закрытии диалога (`closeScannerTest`). В файле нет `useEffect`-очистки, которая останавливала бы `testIntervalRef.current` при размонтировании компонента (проверено — единственные вызовы `clearInterval` находятся внутри самого колбэка интервала и в `closeScannerTest`, вызываемого только руками пользователя). Если пользователь уходит со страницы "Оборудование" во время активного теста сканера (например, переходит в другой раздел), опрос `agentApi.getLastScan()` каждые 500 мс продолжает выполняться в фоне ещё до 30 секунд.
- Влияние: Утечка сетевых запросов к локальному агенту после ухода со страницы; лишняя нагрузка и потенциальные побочные эффекты (`agentApi.clearLastScan()` может "съесть" реальный скан, ожидаемый другим экраном, если пользователь параллельно уже открыл чек-ин).
- Серьёзность: Low
- Уверенность: высокая
- Рекомендация: Добавить `useEffect(() => () => { if (testIntervalRef.current) clearInterval(testIntervalRef.current); }, [])` для очистки таймера при размонтировании.

### WEB-BUG-11: Поток чек-ина не имеет обработки офлайн-режима и повторной синхронизации
- Файл: web/src/pages/CheckinFullscreen.tsx:267-325 (`handleCheckin`, блок `catch` строки 317-324), web/src/hooks/useScanner.ts (весь файл)
- Описание: Во всём `web/src` нет ни одного использования `navigator.onLine`, слушателей `online`/`offline`, ни какой-либо очереди отложенной синхронизации (проверено `grep` по всему каталогу). Единственная реакция на сбой сети при чек-ине — `console.error` и `toast`/`scanResult` с текстом "checkinFailed" на 3 секунды (строки 317-324), после чего запись просто исчезает без сохранения попытки. Ключ локализации `offlineMode` существует в `i18n.ts`, но нигде в `web/src` не используется — похоже на не реализованную/недоделанную фичу.
- Влияние: При потере связи со стойки чек-ина (типичная ситуация на мероприятиях с нестабильным Wi-Fi) каждый неудавшийся чек-ин полностью теряется и требует, чтобы оператор вручную заметил ошибку и повторил скан вручную — никакой очереди "отправить, когда появится сеть" нет.
- Серьёзность: Low
- Уверенность: средняя
- Рекомендация: Добавить локальную очередь неотправленных чек-инов (например, в `localStorage`) с индикатором офлайн-режима и авто-ресинхронизацией по событию `online`, либо явно ограничиться понятным индикатором "нет соединения" вместо использования неподключённого ключа `offlineMode`.
