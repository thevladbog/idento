# DESKTOP-SEC — desktop/src + desktop/src-tauri, Безопасность

### DESKTOP-SEC-01: IPC-команда `agent_request` строит URL без валидации пути → SSRF из доверенного Rust-процесса
- Файл: desktop/src-tauri/src/commands.rs:11-44 (ключевая строка: 16), регистрация команды в desktop/src-tauri/src/lib.rs:11-14, вызывается из desktop/src/lib/agent.ts:12-34
- Описание: `#[tauri::command] agent_request(method, path, body)` строит целевой URL как `format!("http://127.0.0.1:{}{}", AGENT_PORT_STR, path)`, где `path` — строка, полностью контролируемая вызывающим JS-кодом из WebView (через `invoke("agent_request", {method, path, body})`, см. `agentGet`/`agentPost` в desktop/src/lib/agent.ts). Параметр `path` никак не проверяется: не требуется, чтобы он начинался с `/`, не сверяется с allow-list допустимых эндпоинтов агента (`/health`, `/printers`, `/scan/...` и т.д.). Из-за наивной конкатенации строк, а не построения URL через `Url::parse(base).join(path)`, значение `path`, содержащее символ `@` (например, `"@evil.example.com/x"`), меняет итоговую строку на `"http://127.0.0.1:12345@evil.example.com/x"`. При разборе такой строки URL-парсером (`url`/`reqwest`) `127.0.0.1:12345` интерпретируется как userinfo, а `evil.example.com` — как реальный host, то есть запрос уйдёт не на локальный агент, а на произвольный внешний (или внутрисетевой) хост. Команда возвращает тело ответа вызывающему коду (`Result<String, String>`), то есть это полноценный SSRF с чтением ответа, выполняемый из нативного процесса — в обход CSP `connect-src` (см. DESKTOP-SEC-05) и любых сетевых ограничений самого WebView.
- Влияние: любой JS-код, способный вызвать эту IPC-команду (напр. в результате компрометации npm-зависимости во фронтенде — axios/i18next/jsqr/qrcode/sonner и т.д. — или будущей XSS), может заставить нативный процесс приложения выполнять HTTP-запросы к произвольным адресам (включая внутреннюю сеть площадки, где стоит киоск) и получать содержимое ответов обратно в WebView. Даже без трюка с `@`, отсутствие allow-list путей позволяет обращаться к любому эндпоинту локального агента печати/сканера, а не только к тем, что использует UI.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: строить URL через `reqwest::Url::parse` с базой `http://127.0.0.1:{port}` и `.join(path)` с последующей проверкой, что итоговый `host()` действительно `127.0.0.1`/`localhost` и порт совпадает с `AGENT_PORT`; отклонять URL, если разбор дал другой host. Дополнительно ограничить `path` явным allow-list префиксов (`/health`, `/printers`, `/printers/*`, `/scanners`, `/scan/*`, `/print`), например регэкспом `^/[a-zA-Z0-9/_-]+$` плюс allow-list по началу пути.
- Вердикт: ПОДТВЕРЖДЕНО — `commands.rs:16` строит URL наивной конкатенацией без валидации `path`; воспроизведено WHATWG-парсингом (тот же стандарт, что использует `url`/`reqwest`) — `http://127.0.0.1:12345@evil.example.com/x` разбирается с `host=evil.example.com`, `userinfo=127.0.0.1:12345`, что подтверждает SSRF в обход CSP `connect-src`.

### DESKTOP-SEC-02: JWT и сессионные данные хранятся в localStorage без шифрования
- Файл: desktop/src/lib/api.ts:4-19 (SESSION_KEYS, чтение `localStorage.getItem("token")`), desktop/src/pages/Login.tsx:32-38, desktop/src/pages/QRLogin.tsx:31-32
- Описание: После логина/QR-логина `token`, `user`, `tenants`, `current_tenant` кладутся в `localStorage` (обычный, незашифрованный DOM Storage). `api.ts` затем на каждый запрос читает `token` из localStorage и подставляет как `Bearer`. Tauri не предоставляет localStorage какой-либо изоляции сверх обычного WebView storage — данные лежат в открытом виде на диске в профиле WebView (например, в LevelDB/SQLite файлах Local Storage внутри `$APPDATA`), доступны любому JS-коду в этом origin (в т.ч. через supply-chain компрометацию зависимости) и любому процессу/пользователю с доступом к файловой системе киоска.
- Влияние: На физически доступном киоск-устройстве (типичный сценарий для check-in терминала на мероприятии) кража токена оператора возможна через доступ к файлам профиля приложения или через любой JS, исполнившийся в контексте окна (не обязательно классический XSS — например, вредоносная зависимость, dev-tools доступ, или баг в другой части кода, рендерящий непроверенный HTML в будущем). Токен даёт доступ к API бэкенда от имени сотрудника мероприятия.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Хранить токен в защищённом от чтения любым JS хранилище — например, использовать `tauri-plugin-store` совместно с OS keychain/credential vault (`tauri-plugin-stronghold` или отдельный Rust-command, кладущий токен в OS keyring через `keyring-rs`), а фронтенду выдавать только opaque handle/использовать сам Rust-слой для добавления заголовка Authorization к запросам, не передавая сырой токен в JS-контекст.
- Вердикт: ПОДТВЕРЖДЕНО — `api.ts:4` (SESSION_KEYS) и `:19` (`localStorage.getItem("token")`), `Login.tsx:32-38`, `QRLogin.tsx:31-32` точно совпадают с описанием; токен действительно хранится в открытом виде в WebView localStorage.

### DESKTOP-SEC-03: Разрешение `shell:allow-open` выдано без ограничивающего scope и не используется приложением
- Файл: desktop/src-tauri/capabilities/default.json:5-10 (permissions: `"shell:allow-open"`)
- Описание: Capability `main-capability` включает `shell:allow-open` — согласно сгенерированному ACL-манифесту (`desktop/src-tauri/gen/schemas/acl-manifests.json`, permission `allow-open`) это "Enables the open command without any pre-configured scope", в отличие от `shell:default`, который явно ограничивает открытие только `http(s)://`, `tel:` и `mailto:` схемами. В capabilities-файле не задан дополнительный `scope`/`allow` для этой команды. При этом grep по всему `desktop/src/` не находит ни одного использования `@tauri-apps/plugin-shell`/`open()` — разрешение выдано, но фронтенд его нигде не вызывает.
- Влияние: Нарушение принципа минимальных привилегий: неиспользуемое разрешение расширяет поверхность атаки. Если в будущем (или в результате компрометации зависимости) в WebView выполнится код, вызывающий `plugin:shell|open`, он сможет открыть произвольный URL/путь/протокол-хендлер средствами ОС без ограничения по схеме, поскольку разрешение выдано в "широком" варианте, а не в дефолтном ограниченном.
- Серьёзность: Medium
- Уверенность: средняя (semantics разрешения подтверждены из сгенерированного ACL-манифеста; точное рантайм-поведение shell-плагина без scope не проверялось запуском)
- Рекомендация: Убрать `shell:allow-open` из capabilities, так как код им не пользуется. Если функциональность открытия ссылок понадобится — использовать `shell:default` (ограниченный scope http(s)/tel/mailto) вместо "сырого" `shell:allow-open`.
- Вердикт: ПОДТВЕРЖДЕНО — `default.json:7` содержит `shell:allow-open`; ACL-манифест (`acl-manifests.json`) подтверждает описание "without any pre-configured scope"; `grep -rn "shell" desktop/src/` не находит использования плагина.

### DESKTOP-SEC-04: `withGlobalTauri: true` без необходимости расширяет глобальную поверхность IPC в WebView
- Файл: desktop/src-tauri/tauri.conf.json:13
- Описание: `app.withGlobalTauri` включён, что инжектит весь Tauri JS API (включая `invoke`) в глобальный объект `window.__TAURI__` для любого скрипта, выполняющегося в странице. При этом фактический код приложения нигде не использует глобальный объект: `desktop/src/lib/agent.ts:9,14,24` и `desktop/src/pages/CheckinEvent.tsx:313` явно делают `await import("@tauri-apps/api/core")` / `await import("@tauri-apps/api/window")` — то есть используют ES-модули, а не `window.__TAURI__`. Проверка `"__TAURI__" in window` в agent.ts используется лишь как признак окружения, а не для вызова API.
- Влияние: Любой сторонний скрипт, который может оказаться выполненным в контексте окна (скомпрометированная npm-зависимость, будущая ошибка рендеринга непроверенного HTML и т.п.), получает прямой беспарольный доступ к `invoke()` и остальному Tauri API через глобальный объект, без необходимости импортировать модуль. Это увеличивает риск при отсутствии необходимости — приложение и так работает через явные импорты.
- Серьёзность: Low
- Уверенность: высокая
- Рекомендация: Установить `"withGlobalTauri": false` (или убрать поле — по умолчанию false) и продолжать использовать ES-module импорты, как уже сделано в коде.
- Вердикт: ПОДТВЕРЖДЕНО — `tauri.conf.json:13` действительно `"withGlobalTauri": true`; `agent.ts:9,14,24` и `CheckinEvent.tsx:313` точно совпадают со строками, где используются ES-импорты `@tauri-apps/api/*` вместо глобального объекта.

### DESKTOP-SEC-05: CSP `connect-src` жёстко ограничен plaintext HTTP-эндпоинтами, но UI предлагает настраивать произвольный (в т.ч. сетевой) backend URL
- Файл: desktop/src-tauri/tauri.conf.json:14-22 (connect-src, строка 18), desktop/src/lib/config.ts:1-21, desktop/src/pages/Connection.tsx:19-95, desktop/src/i18n.ts (placeholder `"serverUrlDesc": "Set the Idento backend URL (e.g. http://192.168.1.10:8008)"`)
- Описание: CSP `connect-src` в tauri.conf.json жёстко перечисляет только `'self' ipc: http://ipc.localhost http://localhost:8008 http://127.0.0.1:8008` — фиксированный список plaintext (не HTTPS) origin'ов. Однако страница `Connection.tsx` и `config.ts` (`getBackendUrl`/`setBackendUrl`) явно спроектированы для того, чтобы оператор мог указать произвольный backend URL, сохраняемый в localStorage (`idento_backend_url`) и подставляемый как `axios.baseURL`; текст плейсхолдера в UI прямо предлагает пример `http://192.168.1.10:8008` — то есть адрес в локальной сети, отличный от `localhost`. Если такой origin не входит в жёстко заданный CSP-список, все запросы (включая логин и передачу Bearer-токена) к нему будут заблокированы политикой `connect-src` WebView; если же CSP в будущем ослабят до `connect-src *`/широкого списка ради этой функции, то передача JWT и персональных данных участников будет идти открытым HTTP по сети киоска.
- Влияние: Либо заявленная функция "указать URL сервера" фактически не работает для любых адресов, кроме двух захардкоженных (проблема корректности/UX, косвенно секьюрити — путаница у оператора), либо, при ослаблении CSP под реальные развёртывания, Bearer-токен и PII участников мероприятия передаются по сети в открытом виде (HTTP), что даёт возможность перехвата в общей Wi-Fi/LAN сети площадки.
- Серьёзность: Medium
- Уверенность: средняя (не запускал приложение, чтобы наблюдать фактическое поведение CSP при вводе стороннего URL; вывод сделан из статического анализа конфигурации и кода)
- Рекомендация: Если реальные развёртывания используют backend в локальной сети (не localhost), нужно либо: (а) требовать HTTPS для нелокальных backend URL и обновлять CSP `connect-src` динамически/через wildcard по HTTPS-схеме, либо (б) явно документировать и технически ограничить работу только через localhost/loopback (например, обязательный SSH/VPN туннель), убрав вводящий в заблуждение LAN-пример из UI.
- Вердикт: ПОДТВЕРЖДЕНО — `tauri.conf.json:18` жёстко перечисляет только localhost/127.0.0.1:8008 (единственный CSP-конфиг, платформенных override-файлов нет), а `Connection.tsx`/`config.ts` реализуют произвольный backend URL с LAN-примером в `i18n.ts:76`; противоречие реально.

### DESKTOP-SEC-06: Окно киоска не защищено от выхода в ОС (нет fullscreen/kiosk-lock, есть стандартные декорации и resize)
- Файл: desktop/src-tauri/tauri.conf.json:23-32 (конфигурация окна `main`)
- Описание: Единственное окно приложения создаётся с `"resizable": true` и без `"fullscreen": true`, `"decorations": false`, `"always_on_top": true` или иного kiosk-lockdown. Значит, у окна по умолчанию есть стандартная рамка ОС с кнопками свернуть/закрыть, его можно менять в размере и переключать через Alt+Tab/Cmd+Tab. Приложение сознательно позиционируется как "киоск-приложение для чек-ина" (см. product-name "Idento Kiosk", описание в tauri.conf.json), то есть предполагается физический доступ посетителей/операторов к терминалу.
- Влияние: Человек, стоящий у стойки регистрации, может свернуть/закрыть приложение или переключиться на рабочий стол ОС и получить доступ к файловой системе, другим приложениям, настройкам сети и т.д. — классический "kiosk escape". В сочетании с DESKTOP-SEC-02 (токен в localStorage) это также облегчает извлечение файлов профиля WebView с сессионным токеном.
- Серьёзность: Medium
- Уверенность: средняя (это, в первую очередь, эксплуатационная/физическая мера защиты; часть блокировки может обеспечиваться внешними средствами ОС/загрузчика киоска вне рассмотренного кода, что не проверялось в рамках данного аудита)
- Рекомендация: Для продакшн-сборки киоска включить `"fullscreen": true`, `"decorations": false`, отключить системные комбинации переключения окон средствами ОС (или через отдельный kiosk-lock механизм), либо явно задокументировать, что lockdown обеспечивается внешней ОС-конфигурацией, не входящей в этот репозиторий.
- Вердикт: ПОДТВЕРЖДЕНО — `tauri.conf.json:23-32` содержит только `resizable`/`center`, без `fullscreen`/`decorations`/`alwaysOnTop`; описанный риск kiosk-escape корректен.
