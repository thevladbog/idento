### BACKEND-BUG-01: Состояние регистрации на зоне никогда не сохраняется в БД — чек-ин с обязательной регистрацией не проходит навсегда
- Файл: backend/internal/models/models.go:87-89 (поля `PacketDelivered`, `RegisteredAt`, `RegistrationZoneID`); backend/internal/handler/zones.go:372-386 (`ZoneCheckIn`, установка полей и проверка); backend/internal/store/pg_store.go:407-422 (`CreateAttendee`), :424-457 (`GetAttendeesByEventID`), :459-479 (`GetAttendeeByCode`), :481-501 (`GetAttendeeByID`), :503-523 (`UpdateAttendee`)
- Описание: Для зоны регистрации (`zone.IsRegistrationZone == true`) `ZoneCheckIn` выставляет в памяти `attendee.RegisteredAt`, `attendee.RegistrationZoneID`, `attendee.PacketDelivered = true` и вызывает `h.Store.UpdateAttendee(ctx, attendee)`. Но ни один SQL-запрос в store, работающий с таблицей `attendees` (`CreateAttendee`, `GetAttendeeByID`, `GetAttendeeByCode`, `GetAttendeesByEventID`, `UpdateAttendee`), не упоминает колонки `registered_at`, `registration_zone_id`, `packet_delivered` — они добавлены в схему миграцией `000010_event_zones.up.sql`, но полностью отсутствуют во всех SELECT/INSERT/UPDATE запросах Go-кода. Проверено грепом по всему `internal/store` и `internal/handler` — совпадения только в models.go и zones.go.
- Влияние: После "успешной" регистрации (`ZoneCheckIn` возвращает `200 OK, "Check-in successful"`) в БД `registered_at` остаётся NULL, `packet_delivered` — FALSE. При следующем запросе (тот же или другой сотрудник сканирует бейдж на ЛЮБОЙ другой зоне) участник заново читается из БД с `RegisteredAt == nil`, и проверка `if zone.RequiresRegistration && attendee.RegisteredAt == nil` (zones.go:386) навсегда возвращает `403 "Attendee must register first"` — для любой зоны, у которой `requires_registration` не выключен явно (а по умолчанию в схеме `requires_registration BOOLEAN DEFAULT TRUE`). То есть вся функция мультизонного чек-ина с обязательной регистрацией фактически неработоспособна: участник, "зарегистрированный" на входе, не может пройти ни в одну другую зону мероприятия.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Добавить `registered_at`, `registration_zone_id`, `packet_delivered` в SELECT/INSERT/UPDATE `attendees` в pg_store.go (или вынести регистрацию в отдельный, чётко протестированный store-метод) и покрыть сценарий "регистрация → чек-ин в другой зоне" интеграционным тестом.
- Вердикт: ПОДТВЕРЖДЕНО — проверены все перечисленные SQL-запросы в pg_store.go (CreateAttendee:407-422, GetAttendeesByEventID:424-457, GetAttendeeByCode:459-479, GetAttendeeByID:481-501, UpdateAttendee:503-523) — ни один не упоминает `registered_at`/`registration_zone_id`/`packet_delivered`, при этом миграция 000010_event_zones.up.sql:90-92 добавляет эти колонки.

### BACKEND-BUG-02: Слепая перезапись всей строки attendee без блокировки — потерянные обновления при параллельном чек-ине/редактировании
- Файл: backend/internal/handler/attendees.go:182-246 (`UpdateAttendeeHandler`, чек-ин/расчекин), :112-179 (`UpdateAttendeeInfo`), :249-289 (`BlockAttendee`), :292-325 (`UnblockAttendee`), :328-360 (`DeleteAttendee`); backend/internal/store/pg_store.go:503-523 (`UpdateAttendee`)
- Описание: Все перечисленные хендлеры выполняют паттерн read-modify-write без версии/блокировки: читают `attendee` через `GetAttendeeByID`, меняют один-два поля в памяти, затем вызывают `h.Store.UpdateAttendee(ctx, attendee)` — а этот метод одним `UPDATE ... WHERE id = $14` перезаписывает *все* отслеживаемые поля строки (`first_name..block_reason, custom_fields, deleted_at`) значениями из переданной в память структуры, без `WHERE updated_at = $old` / `SELECT ... FOR UPDATE` / оптимистичной версии.
- Влияние: Два параллельных запроса к одному и тому же участнику (например, два сотрудника сканируют один бейдж почти одновременно на входе — ровно сценарий "конкурентный чек-ин одного участника", или чек-ин гонится с одновременной блокировкой участника администратором) читают одну и ту же исходную строку, вносят разные изменения и оба вызывают `UpdateAttendee`; выигравший последним полностью перезаписывает результат первого без какой-либо ошибки конфликта. Для чек-ина это проявляется так: оба сотрудника получают `200 OK`, но в БД останется `checked_in_by` только одного из них, а любое другое одновременное изменение (например, `blocked`/`custom_fields`), сделанное в этот же момент другим запросом, будет молча отброшено. Никакого предупреждения о конфликте клиент не получает.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Перейти на частичные `UPDATE ... SET col = $1 WHERE id = $2` только для изменяемых полей конкретной операции (чек-ин отдельно от общего редактирования), и/или добавить оптимистичную блокировку (`WHERE id = $1 AND updated_at = $2`, возврат конфликта 409 при 0 affected rows), и/или атомарный `UPDATE ... SET checked_in_by = COALESCE(checked_in_by, $1), checked_in_at = COALESCE(checked_in_at, NOW()) WHERE id = $2 AND checkin_status = FALSE` для чек-ина.
- Вердикт: ПОДТВЕРЖДЕНО — `UpdateAttendee` (pg_store.go:503-523) действительно один `UPDATE` перезаписывает все отслеживаемые поля без `WHERE updated_at=...`/версии, и все 5 перечисленных хендлеров в attendees.go делают read-modify-write без блокировки.

### BACKEND-BUG-03: UpdateAttendee никогда не сохраняет колонку `code` — исправление билетного кода тихо не применяется
- Файл: backend/internal/store/pg_store.go:511-522 (SET-список `UpdateAttendee`); backend/internal/handler/attendees.go:162-164 (`UpdateAttendeeInfo`, `req.Code`)
- Описание: `UpdateAttendeeInfo` (`PATCH /api/attendees/:id`) поддерживает изменение `Code` — если `req.Code != nil`, в памяти делается `attendee.Code = *req.Code`, и результат (с новым кодом) возвращается клиенту в JSON с `200 OK`. Но `UPDATE attendees SET first_name = $1, ..., block_reason = $11, custom_fields = $12, deleted_at = $13, updated_at = NOW() WHERE id = $14` в `UpdateAttendee` (pg_store.go) не содержит `code = ...` — эта колонка ни разу не упомянута в SET-списке при полном сравнении со списком аргументов `Exec`.
- Влияние: Оператор мероприятия, пытающийся вручную исправить дублирующийся или ошибочно отсканированный билетный код участника через UI (единственный способ сделать это для уже созданного участника — `UpdateAttendeeInfo`), получает подтверждение об успехе с "новым" кодом в ответе, но в базе данных сохраняется прежнее значение `code`. Последующее чтение участника (обновление страницы, повторная генерация QR через `GetAttendeeQR`, чек-ин по коду через `ZoneCheckIn`/`GetAttendeeByCode`) снова показывает старый код — исправление дубликата/опечатки не срабатывает, а причина не очевидна ни оператору, ни в логах.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Добавить `code = $N` в SET-список `UpdateAttendee` (с осторожностью — учесть уникальный индекс `UNIQUE(event_id, code)` и вернуть `409 Conflict` при коллизии), либо явно документировать/блокировать попытку изменить `code` через этот путь на уровне хендлера, если это осознанное ограничение.
- Вердикт: ПОДТВЕРЖДЕНО — SET-список `UpdateAttendee` (pg_store.go:512-516) действительно не содержит `code`, при этом `UpdateAttendeeInfo` (attendees.go:162-164) меняет `attendee.Code` в памяти и возвращает его клиенту как будто сохранённое.

### BACKEND-BUG-04: Загрузка шрифта мероприятия никогда не проходит аутентификацию — неверное приведение типа контекста
- Файл: backend/internal/handler/fonts.go:51-70 (`UploadEventFont`); backend/internal/middleware/jwt.go:28-45 (`middleware.JWT()`)
- Описание: `middleware.JWT()` парсит токен и сохраняет claims в контексте как `*models.JWTCustomClaims`: `c.Set("user", claims)` (jwt.go:45), где `claims` — результат `jwt.ParseWithClaims(tokenString, &models.JWTCustomClaims{}, ...)`. Все остальные обработчики в проекте (32 места в 11 файлах, включая сам `fonts.go` в других функциях не тронуто, т.к. таких обращений там больше нет) читают это значение как `c.Get("user").(*models.JWTCustomClaims)`. Но `UploadEventFont` — единственное исключение — делает `token, ok := userToken.(*jwt.Token)` (fonts.go:58), а затем `token.Claims.(jwt.MapClaims)` (fonts.go:65) и читает `claims["user_id"]` как из карты. Поскольку реальный тип значения в контексте — `*models.JWTCustomClaims` (структура, не `*jwt.Token` и не `jwt.MapClaims`), приведение `userToken.(*jwt.Token)` завершается с `ok == false` при любом валидном токене.
- Влияние: `POST /api/events/:event_id/fonts` (загрузка кастомного шрифта для шаблона бейджа) всегда возвращает `401 {"error": "Invalid token"}` (fonts.go:60-63), даже с полностью корректным Bearer-токеном администратора нужного tenant'а. Функциональность загрузки собственных шрифтов для бейджей на 100% неработоспособна в текущем виде — воспроизводится в любой момент, без гоночных условий. Как следствие, отсутствие проверки принадлежности tenant в этом же хендлере (описано в backend-sec.md, SEC-09) на практике не эксплуатируется, поскольку запрос никогда не проходит этот код вообще.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Заменить блок извлечения пользователя на стандартный `claims, ok := c.Get("user").(*models.JWTCustomClaims)` и `uuid.Parse(claims.UserID)`, как во всех остальных хендлерах; удалить неиспользуемый импорт `jwt` из fonts.go, если он более не нужен.
- Вердикт: ПОДТВЕРЖДЕНО — `UploadEventFont` (fonts.go:58-70) действительно делает `userToken.(*jwt.Token)`, тогда как `middleware.JWT()` (jwt.go:45) кладёт в контекст `*models.JWTCustomClaims`; приведение типа гарантированно проваливается (`ok == false`) для любого валидного токена → всегда 401.

### BACKEND-BUG-05: Гонка при повторном/параллельном сканировании в ZoneCheckIn возвращает 500 вместо идемпотентного «уже отмечен»
- Файл: backend/internal/handler/zones.go:404-439 (`ZoneCheckIn`, шаги 7–8); backend/migrations/000010_event_zones.up.sql (`UNIQUE(attendee_id, zone_id, event_day)` на `zone_checkins`)
- Описание: `ZoneCheckIn` сначала проверяет отсутствие записи через `h.Store.CheckAttendeeZoneCheckin(...)` (не найдено → `nil, nil`), и только затем делает `h.Store.CreateZoneCheckin(...)`. Между этими двумя вызовами нет ни блокировки строки, ни транзакции — классический check-then-act. От дублирования строк в БД защищает уникальный индекс `UNIQUE(attendee_id, zone_id, event_day)`, поэтому данные не портятся, но конкурентный "проигравший" запрос (два одновременных скана одного бейджа на одном киоске, двойной тап, повтор из-за таймаута сети) получает от `CreateZoneCheckin` голую ошибку нарушения уникальности, которую хендлер (zones.go:434-439) интерпретирует как обычную ошибку и возвращает `500 {"error": "Failed to create check-in"}` — вместо `200 {"success": true, "message": "Already checked in"}`, как это сделано для последовательного дубликата чуть выше (zones.go:412-421).
- Влияние: При штатной ситуации гонки на входе (несколько стойках печатают/сканируют один и тот же QR почти одновременно) один из сотрудников видит внезапную "ошибку сервера" вместо корректного "уже зарегистрирован", что путает персонал на входе и может привести к повторным попыткам/эскалациям в момент пиковой нагрузки регистрации.
- Серьёзность: Medium
- Уверенность: средняя (требует конкурентных запросов для проявления, но конструкция кода делает это гарантированным при гонке)
- Рекомендация: В `CreateZoneCheckin`/`ZoneCheckIn` распознавать код ошибки Postgres `23505` (unique_violation) через `pgconn.PgError` и в этом случае повторно прочитать существующую запись и вернуть тот же идемпотентный "Already checked in" ответ, что и для последовательного случая.
- Вердикт: ПОДТВЕРЖДЕНО — подтверждён check-then-act (`CheckAttendeeZoneCheckin` → `CreateZoneCheckin` без транзакции/блокировки, zones.go:404-439), уникальный индекс `UNIQUE(attendee_id, zone_id, event_day)` подтверждён в migrations/000010_event_zones.up.sql:68, и обработчик ошибки (434-439) действительно не отличает unique_violation от прочих ошибок.

### BACKEND-BUG-06: `time.Truncate(24h)` используется как граница календарного дня — даёт смещённые сутки для мероприятий не в UTC
- Файл: backend/internal/store/pg_store_zones.go:133 (`GetEventZonesWithStats`, `today`), :414 (`GetZoneCheckins`, `dateOnly`), :472 (`CheckAttendeeZoneCheckin`, `dateOnly`); backend/internal/handler/zones.go:601-613 (`GetZoneDays`, `currentDay`/`endDay`/`is_today`/`is_past`/`is_future`)
- Описание: Go документирует `Time.Truncate` как операцию над абсолютной длительностью от нулевого времени (`0001-01-01 00:00:00 UTC`), а не над "настенным" представлением времени — то есть `t.Truncate(24*time.Hour)` всегда даёт полночь по UTC для соответствующих суток, независимо от `Location()` самого `t`. Во всех перечисленных местах это используется как замена вычисления "начала календарного дня" (`today := time.Now().Truncate(24*time.Hour)`, `dateOnly := date.Truncate(24*time.Hour)`, `event.StartDate.Truncate(24*time.Hour)`).
- Влияние: Если сервер и/или входные `time.Time` (например, `event.StartDate`/`EndDate`, введённые с локальным смещением организатора) не в UTC, граница "сегодня"/"день N" смещается на величину часового пояса. Конкретно: `TodayCheckins` в статистике зоны (`GetEventZonesWithStats`) и список дней мероприятия с флагами `is_today`/`is_past`/`is_future` (`GetZoneDays`) могут в течение нескольких часов вокруг местной полуночи показывать неверные сутки (например, для мероприятия в часовом поясе UTC+3..+12 "сегодняшние" чек-ины после полуночи по местному времени, но до полуночи UTC, не попадут в `TodayCheckins`, либо день будет посчитан как предыдущий/следующий из-за сдвига `StartDate`/`EndDate`).
- Серьёзность: Medium
- Уверенность: высокая (детерминированное поведение Go, легко воспроизводимо для любого события не в UTC+0)
- Рекомендация: Вычислять "начало суток" через `time.Date(t.Year(), t.Month(), t.Day(), 0, 0, 0, 0, loc)` с явной локацией мероприятия/тенанта (или хранить и сравнивать чистую `DATE` без времени, полученную с явным часовым поясом на стороне клиента), а не полагаться на `Truncate`.
- Вердикт: ПОДТВЕРЖДЕНО — `Truncate(24*time.Hour)` подтверждён именно на указанных строках (pg_store_zones.go:133,414,472; zones.go:602-613), поведение соответствует документированной семантике Go (усечение от нулевого времени в UTC, а не от локальной полуночи).

### BACKEND-BUG-07: SyncPush слепо принимает изменения клиента и глотает все ошибки; SyncPull никогда не сообщает об удалениях
- Файл: backend/internal/handler/sync.go:44-107 (`SyncPull`), :109-167 (`SyncPush`); backend/internal/store/pg_store_sync.go:47-84 (`GetAttendeesChangedSince`)
- Описание: (1) `SyncPush` для каждого обновлённого/созданного участника в `req.Changes.Attendees.Updated/Created` при любой ошибке `UpdateAttendee`/`CreateAttendee` делает `continue` (sync.go:141-144, 155-157) — без логирования и без включения в ответ; при этом эндпоинт всегда возвращает `200 {"status": "ok", ...}` независимо от того, сколько записей реально не сохранилось. Комментарий в коде прямо признаёт: "For simplicity, we always accept client updates in this MVP" (сравнение `updated_at` не делается вообще) — то есть более старая офлайн-версия записи безусловно перезаписывает более новую серверную (включая `checkin_status`/`blocked`, т.к. `UpdateAttendee` перезаписывает всю строку — см. BUG-02). Дополнительно, в отличие от `CreateAttendeeRequest`-пути (attendees.go:74-76), путь создания через sync не подставляет код, если `attendee.Code == ""` — два офлайн-устройства, создавшие разных участников без кода для одного события, столкнутся с нарушением `UNIQUE(event_id, code)`, и один из них будет молча пропущен (`continue`) без единого сигнала клиенту. (2) `SyncPull` всегда возвращает `Deleted: make([]string, 0)` и никогда не добавляет туда элементы (sync.go:78-84, 95-101); при этом `GetAttendeesChangedSince` при непустом `since` не фильтрует `deleted_at IS NULL` (pg_store_sync.go:52) — мягко удалённые участники попадают в `Updated`, а не в `Deleted`, то есть офлайн-клиент никогда явно не узнаёт об удалении участника на сервере.
- Влияние: Офлайн-мобильный сценарий (главный сценарий использования `sync.go`, судя по комментарию "most common use case: checking in") может незаметно откатить чек-ин/блокировку, сделанные на других устройствах, пока клиент был офлайн, без каких-либо признаков конфликта; часть записей может тихо не синхронизироваться вовсе, что противоречит самой цели офлайн-синхронизации — гарантировать согласованность после реконнекта.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Сравнивать `updated_at` сервера и клиента перед принятием изменения (skip / conflict, не "всегда клиент побеждает"); возвращать список неудачных ID из `SyncPush` в ответе; генерировать код по умолчанию для создаваемых через sync участников без кода, как в `CreateAttendeeRequest`; реализовать реальный список `Deleted` в `SyncPull` (фильтр `deleted_at IS NOT NULL AND deleted_at > since`).
- Вердикт: ПОДТВЕРЖДЕНО — подтверждены все элементы: `continue` без логирования на ошибках (sync.go:141-144,155-157), комментарий "always accept client updates" (135-138), `Deleted` всегда пустой список (78-84,95-101), и `GetAttendeesChangedSince` (pg_store_sync.go:47-52) не фильтрует `deleted_at IS NULL` в ветке `since != zero` (фильтр есть только в ветке `since.IsZero()`, строка 58).

### BACKEND-BUG-08: Диапазон дат в статистике использования тенанта исключает данные за последний день периода
- Файл: backend/internal/handler/super_admin.go:251-264 (`GetTenantUsage`); backend/internal/store/pg_store.go:1178-1201 (`GetUsageStats`)
- Описание: `GetTenantUsage` парсит `end_date` из query-параметра как `time.Parse("2006-01-02", end)`, что даёт полночь (00:00:00) указанной даты. `GetUsageStats` использует этот момент как верхнюю границу: `WHERE tenant_id = $1 AND logged_at BETWEEN $2 AND $3`.
- Влияние: Все записи `usage_logs`, попавшие в указанный конечный день после 00:00:00 (то есть весь день, кроме единственного тик-момента полуночи), исключаются из отчёта — при запросе статистики "с 1 по 9 июля" данные за 9 июля фактически не учитываются вовсе. Это касается отчёта об использовании ресурсов тенантом, который супер-админ использует, в частности, для контроля тарифных лимитов.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Приводить `end_date` к концу дня (`endDate.Add(24*time.Hour - time.Nanosecond)`) или использовать `logged_at < end_date + 1 day` вместо `BETWEEN`.
- Вердикт: ПОДТВЕРЖДЕНО — `GetTenantUsage` (super_admin.go:260-264) парсит `end_date` через `"2006-01-02"` (полночь), а `GetUsageStats` (pg_store.go:1178-1201) использует `BETWEEN $2 AND $3` — весь день после 00:00:00 исключается.

### BACKEND-BUG-09: Register создаёт tenant/user/членство в трёх нетранзакционных шагах — частичный сбой оставляет несогласованные данные
- Файл: backend/internal/handler/auth.go:39-115 (`Register`)
- Описание: `Register` последовательно выполняет: (1) `CreateTenant` (создаёт строку `tenants` безусловно), (2) при необходимости `CreateUser`, (3) `AddUserToTenant`. Ни один из этих шагов не обёрнут в транзакцию БД и не откатывает предыдущие шаги при ошибке последующего.
- Влияние: Если шаг (3) `AddUserToTenant` завершится ошибкой (временный сетевой сбой БД, нарушение constraint и т.п.) уже после того, как шаги (1)/(2) успешно закоммитились, в БД останется "осиротевший" tenant без единого пользователя, и/или пользователь без записи в `user_tenants` — при этом `Login` (auth.go:144-148) требует `len(tenants) > 0` из `GetUserTenants` (основано на `user_tenants`), так что такой пользователь не сможет войти обычным логином до повторной успешной регистрации.
- Серьёзность: Low
- Уверенность: средняя (требует сбоя ровно между шагами; воспроизводимо только точечной инъекцией ошибки/гонкой БД, но не бизнес-логикой запроса)
- Рекомендация: Выполнять шаги (1)-(3) в единой транзакции `pgx.Tx` с rollback при любой ошибке, аналогично `BulkUpdateZoneAccessRules` (pg_store_zones.go:247-280), которая уже реализована корректно.
- Вердикт: ПОДТВЕРЖДЕНО — `Register` (auth.go:39-115) действительно выполняет `CreateTenant`/`CreateUser`/`AddUserToTenant` как отдельные store-вызовы без общей транзакции; `BulkUpdateZoneAccessRules` подтверждена как корректный пример с `tx.Begin`/`tx.Commit`.

### BACKEND-BUG-10: CheckZoneAccess разыменовывает attendee без проверки на nil
- Файл: backend/internal/store/pg_store_zones.go:578-590 (`CheckZoneAccess`); backend/internal/store/pg_store.go:481-501 (`GetAttendeeByID`, возвращает `(nil, nil)` при отсутствии строки)
- Описание: `CheckZoneAccess` вызывает `attendee, err := s.GetAttendeeByID(ctx, attendeeID)` и проверяет только `if err != nil`, после чего сразу обращается к `attendee.Blocked`. Но `GetAttendeeByID` при отсутствии записи возвращает `(nil, nil)` (pg_store.go:489-492) — то есть `err == nil` и `attendee == nil` одновременно возможны, и код не защищён от этого случая.
- Влияние: Сейчас единственный вызывающий (`ZoneCheckIn`, zones.go:395) передаёт `attendee.ID`, полученный секундами ранее из `GetAttendeeByCode` с проверкой на nil, поэтому окно гонки крайне узкое (нужно, чтобы участник был удалён между двумя вызовами в рамках одного запроса). Однако `CheckZoneAccess` — экспортируемый метод интерфейса `Store`, и любой будущий или иной вызывающий код, передающий несуществующий `attendeeID` напрямую, вызовет панику nil pointer dereference и уронит текущий обработчик (перехватится recover-middleware Echo как 500, но без содержательного сообщения).
- Серьёзность: Low
- Уверенность: высокая (код виден напрямую; эксплуатируемость сейчас ограничена единственным безопасным вызывающим)
- Рекомендация: Добавить `if attendee == nil { return false, "Attendee not found", nil }` сразу после вызова `GetAttendeeByID`, не полагаясь на инварианты вызывающей стороны.
- Вердикт: ПОДТВЕРЖДЕНО — `CheckZoneAccess` (pg_store_zones.go:578-590) действительно обращается к `attendee.Blocked` сразу после проверки только `err != nil`, а `GetAttendeeByID` (pg_store.go:489-493) возвращает `(nil, nil)` при отсутствии строки — nil pointer dereference технически возможен.

### BACKEND-BUG-11: GenerateAttendeeCodes тихо пропускает участников при коллизии сгенерированного кода
- Файл: backend/internal/handler/attendee_codes.go:35-46 (`GenerateAttendeeCodes`)
- Описание: Для каждого участника без кода вызывается `generateUniqueCode()` (8 hex-символов, ~32 бита энтропии) и сразу `h.Store.UpdateAttendee` — без проверки, что сгенерированный код уже не используется другим участником того же события (в отличие от `BulkCreateAttendees` в bulk_import.go, где такая проверка есть через карту `existingCodes`). При коллизии `UpdateAttendee` вернёт ошибку нарушения `UNIQUE(event_id, code)`, которая только логируется (`c.Logger().Errorf`) и приводит к `continue` (attendee_codes.go:41-43) — сам HTTP-ответ (`200 {"status": "success", "updated_count": N}`) не содержит никакого указания на то, что часть участников осталась без кода.
- Влияние: Редкий, но реальный сценарий — при массовой генерации кодов для мероприятия с сотнями/тысячами участников без кода вероятность коллизии 8-значного кода растёт; такой участник останется с пустым `code`, что впоследствии сломает генерацию QR/бейджа и чек-ин именно для него, а администратор увидит только "Generated codes for N attendees" без индикации, что N меньше ожидаемого и почему.
- Серьёзность: Low
- Уверенность: высокая
- Рекомендация: Повторять генерацию кода в цикле при конфликте (аналогично bulk_import.go:157-162) вместо однократной попытки, и включать в ответ список ID участников, для которых код не удалось сгенерировать.
- Вердикт: ПОДТВЕРЖДЕНО — `GenerateAttendeeCodes` (attendee_codes.go:35-46) действительно вызывает `generateUniqueCode()` без проверки коллизий и на ошибке `UpdateAttendee` только логирует и делает `continue`, не отражая это в ответе клиенту.
