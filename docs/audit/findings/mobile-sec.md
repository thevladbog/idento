# MOBILE-SEC — mobile/ (Kotlin Multiplatform: shared, android-app, iosApp), БЕЗОПАСНОСТЬ

### MOBILE-SEC-01: Cleartext-трафик разрешён глобально, приложение по умолчанию всегда обращается к HTTP-адресу, production HTTPS URL не используется нигде
- Файл: mobile/android-app/app/src/main/AndroidManifest.xml:39; mobile/shared/src/commonMain/kotlin/com/idento/data/network/NetworkConstants.kt:8-10,32; mobile/shared/src/androidMain/kotlin/com/idento/data/network/NetworkConstants.android.kt:3; mobile/shared/src/iosMain/kotlin/com/idento/data/network/NetworkConstants.ios.kt:3; mobile/shared/src/commonMain/kotlin/com/idento/di/AppModule.kt:41-50; mobile/android-app/app/src/main/java/com/idento/di/NetworkModule.kt:22
- Описание: `android:usesCleartextTraffic="true"` разрешает нешифрованный HTTP для всего приложения, при этом ни `android:networkSecurityConfig` в манифесте, ни файл `network_security_config.xml` в проекте не заданы (не ограничивает cleartext конкретными dev-хостами). Реальная проводка (wiring) подтверждает, что это не декларативная заглушка: `getDefaultBaseUrl()` на Android жёстко возвращает `NetworkConstants.DEV_BASE_URL = "http://10.0.2.2:8080"`, на iOS — `IOS_DEV_BASE_URL = "http://localhost:8080"`; `AppModule.kt` использует именно `getDefaultBaseUrl()` при создании `ApiClient`. Константа `PROD_BASE_URL = "https://api.idento.app"` объявлена, но не используется НИГДЕ в коде (grep по `PROD_BASE_URL` не находит использований, кроме объявления) — нет build-варианта/флага окружения, который переключал бы приложение на боевой HTTPS-эндпоинт. Легаси-модуль `android-app` идёт тем же путём: `NetworkModule.kt:22` содержит `BASE_URL = "http://10.0.2.2:8080/"` без флейворов debug/release.
- Влияние: Если текущая сборка (или обновлённая версия с реальным prod-хостом, но без пересмотра этих же настроек) попадёт в прод, JWT-токен, логин/пароль и PII посетителей будут передаваться в открытом виде — перехват по MITM (публичный Wi-Fi, скомпрометированный роутер) раскрывает учётные данные и токен сессии. Даже сейчас отсутствие Network Security Config означает, что cleartext разрешён приложением ко ЛЮБОМУ хосту, а не только к loopback-адресам разработки.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Убрать `usesCleartextTraffic="true"`, добавить `network_security_config.xml`, разрешающий cleartext только для `10.0.2.2`/`localhost` в debug-сборке (через `debug-overrides`). Завести build-flavor'ы/переменные окружения, которые реально переключают `baseUrl` на `PROD_BASE_URL` в release-сборках, и убрать возможность собрать release с dev-URL.
- Вердикт: ПОДТВЕРЖДЕНО — все цитаты подтверждены построчно: манифест (39), `NetworkConstants*.kt` жёстко возвращают dev-URL, `AppModule.kt:41-50` использует `getDefaultBaseUrl()`, `PROD_BASE_URL` встречается только в своём объявлении (grep не нашёл использований), `network_security_config.xml` в проекте отсутствует.

### MOBILE-SEC-02: Полное логирование тела и заголовков HTTP-запросов/ответов (включая JWT и пароль при логине) без отключения в релизе
- Файл: mobile/android-app/app/src/main/java/com/idento/di/NetworkModule.kt:33-35; mobile/shared/src/commonMain/kotlin/com/idento/data/network/ApiClient.kt:40-43; mobile/android-app/app/src/main/java/com/idento/data/api/AuthInterceptor.kt:29-32; mobile/android-app/app/src/main/java/com/idento/data/model/LoginRequest.kt:5-19
- Описание: В легаси Android-модуле `HttpLoggingInterceptor.Level.BODY` включён безусловно (`NetworkModule.kt:33-35`) — этот уровень OkHttp логирует ПОЛНОСТЬЮ заголовки и тело каждого запроса/ответа в Logcat, включая заголовок `Authorization: Bearer <token>` (добавляется `AuthInterceptor.kt:29-32`) и тело `POST /auth/login` с `email`/`password` в открытом виде (`LoginRequest.kt`), а также тело ответа с выданным JWT (`LoginResponse.token`). В общем KMP-модуле аналогично: Ktor-плагин `Logging` включён с `level = LogLevel.BODY` без всяких условий (`ApiClient.kt:40-43`), что также печатает тела запросов/ответов через `println`, попадающие в Logcat на Android (тег `System.out`) и в консоль на iOS. Ни в одном месте нет проверки `BuildConfig.DEBUG`/аналога, и `proguard-rules.pro` не содержит правил, вырезающих логирование — конфигурация одинакова для debug и release (`isMinifyEnabled = true` в release не удаляет вызовы стороннего логирующего интерцептора).
- Влияние: На релизной сборке пароль пользователя при логине и текущий JWT-токен (в каждом последующем запросе — через заголовок Authorization в OkHttp-варианте, и как минимум в теле ответа логина в Ktor-варианте) регулярно попадают в системный лог устройства. Логи доступны через `adb logcat` при физическом/USB-доступе к разблокированному устройству, через баг-репорты, вложения в тикеты поддержки, а также сторонним приложениям с соответствующими правами на старых версиях Android. Это прямой путь кражи учётных данных и сессии без необходимости root/jailbreak.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Убрать логирование тела/заголовков в production (использовать `HttpLoggingInterceptor.Level.NONE`/`BASIC` в релизе через `BuildConfig.DEBUG`, аналогично для Ktor — `LogLevel.NONE` в релизе или явную маску полей `Authorization`/`password`). Никогда не логировать сырые пароли и токены даже в debug.
- Вердикт: ПОДТВЕРЖДЕНО — `NetworkModule.kt:33-35`/`ApiClient.kt:40-43` безусловно включают `Level.BODY`/`LogLevel.BODY`, `AuthInterceptor.kt:29-32` добавляет заголовок с токеном, grep по `BuildConfig.DEBUG` во всём `mobile/` не даёт совпадений, `proguard-rules.pro` логирование не вырезает.

### MOBILE-SEC-03: JWT-токен и данные пользователя хранятся в незашифрованном виде на диске (Android DataStore-файл и iOS Documents-файл), без Keystore/Keychain
- Файл: mobile/shared/src/androidMain/kotlin/com/idento/data/storage/DataStoreFactory.android.kt:14-20; mobile/shared/src/iosMain/kotlin/com/idento/data/storage/DataStoreFactory.ios.kt:17-36; mobile/shared/src/commonMain/kotlin/com/idento/data/preferences/AuthPreferences.kt:76-83; mobile/android-app/app/src/main/java/com/idento/di/DataStoreModule.kt:14; mobile/android-app/app/src/main/java/com/idento/data/local/TokenManager.kt:34-40
- Описание: `AuthPreferences.saveAuthToken()` кладёт JWT как обычный `stringPreferencesKey("auth_token")` в Jetpack `PreferenceDataStore`. На Android этот DataStore создаётся через `PreferenceDataStoreFactory.createWithPath { File(context.filesDir, "datastore/$fileName.preferences_pb") }` — это обычный файл в `files/datastore/`, не `EncryptedSharedPreferences`/Android Keystore. На iOS `DataStoreFactory.ios.kt` создаёт тот же Preference DataStore поверх обычного файла в `NSHomeDirectory()/Documents/$fileName` — не iOS Keychain. Легаси-модуль `android-app` (Hilt DI, `DataStoreModule.kt` + `TokenManager.kt`) использует тот же паттерн (`preferencesDataStore(name = "idento_preferences")` — обычный файл, не Keystore-backed хранилище).
- Влияние: На Android — токен и email/имя/роль пользователя читаемы кем угодно с доступом к файловой системе приложения (rooted-устройство, `adb backup` для debuggable/legacy-сборок, вредоносный код с тем же UID через уязвимость ОС, форензик-извлечение образа). На iOS — файл в Documents попадает в незашифрованном виде в стандартный бэкап iTunes/Finder (если сам бэкап не зашифрован пользователем) и доступен на джейлбрейкнутом устройстве или через инструменты извлечения файловой системы; Keychain для этого не используется, хотя является стандартным безопасным хранилищем для секретов на iOS. Компрометация токена даёт полный доступ к аккаунту до истечения JWT.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: На Android — хранить токен через `EncryptedSharedPreferences`/Jetpack Security Crypto (AES-256, ключ в Android Keystore) либо шифровать DataStore. На iOS — хранить токен в Keychain (`kSecClassGenericPassword`) вместо обычного файла в Documents. Прочие пользовательские данные (email/name/role) — хотя бы исключить из бэкапов, если не переносить в защищённое хранилище целиком.
- Вердикт: ПОДТВЕРЖДЕНО — `DataStoreFactory.android.kt`/`.ios.kt` создают DataStore поверх обычных файлов (`filesDir`/`Documents`), legacy `TokenManager.kt`/`DataStoreModule.kt` — тот же паттерн; grep по `EncryptedSharedPreferences|MasterKey|Keychain|Keystore` по всему `mobile/` не даёт ни одного совпадения.

### MOBILE-SEC-04: Правила бэкапа Android не приведены в соответствие с реальным расположением данных — отсутствует осознанная защита при будущих изменениях хранилища
- Файл: mobile/android-app/app/src/main/AndroidManifest.xml:32-34; mobile/android-app/app/src/main/res/xml/backup_rules.xml:1-5; mobile/android-app/app/src/main/res/xml/data_extraction_rules.xml:1-7
- Описание: `android:allowBackup="true"` включён, а `backup_rules.xml`/`data_extraction_rules.xml` содержат только `<include domain="sharedpref" path="."/>` и `<include domain="database" path="."/>`. Начиная с использования только `<include>`-правил, поведение бэкапа Android становится «белым списком» — включаются ТОЛЬКО перечисленные домены. Приложение при этом нигде не использует классический `SharedPreferences` (везде DataStore, который физически лежит в домене `file`, не `sharedpref`) и не имеет ни одной реализации `@Database`/Room (в `OfflineDatabaseImpl` — TODO/заглушка в памяти, реальной SQLite/Room БД нет). То есть сейчас правила бэкапа технически не покрывают ни токен (лежит в `file`-домене, не включён), ни что-либо в `database`-домене (там ничего нет) — но это результат случайного несовпадения, а не осознанного проектного решения по защите auth-данных.
- Влияние: Как только кто-то добавит классический `SharedPreferences` или реализует заглушенный Room/SQLite офлайн-кэш посетителей (уже запланировано, судя по TODO в коде), эти данные автоматически попадут в облачный бэкап и в трансфер при смене устройства — включая потенциальный auth-токен или персональные данные посетителей (PII) — без отдельного пересмотра `backup_rules.xml`. Сейчас конкретной утечки токена через бэкап нет, но конфигурация не создаёт для неё преграды на будущее.
- Серьёзность: Low
- Уверенность: средняя
- Рекомендация: Явно исключить чувствительные данные (`domain="file" path="datastore/"` с `<exclude>`, если DataStore когда-либо попадёт в бэкапящийся домен) или отключить `allowBackup` совсем для клиента, работающего с чувствительными токенами/PII. Пересматривать `backup_rules.xml`/`data_extraction_rules.xml` при каждом изменении слоя хранения.
- Вердикт: ПОДТВЕРЖДЕНО — `allowBackup=true` и правила с доменами `sharedpref`/`database` подтверждены построчно; в коде нет ни одного `SharedPreferences`/`@Database`/`@Entity`, `OfflineDatabaseImpl` — in-memory заглушка, так что вывод о случайном (а не осознанном) несовпадении корректен, severity Low обоснован.

### MOBILE-SEC-05: Избыточное разрешение RECEIVE_BOOT_COMPLETED без соответствующего получателя
- Файл: mobile/android-app/app/src/main/AndroidManifest.xml:28
- Описание: Манифест объявляет `<uses-permission android:name="android.permission.RECEIVE_BOOT_COMPLETED" />` с комментарием «Generic scanner access», но в манифесте нет `<receiver>` с intent-filter на `android.intent.action.BOOT_COMPLETED`, и по коду приложения (`android-app/app/src`) не найдено ни одного `BroadcastReceiver`, обрабатывающего это событие.
- Влияние: Минимальный прямой риск (разрешение без реализации не даёт атакующему новых возможностей), но нарушает принцип наименьших привилегий: приложение запрашивает у пользователя лишнее разрешение, что увеличивает поверхность запроса доверия при установке и может быть флагом при ревью в сторах/MDM-политиках предприятия.
- Серьёзность: Low
- Уверенность: высокая
- Рекомендация: Удалить неиспользуемое разрешение `RECEIVE_BOOT_COMPLETED`, либо (если автозапуск на самом деле нужен для аппаратных сканеров) реализовать соответствующий `BroadcastReceiver` и задокументировать назначение.
- Вердикт: ПОДТВЕРЖДЕНО — разрешение объявлено на строке 28 манифеста; ни в манифесте, ни в коде (`HardwareScannerService.kt`/`BluetoothScannerService.kt` регистрируют receiver'ы только для сканер-специфичных action'ов) нет обработчика `BOOT_COMPLETED`.

### MOBILE-SEC-06: Logout — только локальная очистка, серверная инвалидация токена не выполняется
- Файл: mobile/shared/src/commonMain/kotlin/com/idento/data/repository/AuthRepository.kt:84-91; mobile/shared/src/commonMain/kotlin/com/idento/data/network/AuthApiService.kt:92-98
- Описание: `AuthRepository.logout()` вызывает только `authPreferences.clearAuth()` (удаляет токен локально) и никогда не обращается к `AuthApiService.logout()`. Сам `AuthApiService.logout()` при этом — пустая заглушка (`runCatching {}` без единого сетевого вызова), даже если бы её вызывали.
- Влияние: Если токен уже был скомпрометирован (см. MOBILE-SEC-02 — токен и логин/пароль постоянно логируются) или устройство утеряно/украдено до выхода из аккаунта, простой logout в приложении не отзывает токен на сервере — скомпрометированный JWT остаётся действительным до истечения срока его действия (exp), даже после того как пользователь «вышел» из приложения.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: На logout выполнять реальный вызов к backend-эндпоинту инвалидации/отзыва токена (или сессии/refresh-токена), если такой существует или может быть добавлен на бэкенде; в дополнение сократить время жизни access-токена и обеспечить его ротацию.
- Вердикт: ПОДТВЕРЖДЕНО — `AuthRepository.kt:84-91` вызывает только `clearAuth()`, `AuthApiService.kt` (строки ~95-98) содержит пустой `runCatching {}` без сетевого вызова; grep подтверждает, что `AuthApiService.logout()` нигде не вызывается.
