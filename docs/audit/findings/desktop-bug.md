# DESKTOP-BUG — desktop/src/ + desktop/src-tauri/, КОРРЕКТНОСТЬ / БАГИ

### DESKTOP-BUG-01: CSP `connect-src` жёстко ограничен `localhost:8008`, ломает работу с настраиваемым backend URL
- Файл: desktop/src-tauri/tauri.conf.json:18 (совместно с desktop/src/lib/config.ts:1-21 и desktop/src/pages/Connection.tsx:26-50)
- Описание: `app.security.csp.connect-src` в конфиге Tauri задан как `'self' ipc: http://ipc.localhost http://localhost:8008 http://127.0.0.1:8008` — то есть fetch/XHR из WebView разрешены только к двум литеральным хостам. При этом приложение явно поддерживает произвольный адрес бэкенда: `ConnectionPage` (`Connection.tsx`) даёт оператору киоска ввести любой URL и сохранить его через `setBackendUrl()` (`config.ts`), после чего `api.ts` использует именно этот адрес как `baseURL` для всех запросов. Если сохранённый URL отличается от `http(s)://localhost:8008` / `http(s)://127.0.0.1:8008` (что для реального продакшен-развёртывания с centralized backend — обычный случай), WebView будет молча блокировать все сетевые запросы по политике CSP.
- Влияние: на любом киоске, где бэкенд не находится буквально на `localhost:8008` того же устройства, весь функционал (логин, чек-ин, загрузка списка участников) перестаёт работать; `ConnectionPage.checkConnection()` и все последующие вызовы `api` будут падать с сетевой ошибкой, которая выглядит как «сервер недоступен», хотя реальная причина — блокировка CSP, а не проблема сети/сервера. Это ломает ключевой сценарий настройки и эксплуатации киоска.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Либо убрать фиксированные хосты из `connect-src` и разрешить произвольный адрес (например, задавать CSP динамически в зависимости от сохранённого `idento_backend_url`, или использовать `tauri://` proxy/allowlist по паттерну), либо сделать все обращения к бэкенду через Rust-команду (аналогично `agent_request`), чтобы обойти CSP браузера.
- Вердикт: ПОДТВЕРЖДЕНО — `tauri.conf.json:18` точно содержит только `localhost:8008`/`127.0.0.1:8008`; `Connection.tsx`/`config.ts` подтверждённо реализуют произвольный backend URL, других CSP-override файлов в репозитории нет.

### DESKTOP-BUG-02: Список участников загружается один раз и не синхронизируется — двойной чек-ин при нескольких киосках на одно событие
- Файл: desktop/src/pages/CheckinEvent.tsx:84-128 (загрузка), 224-233 (`lookupByCode`), 251-283 (обработка чек-ина)
- Описание: `attendees` загружается один раз в `useEffect` с зависимостью только от `eventId` (строки 84-128) и никогда не перезапрашивается повторно (нет polling/пересинка, нет инвалидации по таймеру). После успешного чек-ина локальное состояние обновляется только через `setAttendees(prev => prev.map(...))` (строка 255) — то есть источник истины о `checkin_status` для решения «первый это чек-ин или повторный» (`isFirstCheckin`, строка 251) целиком берётся из локально закэшированного снимка. Если на одном мероприятии одновременно работает несколько экземпляров этого киоска (типичный сценарий для ивентов с несколькими стойками регистрации), чек-ин, выполненный на одной станции, никак не отражается в памяти другой станции.
- Влияние: один и тот же участник может быть отмечен как «первый чек-ин» (зелёный экран) сразу на нескольких станциях, потому что каждая работает со своим устаревшим локальным снимком `checkin_status=false`. При включённой автопечати (`checkinSettings.printEnabled && !manualPrint`, строка 275) это приводит к повторной печати бейджа/пропуска для одного и того же человека. Дополнительно: участники, зарегистрированные после старта киоска (пока он работает много часов без перезапуска), не находятся через `lookupByCode` вплоть до перезахода на страницу события.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Периодически перезапрашивать список участников (короткий polling, например раз в 10-30 секунд) или подписываться на серверные события/изменения статуса; как минимум — после каждого успешного PUT `checkin_status` перепроверять актуальный статус участника с сервера перед показом «первый/повторный» результата, чтобы избежать гонки между несколькими станциями.
- Вердикт: ПОДТВЕРЖДЕНО — код соответствует описанию (`attendees` грузится один раз, `isFirstCheckin` в CheckinEvent.tsx:251 вычисляется до ответа сервера); проверка `backend/internal/handler/attendees.go` (`UpdateAttendeeHandler`) показала, что PUT всегда отвечает 200 без флага "уже отмечен" и без сравнения с прежним статусом — сервер не даёт фронтенду никакой опоры для обнаружения гонки.

### DESKTOP-BUG-03: Проверка доступности локального агента выполняется один раз при монтировании — нет переподключения при сбое/гонке запуска
- Файл: desktop/src/pages/CheckinEvent.tsx:130-132, desktop/src/pages/Equipment.tsx:123-154, desktop/src-tauri/src/lib.rs:15-29
- Описание: `agentConnected` вычисляется единственным вызовом `checkAgentHealth().then(setAgentConnected)` внутри `useEffect(..., [])` (зависимости пустые) — как на странице чек-ина, так и на странице оборудования. Никакого периодического повторного опроса или ретрая нет. На Rust-стороне сайдкар `idento-agent` запускается один раз в `setup()` (`lib.rs:15-29`); если `spawn()` вернёт ошибку — она только логируется (`log::error!`), процесс продолжает работать без агента и без попыток перезапуска. Комбинация даёт как минимум две проблемы: (1) гонка при старте — если фронтенд успевает вызвать `checkAgentHealth()` раньше, чем поднимется HTTP-сервер сайдкара (антивирус, холодный диск, медленный старт на Windows), `agentConnected` навсегда остаётся `false` для этого монтирования страницы; (2) если агент упадёт/будет перезапущен ОС в процессе работы киоска (многочасовая сессия), фронтенд никогда не узнает об этом и не восстановит соединение.
- Влияние: режим «сканер» (`isScannerMode`, зависит от `agentConnected`, строки 135-138 и 207-222) перестаёт слушать сканы, печать бейджей выдаёт `agentNotConnected` (строка 326-329) — и это состояние не восстанавливается автоматически; требуется, чтобы оператор вручную перешёл на другую страницу и вернулся (ремонтирование компонента), что противоречит требованию необслуживаемой непрерывной работы киоска.
- Серьёзность: High
- Уверенность: высокая
- Рекомендация: Опрашивать `checkAgentHealth()` периодически (например, раз в 5-15 секунд) вместо одного вызова при монтировании; на Rust-стороне добавить наблюдение за процессом сайдкара (`on_exit`/monitor) и автоматический перезапуск при неожиданном завершении.
- Вердикт: ПОДТВЕРЖДЕНО — `CheckinEvent.tsx:130-132` и `Equipment.tsx` действительно вызывают `checkAgentHealth()` один раз в `useEffect(..., [])`; `lib.rs:15-29` запускает сайдкар единожды в `setup()` и лишь логирует ошибку через `log::error!` без ретрая.

### DESKTOP-BUG-04: Отсутствие таймаута у HTTP-клиента и отсутствие повторных попыток после ошибки загрузки — риск зависания киоска
- Файл: desktop/src/lib/api.ts:12-16, desktop/src/pages/CheckinEvent.tsx:84-128, 494-505
- Описание: `axios.create({ baseURL: getBackendUrl() })` не задаёт `timeout`, поэтому запрос может «зависнуть» на неопределённое время при обрыве связи, когда TCP-соединение не получает ни ответа, ни RST (частая ситуация в киосках за NAT/прокси при потере сети). Одновременно на `CheckinEventPage` при ошибке первичной загрузки события/участников (`loadError`, строки 112-128 и рендер 498-505) пользователю показывается только кнопка «Назад» — нет автоматического повтора запроса и нет таймера, который бы сам восстановил загрузку при восстановлении сети.
- Влияние: при кратковременной проблеме сети именно в момент открытия страницы чек-ина (например, во время утреннего запуска киоска, пока Wi-Fi ещё не поднялся) экран навсегда остаётся в состоянии «Loading…» (если запрос завис) либо в состоянии ошибки без ретрая (если запрос быстро зафейлился) — киоск не восстанавливается сам и требует ручного вмешательства персонала, что противоречит требованию к необслуживаемой работе.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Задать разумный `timeout` для `axios.create(...)` (например, 10-15 c) и добавить автоматический повтор загрузки события/участников с экспоненциальной задержкой при `loadError`, а не полагаться на ручную навигацию оператора.
- Вердикт: ПОДТВЕРЖДЕНО — `api.ts:12-16` (`axios.create`) действительно не задаёт `timeout`; рендер `loadError` (CheckinEvent.tsx:498-505) показывает только кнопку «Назад» без авто-ретрая, как описано.

### DESKTOP-BUG-05: Несогласованная валидация пустого/невалидного QR между режимами камеры и сканера
- Файл: desktop/src/pages/CheckinEvent.tsx:188-192 (режим камеры), 210-216 (режим сканера), 224-233 (`lookupByCode`)
- Описание: В режиме сканера перед обработкой явно проверяется, что код после `trim()` не пуст: `if (data.code && data.code.trim())` (строка 212). В режиме камеры такой проверки нет: `if (code && code.data) { handleScanCodeRef.current(code.data.trim()); return; }` (строки 189-190) проверяет только исходную (нетримленную) строку — если `jsQR` вернёт данные, состоящие только из пробельных символов, после `.trim()` в `handleScanCodeRef` попадёт пустая строка. `lookupByCode` (строка 227-230) ищет совпадение через `(a.code ?? "").toLowerCase() === c || (a.id ?? "").toLowerCase() === c` — то есть пустая строка запроса потенциально совпадёт с любым участником, у которого поле `code` не задано (`undefined`/`null` → `""`).
- Влияние: в текущей реализации backend (`backend/internal/handler/attendees.go:74-75`) всегда генерирует `code`, если он пуст, поэтому на практике риск ложного чек-ина невысок — скорее всего просто будет показано «не найдено». Но это два независимых дефекта защиты: (1) путь камеры не защищён так же, как путь сканера, и (2) `lookupByCode` не отбрасывает явно пустой поисковый запрос, полагаясь только на гарантию бэкенда. Любое расхождение (ручное редактирование БД, старые записи, будущие изменения бэкенда) может привести к ложному чек-ину первого участника без кода при сканировании смятого/повреждённого QR.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Добавить в путь камеры ту же проверку `code.data.trim()` перед вызовом `handleScanCodeRef`, и явно отклонять пустую строку в `lookupByCode` (`if (!c) return undefined;`) вместо того, чтобы полагаться на гарантии стороннего сервиса.
- Вердикт: ПОДТВЕРЖДЕНО — камера (`CheckinEvent.tsx:189-190`) проверяет нетримленное `code.data`, тогда как сканер (`:212`) проверяет `.trim()`; `lookupByCode` (`:227-230`) не отклоняет пустую строку; `backend/internal/handler/attendees.go:74-75` подтверждён — код генерируется автоматически только в `CreateAttendee`, так что для других путей создания записи гарантия не абсолютна.

### DESKTOP-BUG-06: Отсутствует защита жизненного цикла окна киоска — нет автоматического полноэкранного режима, окно закрываемо, нет наблюдателя за процессом
- Файл: desktop/src-tauri/tauri.conf.json:23-32, desktop/src/pages/CheckinEvent.tsx:311-321
- Описание: Главное окно (`app.windows[0]`) не задаёт `"fullscreen": true`, `"decorations": false` или `"closable": false` — то есть по умолчанию у окна есть системная рамка/кнопки свернуть-закрыть, и приложение стартует НЕ в полноэкранном режиме; оператору/посетителю нужно каждый раз вручную нажимать кнопку разворачивания (`toggleFullscreen`, строка 311-321). Ни в Rust (`lib.rs`), ни во фронтенде нет обработчика `onCloseRequested`/подтверждения закрытия и нет процесса-наблюдателя (watchdog), который бы перезапускал приложение, если оно было случайно закрыто/свёрнуто. Дополнительно, локальное состояние `isFullscreen` инициализируется `false` при каждом монтировании `CheckinEventPage` независимо от реального состояния окна (реальное состояние спрашивается только в момент клика на кнопку toggleFullscreen, строка 314-316), поэтому после ремонтирования страницы (например, переход `/checkin` → `/checkin/:eventId`) иконка кнопки может не совпадать с фактическим состоянием окна.
- Влияние: посетитель или сотрудник может случайно закрыть/свернуть окно приложения (клик по системной кнопке, Alt+F4/Cmd+Q), либо кликнуть за пределы киоска на рабочий стол ОС, и ничто не восстановит киоск в рабочее состояние автоматически — станция чек-ина «молча» перестаёт работать до вмешательства персонала. Несовпадение иконки развернуть/свернуть — второстепенная, но сбивающая с толку деталь при диагностике на месте.
- Серьёзность: Medium
- Уверенность: высокая
- Рекомендация: Задать в `tauri.conf.json` `fullscreen: true` и `decorations: false` (и/или `closable: false`) для окна `main` в продакшен-сборке; на Rust-стороне подписаться на `WindowEvent::CloseRequested`, чтобы блокировать/подтверждать закрытие, и рассмотреть системный watchdog (авто-рестарт процесса) для действительно необслуживаемого киоска.
- Вердикт: ПОДТВЕРЖДЕНО — `tauri.conf.json:23-32` не задаёт fullscreen/decorations; `isFullscreen` инициализируется `false` (`CheckinEvent.tsx:67`) и сверяется с реальным состоянием окна только по клику (`toggleFullscreen`, :311-321), что подтверждает несовпадение иконки после ремонтирования.
