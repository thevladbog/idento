# AGENT-SEC — Принтер-агент (agent/), БЕЗОПАСНОСТЬ

Проверено: `agent/main.go` (единственная точка входа — HTTP-сервер и все обработчики),
`agent/internal/printer/*.go`, `agent/internal/scanner/*.go`, `agent/README.md`,
`agent/openapi.yaml`, а также клиентский код `web/src/lib/agent.ts` и
`desktop/src/lib/agent.ts` (как агент реально вызывается), чтобы подтвердить
отсутствие каких-либо заголовков аутентификации со стороны легитимных клиентов.
Также проверено поведение `github.com/rs/cors` v1.11.1 (модуль
`github.com/rs/cors@v1.11.1/cors.go`) и `go.bug.st/serial` v1.6.4, чтобы не
голословно утверждать, а показать код, подтверждающий эксплуатируемость.

---

### AGENT-SEC-01: Полное отсутствие аутентификации на всех эндпоинтах агента
- Файл: agent/main.go:234-997 (весь `http.NewServeMux()` — в частности `/print` :488,
  `/print-pdf` :542, `/printers/add` :354, `/printers/remove` :428,
  `/printers/default` (POST) :280, `/scanners/add` :806, `/scanners/remove` :927)
- Описание: Ни один обработчик не проверяет какой-либо токен, API-ключ, сессию или
  иной секрет. Поиск по всему `agent/` на `Authorization|Bearer|ApiKey|token|secret`
  не дал ни одного совпадения. Клиентский код (`web/src/lib/agent.ts`,
  `desktop/src/lib/agent.ts`) тоже не отправляет никаких auth-заголовков — то есть
  аутентификация не предусмотрена архитектурой вообще, а не просто не реализована
  на этом эндпоинте.
- Влияние: Любой процесс/пользователь, способный отправить HTTP-запрос на порт
  агента (по умолчанию 12345), может: напечатать произвольный ZPL/PDF на физическом
  принтере оператора (`/print`, `/print-pdf`), добавить/удалить сетевые принтеры
  (`/printers/add`, `/printers/remove`), подменить принтер по умолчанию
  (`/printers/default`), открыть/закрыть serial-порты сканеров (`/scanners/add`,
  `/scanners/remove`). В сочетании с AGENT-SEC-02 (бинд на все интерфейсы) это
  эксплуатируется соседом по сети без браузера вообще (обычный `curl`); в сочетании
  с AGENT-SEC-03 (CORS не блокирует запрос) — вредоносной веб-страницей в браузере
  оператора.
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: Ввести обязательную аутентификацию (например, случайный
  локальный токен, генерируемый при старте агента и передаваемый десктоп/веб
  клиенту через доверенный канал — файл с правами 0600, IPC, или заголовок,
  генерируемый при пейринге), проверяемую на каждом мутирующем и раскрывающем
  данные эндпоинте, а не полагаться на CORS/сетевую топологию как единственный
  барьер.

### AGENT-SEC-02: HTTP-сервер слушает на всех интерфейсах (0.0.0.0), а не только localhost
- Файл: agent/main.go:1045 (`Addr: ":" + *port,` внутри `&http.Server{...}` и вызов
  `server.ListenAndServe()` на строке 1052)
- Описание: `Addr` задан как `":12345"` без указания хоста. Согласно семантике
  `net/http`/`net.Listen`, пустой хост означает прослушивание на всех доступных
  сетевых интерфейсах (IPv4 `0.0.0.0` и IPv6 `::`), а не только на loopback
  (`127.0.0.1`). Нигде в коде (main.go, флаги `port`/`mock`) нет опции ограничить
  бинд localhost-ом.
- Влияние: Любое устройство в той же Wi-Fi/LAN-сети, где работает машина оператора
  (например, в кафе, коворкинге, на выставке/мероприятии — типичное место работы
  чек-ин агента), может напрямую обратиться к API агента по IP машины оператора,
  минуя браузер и вообще без учёта CORS (CORS — браузерный механизм, для прямых
  TCP-запросов не действует). В сочетании с AGENT-SEC-01 это даёт полный
  неаутентифицированный контроль над печатью и сканерами с любого устройства в
  локальной сети.
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: Явно биндиться на `127.0.0.1:<port>` (или `localhost:<port>`) по
  умолчанию; если требуется доступ с других устройств в сети — делать это по
  явному флагу с одновременным включением аутентификации.

### AGENT-SEC-03: CORS не является реальной защитой — сайт-злоумышленник может вызвать печать/изменение конфигурации (CSRF)
- Файл: agent/main.go:1020-1028 (`cors.New(cors.Options{...})`), в сочетании с
  отсутствием проверки `Content-Type` во всех обработчиках, декодирующих JSON
  (например, `/print` — main.go:495, `/printers/add` — main.go:365,
  `/printers/remove` — main.go:437, `/printers/default` POST — main.go:314,
  `/scanners/add` — main.go:816, `/scanners/remove` — main.go:937)
- Описание: Разработчики явно рассчитывают на CORS как на границу безопасности —
  в `agent/README.md:338-345` написано: «Агент настроен на прием запросов только
  от http://localhost:5173 ... Для production обновите настройки CORS». Это
  ложное чувство защищённости по двум независимым причинам, обе подтверждены
  чтением исходников зависимости `github.com/rs/cors@v1.11.1/cors.go:286-289`:
  для НЕ-preflight ("actual") запроса `Handler()` всегда вызывает
  `c.handleActualRequest(w, r)` (которая лишь выставляет/не выставляет заголовки
  ответа) и **безусловно** вызывает `h.ServeHTTP(w, r)` — то есть сам обработчик
  (реальная печать, реальное изменение конфигурации) выполняется независимо от
  того, разрешён Origin или нет; CORS в лучшем случае мешает JS злоумышленника
  прочитать ответ, но не мешает серверу выполнить действие.
  Дополнительно, ни один обработчик не проверяет заголовок `Content-Type`
  запроса — `json.NewDecoder(r.Body).Decode(&req)` парсит тело как JSON независимо
  от заявленного типа. Это значит, что для "simple"-запросов кросс-доменного
  fetch (`Content-Type: text/plain`, `application/x-www-form-urlencoded` или
  `multipart/form-data`) браузер вообще не отправляет CORS preflight (OPTIONS) —
  запрос уходит сразу, сервер получает и обрабатывает его.
- Влияние: Любая веб-страница, открытая в браузере на машине оператора
  (реклама, вредоносный сайт, скомпрометированный виджет и т.п.), может выполнить,
  например:
  `fetch('http://localhost:12345/print', {method:'POST', mode:'cors', headers:{'Content-Type':'text/plain'}, body: JSON.stringify({printer_name:'...', zpl:'...'})})`
  — запрос не потребует preflight, не будет заблокирован CORS, дойдёт до
  `/print` и вызовет реальную печать на физическом принтере оператора (расход
  расходников / вывод произвольного содержимого) либо изменит список принтеров/
  сканеров через `/printers/add`, `/printers/remove`, `/scanners/add` и т.д. Это
  ровно тот сценарий, который упомянут в модели угроз задания: «вредоносная
  веб-страница ... слать команды на печать».
- Серьёзность: Critical
- Уверенность: высокая
- Рекомендация: Не полагаться на CORS как на контроль доступа. Ввести
  server-side аутентификацию (см. AGENT-SEC-01) — она сама по себе закрывает
  CSRF, так как cross-site запрос не будет знать секрет/токен. Дополнительно
  можно проверять `Content-Type: application/json` строго (`http.Error` при
  несовпадении) и/или заголовок `Sec-Fetch-Site`, чтобы отбрасывать
  cross-site запросы на уровне сервера.

### AGENT-SEC-04: Агент можно превратить в прокси для отправки произвольных TCP-данных на любой IP:port (SSRF-подобный примитив)
- Файл: agent/main.go:354-426 (`/printers/add`, создаёт `printer.NewNetworkPrinterFromIP(req.Name, req.IP, req.Port)` из полностью пользовательских `IP`/`Port`, main.go:381-383) и agent/internal/printer/serial.go:86-103 (`NetworkPrinter.SendRaw` делает `net.DialTimeout("tcp", p.Address, ...)` и пишет в сокет произвольные байты без какой-либо валидации адреса или содержимого)
- Описание: `/printers/add` принимает произвольные `ip`/`port` без проверки, что
  это действительно принтер или что адрес принадлежит "разрешённой" подсети
  (нет allow-list, нет проверки на loopback/линк-локальные/служебные адреса).
  После добавления такого "принтера" вызов `/print` с этим `printer_name`
  заставляет процесс агента открыть TCP-соединение на указанный `ip:port` и
  записать туда ровно то содержимое (`zpl`/`template`), которое пришло в теле
  запроса — без какой-либо валидации содержимого (см. также AGENT-SEC-05).
- Влияние: В сочетании с отсутствием аутентификации (AGENT-SEC-01/03)
  злоумышленник (в т.ч. через CSRF с вредоносной веб-страницы, без прямого
  сетевого доступа к машине оператора) может заставить агент, физически
  находящийся в сети мероприятия/офиса, открыть соединение и отправить
  произвольные байты на любой IP:port, доступный из этой сети (внутренние
  сервисы, роутеры администрирования, другие принтеры и т.д.) — то есть агент
  становится сетевым релеем/прокси в обход обычных сетевых ограничений браузера
  жертвы. Это выходит за рамки предполагаемой функциональности (печать на
  сконфигурированный принтер) и может использоваться для разведки/атаки
  внутренней сети через машину оператора.
- Серьёзность: High
- Уверенность: средняя (эксплуатируется только в сочетании с отсутствием
  аутентификации, но именно эта комбинация и есть реальная угроза для данного
  агента)
- Рекомендация: Ограничить `/printers/add` разрешёнными диапазонами адресов
  (например, только частные подсети оператора, явно исключить loopback/
  метаданные облака и т.п.), и в первую очередь закрыть эндпоинт
  аутентификацией, чтобы только доверенный локальный клиент мог регистрировать
  новые сетевые устройства.

### AGENT-SEC-05: Содержимое печати (`zpl`/`template`) передаётся на принтер без какой-либо валидации
- Файл: agent/main.go:488-539 (`/print`), особенно main.go:511-513 (`content = req.ZPL` без проверки) и main.go:528 (`p.SendRaw([]byte(content))`)
- Описание: `PrintRequest.ZPL` (сырой ZPL/TSPL/ESC-POS) передаётся напрямую в
  `SendRaw` без каких-либо проверок: нет ограничения длины, нет фильтрации
  опасных команд принтера (например, ZPL-команды сброса к заводским настройкам,
  перезаписи шрифтов/графики во флеш-памяти принтера типа `~DY`, `^JUS`, `~JA`
  и т.п.), нет проверки, что это вообще похоже на валидный ZPL/TSPL. Аналогично
  `Template`/`Data` (legacy-путь) просто подставляются строкой без экранирования
  спецсимволов ZPL, что позволяет вставить произвольные ZPL-команды через поля
  `data`, если шаблон формируется на основе пользовательского ввода выше по
  стеку (веб/десктоп UI).
- Влияние: Любой источник, способный вызвать `/print` (при снятии барьеров из
  AGENT-SEC-01/03 — практически кто угодно), может отправить на физический
  принтер команды, которые не являются "печатью бейджа": сброс настроек
  принтера, перезапись хранимых шаблонов/шрифтов, порчу оборудования или
  массовый расход этикеток/ленты (DoS на уровне расходников). Отсутствие
  валидации усиливает воздействие остальных находок этого файла.
- Серьёзность: Medium
- Уверенность: средняя
- Рекомендация: Ввести базовую валидацию содержимого перед отправкой на
  принтер: ограничение размера, проверку формата (например, что ZPL начинается
  с `^XA` и заканчивается `^XZ`, без вложенных управляющих команд смены
  памяти/сброса), и экранирование значений `data` при legacy-подстановке в
  `Template`, чтобы пользовательские данные не могли инжектировать ZPL-команды.
