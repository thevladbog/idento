# P4.3 — Equipment hub + device registry — Design

Third and final P4 ("event day") sub-cycle, following the merged P4.1
check-in loop (PR #77) and P4.2 live monitor (PR #81, plus the
event-wide checkin_actions follow-up PR #82). Parent decomposition:
`docs/superpowers/specs/2026-07-17-panel-p4.1-checkin-loop-design.md` §1.
Board reference: **screens 5a–5d** (hub connected/down + printer/scanner
wizards) — extracted to `.superpowers/sdd/p4.3-board-5a-5d-extract.md`;
all four are fully drawn. Parent-spec backend item: **#5 device registry**
(per-machine saved printers/scanners + the single default-printer rule,
server-side, keyed by agent machine id).

## 1. Scope

Ships:

- **Equipment hub** at `/equipment` (org-level, replaces the placeholder
  route) per board 5a/5d: agent card in both states, device columns fed
  by the server registry reconciled against the live agent, per-device
  liveness ("Saved · not seen since …"), the visible single-default rule,
  and the agent-down state with saved devices grayed + start instructions
  + auto-retry.
- **Two guided wizards** per board 5b/5c: printer Find → Test → Save
  (physical verification with a Cyrillic test label) and scanner
  listen-test (USB-wedge in-browser detection; COM via the agent's
  serial endpoints).
- **Backend #5 device registry**: `equipment_machines` +
  `equipment_devices` tables (migration 000023), tenant-scoped CRUD, the
  default-printer rule enforced by a partial unique index, and
  `test_passed_at` capture.
- **Agent `GET /info`**: machine_id (persisted UUID) + hostname +
  version + uptime — the identity the registry keys on and the metadata
  the 5a agent card shows.
- **Readiness wiring**: the pipeline's `equipment` step leaves its
  permanent `not_done` stub — done when the tenant has ≥1 machine whose
  default printer has a passed test.
- **P3.2 print-surface repoint**: the default-printer precedence becomes
  server registry > agent config > first-in-list, inside
  `useAgentPrinters` — consumers (TestPrintDialog, drawer reprint, bulk
  print, station) keep their existing contract.

Explicitly out of scope (user-approved at brainstorm): **camera** as a
device class in any form (row, wizard tab, permission probe) — camera
scanning has been backlog since P4.1 and an unusable "Fallback QR
reader" row fails the honesty rule (registry schema still reserves the
class); the board's **"Agent settings"** button (nothing is drawn behind
it anywhere); Bluetooth scanners (old web listed the type but never
implemented setup); any `web/` changes (frozen); agent auto-update or
remote agent management; printer ribbon/consumable telemetry (the
board's "ribbon OK" is honest only if the agent reports it — it does
not; the meta line shows what the agent actually knows); monitor/SSE
publishes for equipment mutations (the monitor renders no equipment
data; Home polls readiness).

## 2. Decisions (all user-approved during brainstorm, 2026-07-19)

1. **Camera deferred entirely** — no camera row, no Camera tab, until
   camera scan leaves the backlog. Honest deviation from board 5a/5c,
   recorded in the extract.
2. **Server-authoritative default + mirror to agent**: the registry
   holds THE default rule per machine; saving it also POSTs
   `/printers/default` to the agent so the frozen legacy web stays
   coherent; P3.2 surfaces repoint via `useAgentPrinters` precedence
   (server > agent > first).
3. **Tenant-wide readiness rule**: `equipment` = done when the tenant
   has ≥1 registered machine with a default printer whose
   `test_passed_at` is set. Per-event scoping rejected (equipment is
   per-machine, events share the fleet).
4. **Wedge + COM both in scope** for the scanner wizard (the board's tab
   literally reads "USB / COM scanner"; the agent endpoints exist; old
   web could do COM — dropping it would be a regression).
5. Approach A over alternatives: **normalized tables** (not a JSONB doc
   per machine — no DB-enforced single default, lost concurrency; not
   agent-local-only — violates parent backend #5 and dies with a
   reinstall).

## 3. Agent (Go)

### 3.1 `GET /info`

- New unauthenticated endpoint (same class as `/health` and `/docs`),
  JSON:
  `{"machine_id": "<uuid>", "hostname": "REG-DESK-01", "version":
  "1.9.0", "uptime_seconds": 11520}`.
- `machine_id`: UUID generated once and persisted as a new
  `AgentConfig.MachineID` field in `agent_config.json` (same
  load-modify-save path, `configMu` held; missing field on load ⇒
  generate + save — existing installs upgrade in place). It identifies
  the MACHINE for registry keying and carries no authority — all
  registry writes are authorized by the panel's normal tenant session,
  so exposing it unauthenticated leaks nothing actionable.
- `version`: new package-level `agentVersion` constant, overridable at
  build time via `-ldflags -X` (release pipeline wires it later; dev
  builds show the constant). `hostname`: `os.Hostname()` (error ⇒ empty
  string, panel falls back to showing nothing). `uptime_seconds`:
  seconds since process start (a `time.Now()` captured in `main`).
- `/health` is untouched — its plain-text contract is documented and
  consumed; `/info` is additive. openapi.yaml + README endpoint table
  updated (prose change ⇒ no panel client regen needed — the agent is
  not in `backend/openapi.yaml`).

### 3.2 No other agent changes

Printer add/remove/default, scanner ports/add/remove, print, and
scan/consume already exist and suffice for every wizard step.

## 4. Backend

### 4.1 Migration 000023 — two tables

```sql
equipment_machines (
  tenant_id     uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  machine_id    uuid NOT NULL,          -- from agent /info
  hostname      text NOT NULL DEFAULT '',
  agent_version text NOT NULL DEFAULT '',
  last_seen_at  timestamptz NOT NULL DEFAULT now(),
  created_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (tenant_id, machine_id)
)

equipment_devices (
  id             uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id      uuid NOT NULL,
  machine_id     uuid NOT NULL,
  class          text NOT NULL CHECK (class IN ('printer','scanner','camera')),
  kind           text NOT NULL CHECK (kind IN ('system','network','usb_wedge','com')),
  display_name   text NOT NULL,
  config         jsonb NOT NULL DEFAULT '{}',
  is_default     boolean NOT NULL DEFAULT false,
  test_passed_at timestamptz,
  last_seen_at   timestamptz,
  created_at     timestamptz NOT NULL DEFAULT now(),
  updated_at     timestamptz NOT NULL DEFAULT now(),
  FOREIGN KEY (tenant_id, machine_id)
    REFERENCES equipment_machines (tenant_id, machine_id) ON DELETE CASCADE
)

-- THE single-default rule, in the database not in code:
CREATE UNIQUE INDEX equipment_devices_one_default
  ON equipment_devices (tenant_id, machine_id)
  WHERE is_default AND class = 'printer';
-- CHECK (NOT is_default OR class = 'printer') — default is a printer concept.
```

- `config` carries the per-kind facts the hub's mono meta line shows,
  PLUS the stable link to the agent's own identity for the device —
  `display_name` is user-renameable, so it can never be that link:
  network printer `{"agent_name","ip","port","dpi"?}` (`agent_name` =
  the name registered via the agent's `/printers/add`), system printer
  `{"agent_name"}` (the OS printer name), wedge scanner
  `{"terminator"}`, com scanner `{"port_name"}`. `agent_name` is what
  liveness matching (§5.2) resolves against and what `/print` is called
  with when the registry default is used (§5.1). `class='camera'` is
  reserved (accepted by the CHECK, never written this phase).
- `last_seen_at` NULL on `equipment_devices` = never reconciled (a
  device saved but not yet observed live); the hub then shows the saved
  state without a false "seen" claim.
- Tenant scoping follows the house pattern (every query filters
  tenant_id from the authenticated context; `requireTenant`-style
  ownership before any read/write).

### 4.2 Endpoints (all tenant-scoped, normal panel auth)

- `PUT /api/equipment/machines/{machineId}` — upsert on hub connect:
  body `{hostname, agent_version, seen_device_ids: [uuid…]}`. Updates
  machine metadata + `last_seen_at = now()`, and sets
  `last_seen_at = now()` on the listed devices (the panel computed which
  registry devices the live agent currently shows). One request per hub
  load / reconcile — no per-device chatter.
- `GET /api/equipment/machines/{machineId}` — machine + its devices
  (404 when never registered; the panel treats 404 as "empty registry",
  not an error banner).
- `POST /api/equipment/machines/{machineId}/devices` — create (wizard
  Save). Body: class/kind/display_name/config + `make_default: bool` +
  `test_passed: bool` (a passed in-wizard test stamps `test_passed_at`
  server-side at create time — no second round-trip).
- `PATCH /api/equipment/devices/{deviceId}` — rename / config edit
  (overflow menu).
- `DELETE /api/equipment/devices/{deviceId}` — remove (overflow menu,
  confirm dialog; removing the default clears the rule — no silent
  promotion of another printer).
- `PUT /api/equipment/machines/{machineId}/default-printer` — body
  `{device_id}` or `{device_id: null}` to clear. Transactionally clears
  the old default + sets the new (the partial unique index backstops).
- `POST /api/equipment/devices/{deviceId}/test-passed` — stamps
  `test_passed_at = now()` (hub-card "Test print" success on an
  already-saved device; the wizard path uses the create-time flag).
- openapi-first: all shapes in `backend/openapi.yaml`, contract tests +
  coverage ledger, pgxmock with real SQL, house error shapes.
  **Any openapi.yaml edit ⇒ `npm run generate:api -w panel` committed**
  (CI drift gate).

### 4.3 Readiness wiring

`readiness.go`'s `equipment` step: new store query — EXISTS a
`equipment_devices` row for the tenant with `is_default AND class =
'printer' AND test_passed_at IS NOT NULL`. Count omitted (the step is
boolean; machines-with-tested-default is not a number the pipeline
shows). `ready` stays `attendees && badge && staff` per the parent spec
— equipment joins the step list's honest display but does NOT newly
block launch (changing the launch gate is a product decision this phase
does not take; the ceremony already shows its own live agent check).
The openapi description of the readiness schema loses its "always
not_done" caveat (⇒ client regen).

## 5. Panel

### 5.1 Data layer

- `agentClient` grows `getInfo(): Promise<AgentInfo | null>` (null on
  404 — an older agent without `/info`; every hub surface must degrade:
  registry features need machine identity, so an old agent shows the
  live-only view + an "update the agent" hint line on the agent card).
- New `shared/agent/useAgentInfo.ts` — health + info in one query
  (house pattern of useAgentPrinters: enabled-gated, no retry,
  refetchOnWindowFocus; plus `refetchInterval: 8s` while disconnected —
  the board's "auto-retry in 8 s", with the countdown rendered from the
  interval).
- New `features/equipment/` data hooks over the generated `$api` client:
  `useEquipmentMachine(machineId)` (GET, 404 ⇒ empty),
  mutations for upsert/reconcile, device create/patch/delete,
  default-printer, test-passed — standard TanStack Query + invalidation
  of the machine query.
- **machine_id caching**: last successful `AgentInfo` is cached in
  localStorage (per agent base URL). Agent down ⇒ the hub uses the
  cached machine_id to still fetch the registry (board 5d's grayed
  saved devices). No cache + agent down ⇒ 5d without device cards (the
  honest cold-start).
- `useAgentPrinters` precedence change (decision 2): when the machine's
  registry is readable and has a default whose `config.agent_name` is
  present in the live agent printer list, that agent name wins; else
  the agent's configured default; else first-in-list. `configuredDefault` keeps its "genuinely
  configured or null" meaning — the drawer-reprint ask-vs-don't-ask
  behavior is preserved with the server rule simply becoming the top
  source. All P3.2 consumers compile untouched.

### 5.2 Hub route — `/equipment`

- Replaces `PlaceholderPage` in `router.tsx` (protected, in-shell,
  org-level — nav link already exists). `WorkspaceRail`'s equipment
  entry stays untouched (it references the readiness step, not this
  route).
- **Agent card**: `@idento/ui` `AgentStatus` primitive; connected ⇒
  green pill + mono meta `<base-url> · v<version> · <hostname> ·
  uptime <h> h <m> m`; disconnected ⇒ board 5d red card + "Start the
  agent" steps + Retry button + "auto-retry in 8 s" countdown; the
  download link's href is env-driven
  (`window.__ENV__.AGENT_DOWNLOAD_URL`, default = the repo's GitHub
  releases page) — same `__ENV__` mechanism as `AGENT_URL`.
- **Device columns** (Printers / Scanners): rows from the registry,
  reconciled client-side against the live agent:
  - registry device visible live ⇒ green dot + meta from config;
  - saved but not visible ⇒ amber dot + **text** "Saved · not seen
    since <date> — check power & network" (date = `last_seen_at`, or
    "not seen yet" when NULL; never color-alone) + Retry (refetch).
    Green/amber apply only to liveness-bearing kinds (system/network
    printers by `agent_name`, com scanners by `port_name`) — wedge
    scanners have no observable online state and render no dot at all
    (see the reconcile note below);
  - agent down ⇒ whole cards at reduced opacity, gray dots, actions
    hidden, header caption "saved · unreachable" (board 5d).
  - live agent printers NOT in the registry (e.g. a system printer the
    OS knows) appear as unsaved rows with a "Save…" affordance opening
    the wizard pre-filled at the Test step — the discovery path into
    the registry.
  - DEFAULT chip on the registry default; row overflow menu: rename,
    set/clear default, delete (typed-confirm not needed — deletion of a
    device is re-creatable in one wizard run; a plain confirm dialog
    per house dialog pattern).
  - Printers card footer: the board's one-rule sentence; Scanners card
    footer: the persistence sentence.
- **Reconcile-on-load**: when agent info + live lists + registry have
  all loaded, fire the machine upsert with `seen_device_ids` (printers
  matched by `config.agent_name` against the live `/printers` list —
  the ONLY available key: the agent's printer list carries name+type,
  never ip/port, so no ip cross-check is possible — and com
  scanners by `config.port_name` against `/scanners`; wedge scanners
  are browser-side devices the agent cannot see — they are matched
  never, and their rows show config facts without a liveness dot (a
  wedge scanner has no observable "online" state; showing green would
  be a fabrication)). One upsert per hub visit, not a poll.
- "+ Add device" (header) opens a class chooser (Printer / Scanner) →
  respective wizard; the per-card "+ Set up printer/scanner" buttons
  skip the chooser.

### 5.3 Printer wizard (board 5b)

Modal, 3 steps, house dialog primitives:

1. **Find**: live agent printer list (system + already-added network),
   pick one — or "Enter IP manually" ⇒ name/ip/port form that POSTs the
   agent's `/printers/add` (the agent persists it locally and it joins
   the live list). Old-agent/down states disable the wizard entry
   points (the hub's buttons are gated on connected).
2. **Test**: POST `/print` with a generated test ZPL — alignment marks +
   `«Кириллица 123»` + printer name + timestamp (ZPL built panel-side;
   `^CI28`/UTF-8 per the existing badge-ZPL pipeline's conventions).
   Buttons per board: "Yes, looks right" (⇒ `test_passed`), "Print
   again", "Something's off…" (⇒ inline troubleshooting hints: check
   darkness/media/driver, link to re-run Find; does NOT block Save —
   saving an untested printer is allowed, it just doesn't stamp
   `test_passed_at` and doesn't satisfy readiness).
3. **Save**: display name (prefilled from discovery) + the default
   checkbox (board copy). Save ⇒ registry create (+`make_default`,
   +`test_passed`) and, when default, mirror POST `/printers/default`
   to the agent. Mirror failure = non-fatal warning toast (server rule
   saved; agent mirror retries on next hub load — document as the
   deliberate consistency direction: server wins).

### 5.4 Scanner wizard (board 5c)

One modal, kind toggle **USB wedge | COM** (camera tab dropped):

- **USB wedge**: in-browser listen test — keydown capture in the modal
  (same capture technique as `useScanInput`'s wedge mode, scoped to the
  dialog), detects the code, the terminator (Enter / Tab / none) and
  keystroke latency; shows the board's "Scan received" row; fields:
  device name + terminator (prefilled from detection, editable). Save ⇒
  registry create (`kind: usb_wedge`, config `{terminator}`).
- **COM**: port select from `/scanners/ports` ⇒ `/scanners/add` (agent
  opens + persists the port) ⇒ listen phase polls `/scan/consume` (the
  atomic primitive; anything consumed during the listen window is the
  test read — the station is not running while the wizard is open on
  this machine) ⇒ same result row ⇒ Save (`kind: com`, config
  `{port_name}`). Cancel after `/scanners/add` leaves the port open
  agent-side but unsaved server-side — acceptable; the hub's unsaved-
  device affordance (5.2) offers saving it later; `/scanners/remove`
  from the overflow menu closes it.
- Scanner rows have no Test button on the hub this phase beyond the
  wizard ("Test scan" re-opens the wizard's listen step for that
  device) — one listening surface, no competing keyboard captures.

### 5.5 Cross-surface effects

- Readiness pipeline (Home `ReadinessCell`, workspace rail) starts
  showing `equipment: done` for real — no panel code change needed
  beyond the regenerated client type (the step key already renders).
- Launch ceremony / print dialogs pick up the server default via the
  `useAgentPrinters` precedence change transparently.
- i18n: EN + RU flat keys `equipment*`, keyParity test, real Russian
  copy (the board's RU-adjacent strings translated properly).

## 6. Correctness

- **Single default**: enforced by the partial unique index; the PUT
  endpoint is transactional (clear-then-set); concurrent set from two
  browsers cannot yield two defaults — one loses with a constraint
  error surfaced as the house 409 shape.
- **Honest liveness**: green requires BOTH registry presence and live
  visibility now; amber text always accompanies the amber dot (WCAG
  1.4.1, the P4.2 never-color-alone rule); wedge scanners get no
  liveness claim at all.
- **Old agent degradation**: `/info` 404 ⇒ live-only hub + update hint;
  nothing crashes; registry features gate on machine identity.
- **Server-wins consistency**: registry is the rule; the agent mirror
  is best-effort and re-converges on every hub load (documented in
  code at the mirror call site).
- **Tenant isolation**: two tenants sharing one physical machine keep
  disjoint registries (the PK is (tenant_id, machine_id) by design —
  a rented laptop moving between orgs leaks nothing).
- web/ untouched; monitor untouched; no SSE publishes.

## 7. Testing

- **Agent (Go)**: `/info` handler tests (shape, machine_id persistence
  across a reload of the config, uptime monotonicity, hostname
  fallback); config upgrade test (pre-existing config without
  machine_id gains one and keeps every other field byte-identical).
- **Backend**: contract tests per endpoint (openapi validateResponse +
  coverage ledger); pgxmock with real SQL for the upsert +
  seen-devices update, default clear-then-set transaction, readiness
  EXISTS query; migration up/down; constraint tests for the partial
  unique index + class/kind CHECKs (env-gated real-PG per the
  000020/000022 precedent).
- **Panel**: MSW suites per surface — hub (connected: reconcile fires
  once with correct seen ids; amber not-seen rows; unsaved live
  printer row; default chip), agent-down (cached machine_id path:
  grayed cards; cold-start path: no device cards; 8s auto-retry),
  printer wizard (find→test→save happy path; manual-IP path; save
  without test leaves readiness untouched; default checkbox mirrors to
  agent; mirror-failure toast), scanner wizard (wedge detection state
  machine incl. terminator detect; COM port flow via mocked agent),
  `useAgentPrinters` precedence (server > agent > first, and the
  registry-default-not-in-live-list fallback), readiness cell
  regression (equipment done renders).
- keyParity EN/RU; gates = the standard full set (panel
  typecheck/test/lint/build via `npm run typecheck` (NOT bare tsc),
  ui tests, schema drift incl. regenerated client, backend
  tests+race+lint, agent `go test -race` + golangci-lint).

## 8. Open items deliberately deferred

- Camera class end-to-end (row, wizard tab, station camera scan) — one
  backlog item, owns its own cycle.
- Bluetooth scanners.
- Agent settings surface; agent auto-update / remote management.
- Printer consumable telemetry (ribbon state) — needs agent support
  first.
- Machines list view (seeing OTHER machines' registries from this
  browser) — the hub is deliberately "this computer" scoped like the
  board; a fleet view is a future phase if ever needed.
- Launch-gate tightening (making `ready` require equipment) — product
  decision, not taken here.
