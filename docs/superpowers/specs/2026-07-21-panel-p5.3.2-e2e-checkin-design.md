# Panel P5.3.2 — Playwright e2e (check-in scan→verdict) — Design

Second of five P5.3 sub-cycles (the final phase of the panel rewrite
initiative). Parent decomposition: the P5.3 scoping brainstorm's 5-way split
(P5.3.1 docs, done; P5.3.2 e2e, this one; P5.3.3 WCAG-AA a11y; P5.3.4 EN/RU
parity for `web/`; P5.3.5 5k-attendee perf). Ships the **first e2e test
infrastructure for `panel/`** — none exists today (`landing/` has a local-only
Playwright harness, unrelated and not wired into CI).

## 1. Scope

Covers the check-in station's **scan→verdict flow**, end to end against a
real backend + real Postgres (no mocks): given a ready event with a station
registered, submitting an attendee code produces the correct verdict card for
three outcomes reachable from one setup — `checked_in` (first scan),
`already_checked_in` (repeat scan of the same code), `not_found` (unknown
code). This is deliberately the flow the parent P5 spec named explicitly
("Playwright e2e (scan→verdict happy path)").

Out of scope: the login UI itself (driven via API, not clicked through — see
§3), the launch ceremony UI (bypassed via direct station registration — see
§4), print/agent behavior (explicitly disabled for this suite — see §5),
`blocked` outcomes (zone-rule-gated check-ins — a real but separate scenario,
not part of the "happy path" set), cross-browser coverage (chromium only —
see §6), WCAG-AA (P5.3.3), EN/RU parity (P5.3.4), performance (P5.3.5).

## 2. Decisions (user-approved during brainstorm, 2026-07-21)

1. **CI environment = dev-mode, not prod-like.** The e2e job brings up
   `docker compose up -d db redis` (the existing dev-infra compose), runs the
   backend via `go run main.go`, runs panel via `npm run dev` (port `5174`,
   set in P5.3.1), and points Playwright at `http://localhost:5174`. This
   mirrors what a contributor runs locally (`bash scripts/start-all.sh`)
   rather than building the P5.1 combined production image — faster CI, and
   the P5.1 cutover's own routing/path-split was already verified separately
   via a live curl matrix in that phase. Not covered by this suite: whether
   the *production* nginx image serves the app correctly (P5.1's concern) —
   only whether the panel *application* behaves correctly.
2. **CI gating = soft at first, not a required check.** The new e2e job runs
   on every PR that touches `panel/`/`backend/` and is visible in the PR
   checks list, but is **not** added to `ci-success`'s `needs:` — a failure is
   visible, not blocking. Promote to a required check once the suite has
   demonstrated stability over real PRs (a later, explicit follow-up, not
   part of this cycle).
3. **Backend runs in its DEFAULT deployment mode (on-prem), not SaaS.**
   Investigated during brainstorm: `DEPLOYMENT_MODE` defaults to `onprem`
   both in local dev (unset in `scripts/start-all.{sh,ps1}` and commented out
   in `.env.example`) and would need an explicit override for SaaS-only
   routes (`POST /auth/register`, super-admin console routes) to be mounted
   at all (`backend/internal/handler/handler.go:65-66`,
   `RegisterRoutes(e, mode)`). Rather than introduce a SaaS-only code path
   into the e2e stack (deviating from what a contributor's default `go run`
   actually exposes), the test authenticates as the **seeded**
   `admin@test.com` account and creates a **fresh event** via the real API
   for per-run isolation — tenant-level isolation is unnecessary because CI
   already gets a genuinely empty Postgres each run (no persisted volume).

## 3. Auth strategy

No login-UI interaction. `panel/`'s route guard
(`ProtectedLayout.tsx`/`session.ts`) only checks `localStorage.getItem("token")
!== null` — no server-side verification on route entry
(`StationPage.test.tsx` already relies on exactly this for its own
unit-level setup). The e2e setup:

1. `POST /auth/login` with `admin@test.com` / **`password`** (verified via
   bcrypt against the seeded hash during brainstorm — `seed.sql`'s own
   comment says `password`; `scripts/seed.sh`'s printed hint says
   `password123`, which is wrong — see §7's incidental fix) → real JWT.
2. `page.addInitScript()` (or an equivalent pre-navigation script) sets
   `localStorage.setItem("token", <jwt>)` (plus whatever `saveSession()`
   also persists — `user`, `tenants`, `current_tenant` — sourced from the
   same login response) before the first navigation.
3. Navigate straight to the target route; the guard passes on token
   presence, and the token is genuinely valid for subsequent API calls the
   panel itself makes (readiness, attendee search, checkin POST, etc.).

## 4. Test data seeding (per test file, self-contained)

All via real HTTP calls (Playwright's `request` fixture or plain `fetch`
against `http://localhost:8008`), not the panel's generated client (which is
browser-coupled — confirmed during brainstorm, `panel/src/shared/api/http.ts`
depends on `window.__ENV__`/`localStorage` and isn't usable from Node
context) and not reused MSW handlers (Vitest-only). Sequence, using the
`admin@test.com` session from §3:

1. `POST /api/events` — `{name: "<unique per run>"}` → `event.id`.
2. `POST /api/events/{event_id}/attendees` (single-create, not the bulk/CSV
   endpoint — simpler for 2-3 fixture attendees) → at least 2 attendees, each
   with a known `code` (for the `checked_in`/`already_checked_in` scans) plus
   one code that is deliberately never created (for the `not_found` scan).
3. `PUT /api/events/{event_id}/badge-template` — `{template: {width_mm: 90,
   height_mm: 55, dpi: 203, elements: [{type: "text", ...}]}, version: 0}` —
   MUST have a non-empty `elements` array (readiness's `badgeDone` check,
   `backend/internal/handler/readiness.go:74-91`, requires it; an
   empty-canvas template does not count).
4. `POST /api/events/{event_id}/staff` — `{user_id: <the logged-in admin's
   own id, from the login response>}` (readiness's `staffCount > 0`
   requirement — the endpoint assigns an *existing* tenant user, not a
   free-text name, so reusing the admin avoids creating a second account).
5. `GET /api/events/{event_id}/readiness` → assert `ready: true` before
   proceeding (a fast-failing sanity check — if this ever goes false, every
   subsequent step's failure would otherwise be confusing).
6. `PUT /api/events/{event_id}/checkin-settings` — `{print_on_checkin:
   false, ...other fields at their defaults}` — see §5 for why.
7. `POST /api/events/{event_id}/checkin-stations` — registers a station,
   returns `station.id`, used as the `?station=` search param (bypasses the
   `LaunchCeremony` UI entirely — `searchParams.ts`'s `checkinStationBeforeLoad`
   only requires a UUID-shaped `station` param, doesn't verify it server-side
   at the route-guard level).

## 5. Print handling

`print_on_checkin` defaults to `true`
(`panel/src/features/checkin/settingsTypes.ts:25`), so a `checked_in`
verdict would otherwise attempt to reach the local print agent at
`http://localhost:12345` — a real separate process this e2e suite does not
run. Per `useCheckinFlow.ts:169-241`, a failed print attempt is fully
non-blocking (the verdict card renders regardless — confirmed by reading the
actual code path, not assumed), but it still fires a network request that
would hang/fail noisily in a suite with no agent present. Setting
`print_on_checkin: false` (§4 step 6) avoids the attempt entirely — the
cleanest option for a suite that isn't testing print at all (print/ZPL
correctness is already covered by existing Vitest suites from P3.2, and
real-printer verification was done manually on the Zebra ZD410 during the
P4.3 exit gate).

## 6. Verdict coverage (one setup, three assertions)

Using the fixture data from §4:

- **`checked_in`**: submit the first fixture attendee's code via the wedge
  scan input (`page.getByLabel(t("checkinScanWedgeInputLabel"))` — the
  `sr-only` input that captures keystrokes, `useScanInput.ts:160-171`) →
  assert `[data-testid="checkin-verdict-card"][data-verdict="allowed"]`
  becomes visible.
- **`already_checked_in`**: submit the SAME code again directly — no dismiss
  step needed. `submitCode` (`useCheckinFlow.ts:244-246`) unconditionally
  calls `clearDismissTimer()` and sets `status: "resolving"` regardless of
  the current state, so a second scan while the verdict card is still showing
  transitions straight into the next resolution (confirmed by reading the
  actual code, not assumed — the wedge input stays interactive throughout;
  there is no need to wait out `verdict_auto_dismiss_sec`, default 4s, or
  find a manual dismiss control). Assert `data-verdict="already_checked_in"`
  and that `data-testid="checkin-first-scan-meta"` renders (the original
  scan's timestamp/operator info).
- **`not_found`**: submit a code that was never created in §4 → assert
  `data-verdict="not_registered"` (this one resolves client-side per
  `useCheckinFlow.ts`'s `submitCode`, no server round trip — still worth
  covering since it's a distinct, real branch of the same component).

`blocked` (zone-rule denial) is a real fourth outcome but requires zone-rule
fixture setup this cycle doesn't otherwise need — deferred, not part of the
"happy path" set this spec targets.

## 7. Incidental fix: `scripts/seed.sh`'s wrong printed password

Discovered during brainstorm (§3): `scripts/seed.sh` echoes `Password:
password123` after seeding, but the actual bcrypt hash in
`backend/migrations/seed.sql` only matches the plaintext `password`
(empirically verified: `bcrypt.CompareHashAndPassword` against the seeded
hash succeeds for `"password"`, fails for `"password123"`). This is a
real, pre-existing bug (misleads any human reading the terminal output, not
just this e2e suite) directly touching this cycle's own test credentials —
fixed as a one-line correction to the echoed string, not a functional change.

## 8. Tooling & file layout

- `@playwright/test` added to `panel/package.json` devDependencies (matching
  `landing/`'s pinned version, `^1.58.2`, for consistency unless a newer
  patch is current at implementation time).
- `panel/playwright.config.ts` — chromium only (`devices["Desktop Chrome"]`,
  matching `landing/playwright.config.ts`'s established precedent — no
  cross-browser matrix for this first suite), `reporter: "list"`,
  `baseURL: "http://localhost:5174"`, `webServer: { command: "npm run dev",
  url: "http://localhost:5174", reuseExistingServer: true }` (works
  identically whether a dev server is already running locally or needs to be
  started fresh in CI).
- Tests live in `panel/e2e/` (co-located with the app under test, mirroring
  `landing/tests/`). First spec: `panel/e2e/checkin.spec.ts`. A shared setup
  helper (the §4 seeding sequence) lives alongside it, e.g.
  `panel/e2e/fixtures/seedCheckinEvent.ts` — reusable by later P5.3.x specs
  if any need a ready-for-checkin event.
- `panel/package.json` gains a script, e.g. `"e2e": "playwright test"`.

## 9. CI wiring

New job in `.github/workflows/ci.yml`, e.g. `e2e-panel`:

- Gated on `needs.changes.outputs.panel == 'true' || needs.changes.outputs.backend
  == 'true'` (an e2e failure could stem from either side).
- Steps: checkout → setup Go + Node → `docker compose up -d db redis` (wait
  for healthy) → `cd backend && go run main.go &` (background, poll
  `/api/instance` until it responds — backend must be reachable before
  Playwright's `webServer` starts panel, since panel's own dev server makes
  API calls on load) → `cd panel && npx playwright install --with-deps
  chromium` → `cd panel && npm run e2e`, letting `playwright.config.ts`'s own
  `webServer` block start and wait on the panel dev server (no separate
  manual `npm run dev &`/port-poll step in the CI script — one less moving
  part, and it's the same mechanism a local `npx playwright test` run uses).
- Per decision 2 (§2): **not** added to `ci-success`'s `needs:` array. Runs
  and reports, doesn't block merge yet.
- On failure, Playwright's HTML report / trace artifacts should be uploaded
  (`actions/upload-artifact`) so a failing run is debuggable from the Actions
  UI without local reproduction being the only option.

## 10. Testing (of this cycle's own deliverable)

There's no meta-test for an e2e suite beyond running it. Verification is:
running `npm run e2e` locally against a real `go run` backend + `npm run dev`
panel (the same manual stack a contributor would start) and confirming all
three verdict assertions pass; then confirming the new CI job runs green
(or, if it fails, that the failure is a real bug the suite caught — not a
suite defect) once wired in.
