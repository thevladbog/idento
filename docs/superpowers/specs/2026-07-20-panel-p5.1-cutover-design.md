# Panel P5.1 — Cutover mechanics (combined nginx image) — Design

First of three P5 ("Parity & cutover") sub-cycles, closing the panel rewrite
initiative. Parent decomposition: `docs/superpowers/specs/2026-07-13-panel-rewrite-design.md`
§4 ("One image, two SPAs") and §5 (P5 = Parity & cutover). This sub-cycle
ships the CUTOVER MECHANICS only; the legacy-code deletion (badge-template
dual-sync) is P5.2 and the quality gates (Playwright e2e, a11y, EN/RU parity,
5k perf) are P5.3.

## 1. Scope

Ships:

- A **single combined nginx image** that serves the new `panel/` SPA at `/`
  and the old `web/` app — reduced to the super-admin **console only** —
  under `/super-admin/`.
- `web/` **stripped to the console**: every customer-panel route the new
  `panel/` now owns (auth, dashboard, events, users, equipment, organization,
  and the browser check-in kiosk) is deleted; only the super-admin surface
  remains, rebased under `/super-admin/`.
- **prod-compose cutover**: the existing single `web` frontend service builds
  and runs the combined image — the docker-compose topology is unchanged
  (still one frontend service, SaaS and on-prem alike), per the parent spec's
  constraint.

Explicitly out of scope: badge-template dual-sync + legacy-reader deletion
(P5.2); Playwright e2e, WCAG-AA a11y audit, EN/RU parity check, 5k-attendee
perf validation (P5.3); any console REWRITE (the console stays the old `web/`
React app — only rebased and route-stripped); the Tauri `desktop/` kiosk (a
separate app, untouched); backend/agent changes (none).

## 2. Decisions (user-approved during brainstorm, 2026-07-20)

1. **`web/` after cutover = console only.** The browser check-in kiosk
   (`/checkin-fullscreen`) is retired with the rest of the customer surface —
   the panel's own station (P4.1) and the Tauri desktop kiosk cover check-in.
   No kiosk route survives in `web/`, so a single `/super-admin/` base is
   sufficient (no split-path nginx routing needed).
2. **One combined image, one frontend service** — not two services behind a
   router. The parent spec's "one image, two SPAs" model; docker-compose
   topology unchanged.
3. **Shared same-origin auth** — panel and console already use the identical
   `localStorage` keys (`token`, `operator_token`); on one origin they share
   them by construction, so cross-SPA impersonation (console → panel) works
   with no new plumbing. This is made an explicit, documented contract.

## 3. Combined image

A single multi-stage `Dockerfile` at the repo root (build context = repo
root, mirroring `panel/Dockerfile`'s existing root-context rationale):

```
# Stage 1 — build panel (npm workspace: needs packages/)
FROM node:24-alpine AS build-panel
WORKDIR /src
COPY package.json package-lock.json .npmrc ./
COPY packages ./packages
COPY panel ./panel
RUN npm ci && npm run build -w panel        # → /src/panel/dist

# Stage 2 — build web (console) with base=/super-admin/ (web/ is NOT a
# workspace member — it has its own package.json + lockfile)
FROM node:24-alpine AS build-web
WORKDIR /web
COPY web/package.json web/package-lock.json web/.npmrc ./
RUN npm ci
COPY web ./
RUN npm run build                            # base set in vite.config (§4) → /web/dist

# Stage 3 — one nginx serving both
FROM nginxinc/nginx-unprivileged:1.27-alpine
USER root
COPY --chown=nginx:nginx --from=build-panel /src/panel/dist /usr/share/nginx/html
COPY --chown=nginx:nginx --from=build-web   /web/dist        /usr/share/nginx/html/super-admin
RUN chown nginx:nginx /usr/share/nginx/html /usr/share/nginx/html/super-admin
USER nginx
COPY deploy/nginx.conf /etc/nginx/conf.d/default.conf
# envsubst mirrors template subpaths into the output dir, so the console's
# env.js lands at super-admin/env.js (see §5).
COPY panel/env.js.template       /etc/nginx/templates/env.js.template
COPY web/env.js.template         /etc/nginx/templates/super-admin/env.js.template
ENV NGINX_ENVSUBST_OUTPUT_DIR=/usr/share/nginx/html
EXPOSE 8080
```

- Asset paths never collide: panel emits `/assets/*`, the console (base
  `/super-admin/`) emits `/super-admin/assets/*`.
- The pre-existing per-app `panel/Dockerfile` and `web/Dockerfile` are
  superseded by this combined one for production. **CI change**: the
  release workflow's separate panel-image and web-image jobs collapse into
  one combined-image job (`ghcr.io/…-web` keeps its name so the compose
  reference and existing deploys are unchanged; drop the standalone
  panel-image publish). The per-app Dockerfiles may stay for local
  single-app dev builds (documented) or be removed — the plan decides,
  defaulting to keeping `panel/Dockerfile` for `preview`/dev and removing the
  now-dead standalone web image job.

## 4. nginx + web base

`deploy/nginx.conf` (new; combines the two identical current confs plus the
console location):

```
server {
    listen 8080;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    # Console SPA (old web/, rebased). Its own fallback + immutable assets.
    location /super-admin/ {
        try_files $uri $uri/ /super-admin/index.html;
    }
    location /super-admin/assets/ {
        add_header Cache-Control "public, max-age=31536000, immutable";
        try_files $uri =404;
    }
    location = /super-admin/env.js {
        add_header Cache-Control "no-store";
        try_files $uri =404;
    }

    # Panel SPA (new panel/) at root. MUST be declared AFTER the console
    # locations so /super-admin/* matches the console, not the panel fallback.
    location / {
        try_files $uri $uri/ /index.html;
    }
    location /assets/ {
        add_header Cache-Control "public, max-age=31536000, immutable";
        try_files $uri =404;
    }
    location = /env.js {
        add_header Cache-Control "no-store";
        try_files $uri =404;
    }
}
```

(nginx `location` longest-prefix matching means `/super-admin/…` resolves to
the console blocks regardless of declaration order; the ordering comment is a
readability aid, and a routing test in §7 pins the behavior.)

`web/vite.config.ts`: add `base: '/super-admin/'`. `web/`'s router
(`web/src/App.tsx`, `BrowserRouter`) gets `basename="/super-admin"` so its
in-app links and history resolve under the base.

## 5. Runtime config (env.js)

Both SPAs need `PUBLIC_API_URL` injected at container start (the proven
`env.js.template` + envsubst pattern):

- Panel: `panel/env.js.template` → `/etc/nginx/templates/env.js.template` →
  envsubst → `/usr/share/nginx/html/env.js` (fetched by the panel at `/env.js`).
  Also carries `PUBLIC_AGENT_URL` (unchanged).
- Console: `web/env.js.template` → `/etc/nginx/templates/super-admin/env.js.template`.
  The base image's `20-envsubst-on-templates.sh` mirrors the template's
  relative sub-path into `NGINX_ENVSUBST_OUTPUT_DIR`, so it writes
  `/usr/share/nginx/html/super-admin/env.js` (fetched by the console at
  `/super-admin/env.js`). **Confirmed required fix**: `web/index.html`
  currently hardcodes `<script src="/env.js"></script>` (absolute), which
  under the new base would wrongly load the PANEL's `/env.js`. Change it to
  `<script src="%BASE_URL%env.js"></script>` — Vite substitutes `%BASE_URL%`
  with the configured `base` (`/super-admin/`) at build, so it resolves to
  `/super-admin/env.js`. (The panel's `index.html` already loads its env.js
  correctly at the root base — unchanged.)

## 6. web/ route strip (console only)

`web/src/App.tsx` currently mounts customer routes AND the super-admin
console. Delete the customer half; keep the console, promoted to the app root
under the `/super-admin` basename:

- **Delete routes + their page modules**: `/login`, `/qr-login`, `/register`,
  `/dashboard`, `/events`, `/events/:eventId` (+ zones/template/staff/checkin/
  settings children), `/users`, `/checkin`, `/checkin-fullscreen`,
  `/equipment`, `/organization`, and the `/` catch-all's customer landing.
  Remove the now-orphaned pages (`Dashboard.tsx`, `Events.tsx`,
  `BadgeTemplateEditorV2.tsx`, `CheckinFullscreen.tsx`, `CheckinSelectEvent.tsx`,
  `EquipmentSettings.tsx`, `OrganizationSettings.tsx`, `Users.tsx`, plus the
  `pages/event/*` customer pages) and their exclusive helpers/components,
  guided by the compiler (unused-import/dead-file errors) — do NOT delete
  anything the console still imports.
- **Keep**: everything under `web/src/pages/super-admin/*`
  (`SuperAdminLayout`, `Dashboard`, `Organizations`, `OrganizationDetail`,
  `SubscriptionPlans`, `AllUsers`, `Analytics`, `AuditLog`) + the console's own
  login (a super-admin entry point) rebased to `/super-admin/login`, and the
  shared infra the console needs (`lib/api.ts`, `lib/impersonation.ts`, auth
  guard, `OrganizationSwitcher`, `Layout`).
- **Router shape** under `basename="/super-admin"`: the console routes drop
  their now-redundant `/super-admin` prefix (they ARE the app root now), e.g.
  the current `<Route path="/super-admin">…<Route path="organizations">` tree
  becomes the top-level route set; `/` (app root) → the console dashboard;
  `/login` → the console login. A super-admin visiting `/super-admin/` lands
  on the dashboard (or `/super-admin/login` if unauthenticated).
- **web/'s own tests** for deleted pages are removed with them; the surviving
  console tests keep passing (the route-tree change may need their harness to
  wrap in a `MemoryRouter basename`/updated paths — adjust minimally).

## 7. Auth / impersonation contract

Made explicit (already true by shared keys, now depended-upon):

- Panel (`panel/src/shared/api/session.ts`) and console (`web/src/lib/*`) both
  read/write `localStorage["token"]` and `localStorage["operator_token"]`.
  On the combined image they are the **same origin**, so the keys are shared.
- **Super-admin login**: at `/super-admin/login` (console's own), which writes
  the shared `token`.
- **Impersonation round-trip**: console's "impersonate tenant" stashes the
  operator token as `operator_token` and overwrites `token` with the tenant
  token (existing `web/src/lib/impersonation.ts`); the panel at `/` reads the
  same keys and renders the impersonation banner (built in P0.2). "Stop
  impersonating" in either SPA restores `operator_token` → `token`. No new
  code — the spec pins this as a supported, tested flow (§8 staging step).

## 8. Testing

- **Combined-image build + staging compose run** (the phase's proof): a
  `docker compose` bring-up of the combined image (a scratch/staging compose
  or the prod compose against a local backend) verifying, with real HTTP:
  panel loads at `/` (its `index.html` + `/assets/*` + `/env.js` with the
  substituted API URL); console loads at `/super-admin/` (its `index.html` +
  `/super-admin/assets/*` + `/super-admin/env.js`); a deep-link reload of a
  console route (`/super-admin/organizations`) serves the console
  `index.html` (SPA fallback), and a deep-link reload of a panel route
  (`/events/x`) serves the panel `index.html` — the two fallbacks don't cross.
  Recorded in the ledger; the exact harness (compose file vs a container run
  + curl matrix) is chosen at plan time.
- **web/ unit suite** green after the route strip (console tests only).
- **panel/ + packages/ui** untouched → their suites stay green (regression
  guard only, no changes expected).
- **Impersonation round-trip** exercised in the staging run (log into console,
  impersonate, confirm the panel origin shows the banner + tenant; stop,
  confirm restore).
- Gates: `npm run build -w panel`, `cd web && npm run build` (with the base),
  `npm run test -w web` (or web's test command), the combined image builds,
  and the staging bring-up passes. Backend/agent unchanged.
