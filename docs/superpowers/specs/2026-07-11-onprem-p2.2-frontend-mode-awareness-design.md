# On-Prem Phase 2 — P2.2 Frontend Mode Awareness — Design

- **Date:** 2026-07-11
- **Status:** approved by user, pending implementation plan
- **Depends on:** P2.1 (merged, PR #40) — no code dependency, but P2.1 established the pattern of "on-prem is the safe default" this design mirrors. `GET /api/instance` (Phase 0, unauthenticated, already returns `{mode, version, license}`) is the only backend surface this item consumes — no backend changes in this batch.
- **Roadmap item:** `docs/DUAL_DISTRIBUTION_REWORK.md` § Phase 2, P2.2.

## Goal

The web app hides self-signup and platform-admin surfaces when running against an `onprem` backend: the "Register" link disappears from the login page, direct navigation to `/register` redirects to `/login`, and the "Super Admin" nav item disappears from the top nav — closing the gap where an on-prem operator could reach a registration form that the backend already 404s on submit, or see a dead nav link to a section the backend already blocks.

## Key decisions (confirmed with user)

1. **Active redirect, not just a hidden link.** Direct URL navigation to `/register` in `onprem` mode redirects to `/login`, mirroring how the existing `ProtectedRoute` already redirects `/super-admin` to `/dashboard` when the viewer isn't a super-admin. A hidden link alone would leave the broken-form UX gap open for anyone who types or bookmarks the URL.
2. **Safe-default gating: allow-list, not block-list.** Registration/nav visibility require an explicit, confirmed `mode === 'saas'` response. While `/api/instance` is loading, or if it fails, the app behaves as `onprem` (hidden/redirected) — mirroring the backend's own `DeploymentMode` default (`onprem` when `DEPLOYMENT_MODE` is unset, "the safe default for a binary running outside our infrastructure").
3. **One shared fetch, not one per consumer.** `/api/instance` is fetched once per browser session (memoized), not separately by `Login.tsx`, `App.tsx`'s route guard, and `Layout.tsx` each making their own request.
4. **No change to the `/super-admin` route's existing guard.** `ProtectedRoute requireSuperAdmin` already blocks it unless `user.is_super_admin` is true, and P2.1's bootstrap flow creates a `role: "admin"` user, never `is_super_admin: true` — so `/super-admin` is already unreachable on a bootstrap-only on-prem install. Only the *nav link* (which doesn't check the route guard, just renders unconditionally on `is_super_admin`) needs a mode check, as defense-in-depth for the theoretical case of a super-admin flag existing in an on-prem database (e.g. a database migrated from a SaaS export — out of scope to prevent, but the nav shouldn't advertise a section that behaves inconsistently).
5. **No backend changes.** `GET /api/instance` already returns everything needed; this is a frontend-only batch.

## Design

### New module: `web/src/lib/instance.ts`

```ts
import api from './api';

export type InstanceInfo = {
  mode: 'saas' | 'onprem';
  version: string;
  license: unknown;
};

let cached: Promise<InstanceInfo> | null = null;

/**
 * Fetches GET /api/instance once per browser session and memoizes the
 * result (mode never changes without a redeploy). On failure, resolves to
 * `{mode: 'onprem', ...}` — mirrors the backend's own DeploymentMode
 * default, so a network hiccup fails toward hiding saas-only surfaces,
 * never toward exposing them.
 */
export function getInstanceInfo(): Promise<InstanceInfo> {
  if (!cached) {
    cached = api
      .get('/api/instance')
      .then((res) => res.data as InstanceInfo)
      .catch(() => ({ mode: 'onprem' as const, version: 'unknown', license: null }));
  }
  return cached;
}
```

### New hook: `web/src/hooks/useInstanceMode.ts`

```ts
import { useEffect, useState } from 'react';
import { getInstanceInfo } from '@/lib/instance';

/**
 * Returns the current deployment mode, defaulting to 'onprem' (the safe
 * default) until the one-time GET /api/instance fetch resolves.
 */
export function useInstanceMode(): { mode: 'saas' | 'onprem'; loading: boolean } {
  const [mode, setMode] = useState<'saas' | 'onprem'>('onprem');
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;
    getInstanceInfo().then((info) => {
      if (!cancelled) {
        setMode(info.mode);
        setLoading(false);
      }
    });
    return () => {
      cancelled = true;
    };
  }, []);

  return { mode, loading };
}
```

### `Login.tsx` — hide the Register link

The existing block:

```tsx
<p className="text-sm text-muted-foreground">
  {t("dontHaveAccount")}{" "}
  <Link to="/register" className="underline hover:text-primary">
    {t("register")}
  </Link>
</p>
```

is wrapped in `{mode === 'saas' && (...)}`, using `useInstanceMode()`. The "Staff login? QR Login" block below it is unaffected — QR login has no registration/self-signup implications and works identically in both modes.

### `App.tsx` — new `SaasOnlyRoute` guard, mirrors `ProtectedRoute`

```tsx
function SaasOnlyRoute({ children }: { children: JSX.Element }) {
  const { mode, loading } = useInstanceMode();
  if (loading) {
    return null; // brief blank render while the one-time fetch resolves; avoids a flash of the form that then redirects
  }
  if (mode !== 'saas') {
    return <Navigate to="/login" replace />;
  }
  return children;
}
```

Route registration changes from:

```tsx
<Route path="/register" element={<RegisterPage />} />
```

to:

```tsx
<Route path="/register" element={<SaasOnlyRoute><RegisterPage /></SaasOnlyRoute>} />
```

### `Layout.tsx` — gate the Super Admin nav item on mode too

The existing:

```tsx
{user?.is_super_admin && (
  <Button variant="ghost" size="sm" asChild className="text-primary">
    <Link to="/super-admin">
      <Shield className="mr-2 h-4 w-4" />
      {t('superAdmin')}
    </Link>
  </Button>
)}
```

becomes:

```tsx
{user?.is_super_admin && mode === 'saas' && (
  <Button variant="ghost" size="sm" asChild className="text-primary">
    <Link to="/super-admin">
      <Shield className="mr-2 h-4 w-4" />
      {t('superAdmin')}
    </Link>
  </Button>
)}
```

with `const { mode } = useInstanceMode();` added alongside `Layout`'s existing `user` memo.

### Data flow

```
App mount (any page) → first call to useInstanceMode() anywhere
  → getInstanceInfo() → GET /api/instance (once, memoized)
    → {mode: 'saas'|'onprem', version, license}   (success)
    → {mode: 'onprem', ...}                        (network/parse failure — safe default)

Login page:            mode === 'saas' → show "Register" link; else hidden
/register navigation:  mode === 'saas' → render RegisterPage; else redirect to /login
Layout nav (any page):  user.is_super_admin && mode === 'saas' → show "Super Admin" link; else hidden
```

### Error handling

No new error classes. `getInstanceInfo()`'s own `.catch()` is the only error handling needed — every consumer receives a resolved `InstanceInfo` (never a rejected promise), defaulting to `onprem` on failure. No loading spinners needed beyond `SaasOnlyRoute`'s brief `null` render (the fetch is small and local-network-fast; `Login.tsx`/`Layout.tsx` simply render as `onprem` until the fetch resolves, then re-render if it turns out to be `saas` — an acceptable, brief layout shift, not a blocking spinner).

## Testing

- **`instance.ts`:** `getInstanceInfo()` returns the parsed response on success; returns the `onprem` fallback on a rejected request; a second call does not issue a second HTTP request (memoization).
- **`useInstanceMode.ts`:** resolves to `saas`/`onprem` matching the mocked fetch; starts as `onprem`/`loading=true` before resolution.
- **`Login.tsx`:** Register link present when mode is `saas`, absent when `onprem`.
- **`App.tsx` / `SaasOnlyRoute`:** navigating to `/register` renders `RegisterPage` when mode is `saas`; redirects to `/login` when mode is `onprem`.
- **`Layout.tsx`:** Super Admin nav item requires BOTH `user.is_super_admin` AND `mode === 'saas'` — test the case where `is_super_admin` is true but mode is `onprem` (item must still be hidden), not just the existing `is_super_admin` true/false cases.
- **Manual/live-browser click-through** (established pattern): start the web app against an `onprem` backend instance — confirm no Register link on `/login`, confirm typing `/register` in the address bar redirects to `/login`, confirm no Super Admin nav item even when manually flipping a test user's `is_super_admin` flag in the database. Then repeat against a `saas` backend instance confirming all three are present/reachable. Light/dark and EN/RU as usual.

## Out of scope

- No backend changes — `/api/instance`'s shape and mounting are unchanged from Phase 0/P1.1.
- No change to `/super-admin`'s own route guard (`ProtectedRoute requireSuperAdmin`) — already correctly blocks by `is_super_admin`, independent of mode; this batch only adds a mode check to the *nav link's* visibility, not the route's access control.
- No change to QR login, staff login, or any other on-prem/saas-agnostic auth surface.
- No app-wide "mode" React Context — a module-level memoized fetch + hook is enough for three call sites; introducing Context for this would be premature abstraction (YAGNI) unless a future batch needs mode in many more places.

## Risks

| Risk | Mitigation |
|---|---|
| A `saas`-mode viewer sees a brief flash of `onprem` behavior (hidden Register link, momentary blank `/register` route) before the one-time fetch resolves | Accepted — the fetch is a same-origin, unauthenticated, small JSON response; the window is milliseconds on any real network. Not worth a global loading gate on the whole app for this. |
| `getInstanceInfo()`'s module-level cache means a mode change would require a full page reload to pick up (no live-updating) | Not a real risk — `DEPLOYMENT_MODE` is a deploy-time backend config value, never changed while a browser tab is open. |
| Hiding the Super Admin nav item doesn't prevent a determined on-prem `is_super_admin` user from typing `/super-admin` directly (only the nav *link* is gated by mode in this batch, the route itself isn't) | Accepted per decision #4 — the route's existing `is_super_admin` guard is the real access control; this batch's mode check is cosmetic-but-honest nav hygiene, not a new security boundary. If reached, `/api/super-admin/*` calls still 404 from the backend. |
