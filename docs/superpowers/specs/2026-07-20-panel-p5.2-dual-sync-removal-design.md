# Panel P5.2 — Remove badge-template dual-sync + legacy readers — Design

Second of three P5 ("Parity & cutover") sub-cycles, closing the transitional
badge-template machinery introduced in P3.1. Parent decomposition:
`docs/superpowers/specs/2026-07-13-panel-rewrite-design.md` §5 (P5) and the
P5.1 spec's §9 ("P5.2 — delete the badge-template dual-sync (both directions)
+ the column-first legacy readers, together"). This is a **backend-only**
cleanup cycle; no panel/web runtime behavior changes.

## 1. Background — the transitional machinery

P3.1 added a dedicated `events.badge_template` JSONB column (+
`badge_template_version`) as the new source of truth, but kept the legacy
`custom_fields['badgeTemplate']` key alive with two-way sync so the OLD web
badge editor and the new panel editor stayed mutually visible during the
rewrite:

- **Direction A — legacy → column.** `UpdateEvent` (`PUT /api/events/{id}`,
  `backend/internal/handler/events.go:140-152`) calls
  `Store.SyncBadgeTemplateFromLegacy` (`pg_store.go:717-728`) to mirror an
  object-typed `custom_fields['badgeTemplate']` into the column with an
  unconditional version bump.
- **Direction B — panel → legacy.** `Store.UpdateEventBadgeTemplate`
  (`pg_store.go:668-682`, the panel's `PUT /badge-template` path) writes the
  column AND mirrors the same bytes back into `custom_fields['badgeTemplate']`
  via `jsonb_set`, in the same version-guarded `UPDATE`.
- **Column-first reader.** `effectiveBadgeTemplate`
  (`badge_template.go:143-156`) reads the column first, falling back to
  `event.CustomFields['badgeTemplate']`. Its two callers are
  `badge_zpl.go:76` (the `POST /badge-zpl` handler) and `readiness.go:87`.

P5.1 deleted the OLD web badge editor (`web/src/pages/BadgeTemplateEditorV2.tsx`)
and the browser check-in kiosk (`web/src/pages/CheckinFullscreen.tsx`). The
panel writes badge templates ONLY through `PUT /badge-template` (never
`custom_fields`); its event-save paths are PATCH-only and carry no
`custom_fields`. So Direction A is now unreachable and the legacy key is
write-dead — the transitional machinery can come out.

## 2. What must NOT change (load-bearing)

- **`POST /api/events/{id}/badge-zpl` + `zpl.Generate` stay.** The Tauri
  `desktop/` kiosk (`desktop/src/pages/CheckinEvent.tsx:343`, a separate app
  P5.1 did not touch) still POSTs to it and trusts the backend to resolve the
  template server-side. So `effectiveBadgeTemplate` and its column read stay —
  only the legacy fallback is removed.
- **Readiness** (`GetEventReadiness`) keeps reading the template via the same
  `effectiveBadgeTemplate`; its "badge done" logic (non-empty string, or a map
  with a non-empty `elements` array) is unchanged — it just reads the column
  now.
- **`custom_fields` as a field on `PUT /api/events/{id}`** stays (it may carry
  other custom fields). We stop syncing `badgeTemplate` out of it; we do not
  remove the field or change the request/response contract. No OpenAPI change,
  so no panel client regeneration.
- `models.Event.BadgeTemplate` (`json:"-"`) and mobile's dead
  `Event.badge_template` field are out of scope (separate mobile track).

## 3. Data decision (user-approved 2026-07-20)

The environment holds **test data only** — legacy `custom_fields.badgeTemplate`
data can be discarded freely. Chosen approach: **backfill + scrub** via a new
migration, so the DB ends fully consistent (column authoritative, legacy key
gone), rather than leaving an inert stale key.

### Migration `000024_drop_badge_template_legacy_sync`

`up`:

```sql
-- Safety net: copy any object-typed legacy template into the column where the
-- column is still NULL (catches any event a transient log-don't-fail sync
-- missed). Idempotent re-run of 000018's object-typed-only backfill.
UPDATE events
SET badge_template = custom_fields->'badgeTemplate',
    badge_template_version = GREATEST(badge_template_version, 1)
WHERE badge_template IS NULL
  AND jsonb_typeof(custom_fields->'badgeTemplate') = 'object';

-- The column is now the sole source of truth; drop the redundant legacy key.
UPDATE events
SET custom_fields = custom_fields - 'badgeTemplate'
WHERE custom_fields ? 'badgeTemplate';
```

`down` (schema-reversible — reconstructs the legacy key from the column, which
held identical bytes while the sync was live):

```sql
UPDATE events
SET custom_fields = jsonb_set(coalesce(custom_fields, '{}'::jsonb),
                              '{badgeTemplate}', badge_template)
WHERE badge_template IS NOT NULL;
```

Next free migration number is `000024` (highest existing is
`000023_equipment_registry`). Column stays `JSONB NULL`.

## 4. Code removals

- **Direction A:** delete the `if legacyTemplate, ok := ...` sync block in
  `UpdateEvent` (`events.go:125-152`, plus the explanatory comment). Remove any
  now-unused imports (`json`, `log`) from `events.go` if nothing else uses them.
  Delete `SyncBadgeTemplateFromLegacy` from the store implementation
  (`pg_store.go`), from the `Store` interface, and from the test fake
  (`testsupport_test.go`).
- **Direction B:** in `UpdateEventBadgeTemplate` (`pg_store.go:668-682`), remove
  the `custom_fields = jsonb_set(...)` fragment so the `UPDATE` writes only
  `badge_template` + `badge_template_version` (+ `updated_at`), still guarded by
  `badge_template_version = $3`. Update its doc comment (drop the "transitional
  mirror" rationale).
- **Reader:** simplify `effectiveBadgeTemplate` (`badge_template.go`) to
  column-only — decode and return the column when non-empty, else return `nil`
  (no `custom_fields['badgeTemplate']` fallback). Update the doc comment. Both
  callers (`badge_zpl.go`, `readiness.go`) already null-check the result, so a
  NULL column reads as "no template" uniformly.

## 5. Test changes

- **Delete** `events_badge_template_sync_test.go` (tests the removed Direction-A
  sync) and `events_badge_template_dualwrite_contract_test.go` (tests both
  directions together).
- **Update** `pg_store_badge_template_test.go`: remove
  `TestUpdateEventBadgeTemplateMirrorsSameBytesIntoLegacyKey` (or invert it to
  assert the query no longer contains `jsonb_set`/`custom_fields`), remove the
  two `SyncBadgeTemplateFromLegacy` tests; keep the column read/write and
  version-conflict tests.
- **Update** `openapi_contract_badge_template_p3_test.go`: remove
  `TestOpenAPIContract_BadgeZPL_FallsBackToLegacyMapWhenColumnAbsent`; keep the
  column test; ADD a test that a NULL column (no legacy key) yields "no badge"
  (badge-zpl errors / readiness `badgeDone=false`), pinning the column-only
  behavior.
- **Update** `openapi_contract_events_p1_test.go`: the readiness-shape and
  PATCH tests that seed `CustomFields['badgeTemplate']` must instead seed the
  `badge_template` column (`event.BadgeTemplate`), since readiness now reads the
  column. `TestContractPatchEvent`'s "custom_fields immune to PATCH" assertion
  stays valid (PATCH still never writes custom_fields) but should no longer rely
  on a `badgeTemplate` sub-key.
- **Update** `testsupport_test.go`: remove the `syncBadgeTemplateFromLegacy`
  fake field + its wiring; keep `getEventBadgeTemplate`/`updateEventBadgeTemplate`.
- Migration 000018's backfill has no dedicated test; 000024 gets one only to
  the extent the existing migration-parse tests
  (`pg_store_migrations_test.go`: no-collision / no-duplicate-version) cover it —
  no data-behavior test harness exists for migrations, so none is invented here.
