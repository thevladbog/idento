# Panel P5.3.1 — Docs refresh (panel-rewrite driven fixes) — Design

First of five P5.3 sub-cycles (the final phase of the panel rewrite
initiative). Parent decomposition: the P5 spec's §9 "docs" line item,
expanded 2026-07-21 into five independent sub-cycles — P5.3.1 (this one),
P5.3.2 (Playwright e2e), P5.3.3 (WCAG-AA a11y), P5.3.4 (EN/RU parity for
`web/`), P5.3.5 (5k-attendee perf validation).

## 1. Scope

Fixes the top-level docs (`README.md`, `README.ru.md`, `DEVELOPMENT.md`,
`onprem/INSTALL.md`) and the local dev startup scripts wherever they are
**factually wrong because of the panel rewrite / P5.1 cutover** — they still
describe `web/` as the sole customer-facing app and never mention `panel/`,
which per P5.1 is now the primary app (served at `/`), with `web/` reduced to
the super-admin console (served at `/super-admin/` in the combined production
image; a separate origin/port in local dev).

Explicitly OUT of scope (pre-existing staleness, not caused by the rewrite,
left untouched to avoid scope creep beyond this cycle's purpose):

- README's "Printing Agent: `http://localhost:3000`" — the agent's real
  default port is `12345` (`agent/main.go:302`). Wrong before the rewrite,
  wrong after; unrelated to panel/web.
- Redis appears in `docker-compose.yml` (`idento_redis` container) but no Go
  file under `backend/internal` imports/uses a Redis client. Whether that's
  dead infra or a stale doc claim is a separate investigation.
- `CONTRIBUTING.md` — checked, contains zero `web/`/`panel/`/port references;
  nothing to fix.
- Any of P5.3.2-5 (e2e, a11y, i18n parity, perf) — separate sub-cycles.

## 2. Decisions

1. **The local-dev startup scripts (`scripts/start-all.{sh,ps1,bat}`,
   `make dev`) currently start ONLY `web/`, never `panel/`.** This is a real
   functional gap the rewrite created (before P0, `web/` was the only
   frontend to start), not just wording — the "one command starts
   EVERYTHING" promise in the README is false today for the app that
   actually matters post-cutover. In scope: update the scripts to also start
   `panel/`'s dev server, mirroring the existing `web/`/`agent/` pattern
   (background process + its own log file).
2. **`panel/vite.config.ts` has no explicit `server.port`**, so it defaults
   to Vite's `5173` — the same default `web/` uses (also unset). Running both
   simultaneously today gives an unpredictable port for whichever starts
   second (Vite auto-increments on conflict rather than failing, since
   neither sets `strictPort`). Docs cannot honestly state a fixed panel URL
   against that. In scope: set `server: { port: 5174 }` in
   `panel/vite.config.ts` so panel has a stable, documented local URL and
   both dev servers can run side by side deterministically.
3. **Local dev vs production URL shape differ and both must be documented
   correctly, not conflated:** in local dev, `panel/` and `web/` are two
   separate Vite dev servers on two different ports/origins (`5173`
   web-console, `5174` panel, per decision 2). In the deployed combined image
   (P5.1), they are the SAME origin, path-split (`panel/` at `/`, console at
   `/super-admin/`). The docs must describe the dev-mode reality (two ports)
   in the Quick Start section, not the production path-split.

## 3. Files & changes

### `scripts/start-all.sh` / `.ps1` / `.bat`

Add a `panel/` startup step alongside the existing `web/` step: `cd panel &&
npm run dev` backgrounded to its own log file (`logs/panel.log`), following
the exact pattern already used for `web`/`agent`. Update the printed summary
lines (e.g. the `.sh`'s `echo "🌐 Web: ..."` block) to list both URLs.

### `panel/vite.config.ts`

Add `server: { port: 5174 }` to the `defineConfig` call.

### `README.md` + `README.ru.md` (mirrored — identical structural edits, RU
   translated)

- **Quick Start → "What happens on startup?"**: add "Starts Panel (React)"
  to the bullet list, alongside the existing "Starts Web Frontend (React)".
- **"System Access"**: split into panel (primary app, `http://localhost:5174`)
  and console/super-admin (`http://localhost:5173`), both noted as local-dev
  ports (distinct from the combined-image production path-split). Keep the
  existing login credentials line, backend/agent/pgAdmin lines unchanged
  (still accurate).
- **"How to Use" → "4️⃣ Check-in"**: replace `**Web**: /checkin - search,
  scan, one button` with the panel's check-in station flow (open an event →
  Check-in station — scan, search, one-button verdict).
- **Architecture diagram**: add a "Panel (Check-in, Events)" box distinct
  from the existing "Web Admin" box (rename to "Console (Super-Admin)" to
  match its post-cutover role), both still flowing into the same Backend box.
- **Project Structure tree**: add a `├── panel/` line (React Panel — Vite,
  TanStack Router/Query, `@idento/ui`) and reword the existing `├── web/`
  line to describe it as the Super-Admin Console (not "React Admin"
  generically).
- **Tech Stack table**: add a **Panel** row (React 19, TanStack Router+Query,
  `@idento/ui`, Tailwind v4) distinct from the existing **Web** row (kept as
  React 18 — that's still accurate for the console).

### `DEVELOPMENT.md`

- **"Starting Development Environment" → Manual commands**: add `cd panel &&
  npm run dev &` alongside the existing `cd web && npm run dev &` line.
- **Ports-in-use troubleshooting** (Windows/macOS sections currently only
  mention `5173`): add the panel port (`5174`) alongside each existing
  `5173` reference so the troubleshooting steps cover both dev servers.

### `onprem/INSTALL.md`

The main install walkthrough (steps 1-7) generically says "Open the web UI"
and "In the web UI, go to the Users page" without naming which app that is.
The Badge Printing & Scanning section already correctly names `panel/`. Bring
the main walkthrough in line: clarify that the day-to-day operator app is
`panel/` (served at the deployment's root URL) and the tenant-admin console
is at `/super-admin/` on the same origin — matching the ALREADY-CORRECT
production path-split (this file describes a deployed instance, not local
dev, so no port split here — this is the one file in scope where the
combined-image path-split IS the right model to describe).

## 4. Testing

Documentation + two small config/script changes — no application test
suite exercises these. Verification is manual:

- Read-through each edited file for internal consistency (no leftover
  contradictory "web is the admin" sentence elsewhere in the same file).
- Run the updated `scripts/start-all.sh` (or `.ps1` on Windows, whichever
  the implementer has available) and confirm `panel/` actually starts and
  serves on `5174` alongside `web/` on `5173`, with a `logs/panel.log`
  created.
- `cd panel && npm run dev` manually, confirm it binds to `5174` (the new
  explicit port) via the vite.config.ts change, without needing the
  start-all script.
- No CI job covers markdown prose; the config/script changes are covered by
  the manual startup check above, not an automated gate.
