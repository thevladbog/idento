# P0.3 ‚Äî openapi truth-up, generated panel client, CI/Docker ‚Äî Design

**Date:** 2026-07-14
**Parent spec:** [2026-07-13-panel-rewrite-design.md](2026-07-13-panel-rewrite-design.md) (¬ß3.2 data layer, ¬ß5 P0, ¬ßrisks "openapi.yaml drift")
**Status:** approved by user 2026-07-14

## 1. Goal

Make `backend/openapi.yaml` the enforceable contract for every endpoint the
panel consumes, generate the panel's typed API client from it, and give the
panel a production Docker image with CI verification ‚Äî completing phase P0 of
the customer-panel rewrite.

Three deliverables, one branch:

1. **Truth-up** ‚Äî `backend/openapi.yaml` rewritten to match the Go handlers
   exactly for the panel-consumed + shared surface (~60 routes).
2. **Enforcement** ‚Äî Go contract tests (kin-openapi) validate real handler
   responses against the spec, with a coverage ledger so the spec cannot rot
   silently; a CI drift check keeps the generated client in sync.
3. **Client + image** ‚Äî `openapi-typescript` + `openapi-fetch` client in
   `panel/`, migration of the P0.2 hand-written calls, `panel/Dockerfile`
   (repo-root context, unprivileged nginx + `env.js`), published from
   `release.yml` as `ghcr.io/<repo>-panel`.

## 2. Decisions (user-approved)

| Question | Decision |
|---|---|
| Truth-up depth | **Panel + shared endpoints** (~60 routes). Mobile-only, sync, public-import, and super-admin surfaces stay undocumented, marked out-of-scope in the file header; they get documented in their own tracks. |
| Spec honesty mechanism | **Go contract tests with kin-openapi** in the existing httptest harness, plus a coverage ledger asserting every documented (path, method) was validated at least once. |
| Docker scope | **Dockerfile + publish at release.** CI builds the image (no push) on panel changes; `release.yml` publishes `ghcr.io/<repo>-panel:{tag,latest}`. Compose topology unchanged until P5 cutover. |
| Client integration | **Full `openapi-fetch` client** (spec ¬ß3.2): generated `schema.d.ts` committed, thin middleware for auth + error normalization; the five hand-written P0.2 calls migrate to it. Types-only integration was rejected ‚Äî it doesn't enforce paths/params at call sites. |

- `/api/mobile/*` and `POST /api/zones/{zone_id}/scan` ‚Äî mobile track
- `POST /api/events/{event_id}/checkins/batch` ‚Äî mobile offline sync

- `/api/super-admin/*` ‚Äî console track (stays in `web/`)

### Spec conventions

- Request/response schemas derived from the actual Go structs (`internal/store`
  models and handler request/response types), not invented.
- One shared `Error` component: `{code?: string, error: string}` ‚Äî matches
  what handlers return today (P0.2 verified `{code:"tenant_suspended",error}`
  on 403).
- Rate-limited endpoints (`login`, `login-qr`, `stations/provision`,
  `zones/checkin`) document the 429 response.
- Auth: bearer JWT security scheme applied to the `/api/*` protected group;
  public endpoints explicitly `security: []`.
- Non-JSON responses documented with their real content types: attendee CSV
  export (`text/csv`), font files (`font/ttf` / binary), font CSS
  (`text/css`), QR images (`image/png`), badge ZPL (`text/plain`). Contract
  validation for these checks status + content type, not a JSON schema.

## 4. Contract tests (backend)

- New dependency: `github.com/getkin/kin-openapi` (used by tests only; Go has
  no test-scoped deps, acceptable).
- Test helper in the existing `backend/internal/handler` test support:
  `validateResponse(t, method, path, rec)` ‚Äî loads `openapi.yaml` once
  (`sync.Once`), routes the request to the matching operation, validates
  response status + body against the schema, fails the test with a readable
  diff on mismatch.
- **Coverage ledger:** the helper records every validated `(method, path)`
  pair in a package-level registry. `TestMain` (after `m.Run()`) asserts the
  registry covers the full documented-endpoint list. Adding a path to
  openapi.yaml without a contract test fails the backend test job; the error
  message names the missing pairs.
- The ledger assertion is gated behind an env var set in CI (e.g.
  `OPENAPI_COVERAGE=1`) ‚Äî otherwise a local `go test -run TestX` on a single
  test would spuriously fail coverage. CI always sets it; locally the full
  suite can opt in.
- Existing handler tests get `validateResponse` calls wired in where they
  already exercise a documented endpoint; new minimal happy-path tests are
  added only for documented endpoints no existing test covers.

## 5. Generated client (panel)

- `openapi-typescript` (dev dependency) generates
  `panel/src/shared/api/schema.d.ts` from `backend/openapi.yaml`;
  the file is **committed** (no codegen at build time).
- `openapi-fetch` (runtime, ~2 KB) client:
  `createClient<paths>({ baseUrl: getApiBaseUrl() })` with middleware:
  - `onRequest`: attach `Authorization: Bearer <token>` from the session
    module when a token exists;
  - `onResponse`: on `!res.ok`, parse the error body and **throw `ApiError`**
    (same class as today) ‚Äî so TanStack Query's `QueryCache`/`MutationCache`
    global handlers (tenant-suspended detection, global 401) keep working
    unchanged.
- npm script `generate:api` in `panel/package.json`
  (`openapi-typescript ../backend/openapi.yaml -o src/shared/api/schema.d.ts`).
- Migration: the five hand-written functions in
  `panel/src/shared/api/client.ts` (`login`, `register`, `loginWithQr`,
  `getInstance`, `switchTenant`) are re-implemented on the generated client;
  `getApiBaseUrl`, `ApiError`, `authErrorKey`, session module, and all
  screens' call signatures stay unchanged. Hand-written `types.ts` shapes are
  replaced by generated types where they describe API payloads.
- `panel/AGENTS.md` gains the openapi-first workflow rule: new endpoint ‚Üí
  document in `backend/openapi.yaml` ‚Üí add Go contract test ‚Üí
  `npm run generate:api -w panel` ‚Üí use typed client. Never hand-write a
  fetch against an undocumented path.

## 6. CI / Docker

### panel/Dockerfile (repo-root build context)

- Build stage (`node:24-alpine`): copy root `package.json`,
  `package-lock.json`, `.npmrc`, `packages/`, `panel/` ‚Üí `npm ci` ‚Üí
  `npm run build -w panel`. Repo-root context is required because the panel
  source-imports `@idento/ui` via npm workspaces.
- Runtime stage: byte-for-byte the proven `web/` pattern ‚Äî
  `nginxinc/nginx-unprivileged:1.27-alpine`, chown'd html root, SPA-fallback
  `nginx.conf`, `env.js.template` (already exists from P0.2) rendered by the
  base image's envsubst entrypoint at container start (`PUBLIC_API_URL`),
  `Cache-Control: no-store` on `/env.js`, immutable caching on `/assets/`.
- New root `.dockerignore` keeps the context small (`.git`, `node_modules`,
  `mobile/`, `desktop/`, `web/`, `backend/`, `onprem/`, `docs/`, `.claude/`,
  build outputs). `web/`'s own build is unaffected (its context is `./web`).

### CI wiring

- **Drift check:** panel validate jobs run `npm run generate:api -w panel`
  then `git diff --exit-code` on `schema.d.ts` ‚Äî a spec edit without a
  committed regeneration fails CI.
- **`changes` filter:** `backend/openapi.yaml` added to the `panel` filter in
  `ci.yml` so spec edits trigger the panel jobs (backend jobs already trigger
  via `backend/**`).
- **Image build verification:** new `ci.yml` job (on `panel == 'true'`)
  builds the image with `push: false`.
- **Contract tests** run inside the existing backend test job automatically
  (`go test ./...`).

### release.yml

- New `panel-image` job mirroring `web-image`: `context: .`,
  `file: panel/Dockerfile`, tags `ghcr.io/<repo>-panel:{tag,latest}`,
  multi-arch (amd64/arm64), needs `validate`.
- Compose files (`docker-compose*.yml`, `onprem/`) are **not** modified ‚Äî
  topology changes only at P5 cutover.

## 7. Testing summary

| Layer | Mechanism |
|---|---|
| Spec ‚Üî Go code | kin-openapi contract tests + coverage ledger (backend test job) |
| Spec ‚Üî TS client | `generate:api` drift check (panel validate job) |
| Client behavior | Vitest unit tests for middleware (auth header attach, ApiError normalization incl. `tenant_suspended` and 401 shapes), migrated call-site tests keep passing unchanged |
| Image | CI docker build (PR) + published multi-arch image (release) |

## 8. Out of scope

- Combined nginx image serving panel + console (`/super-admin/`) ‚Äî P5.
- Any docker-compose / onprem bundle changes ‚Äî P5.
- TanStack Query hooks per feature ‚Äî land with real screens in P1+.
- Documenting mobile/sync/public-import/super-admin endpoints ‚Äî own tracks.
- JWT ‚Üí cookie migration ‚Äî separate track.

## 9. Risks

- **Schema mistakes in rarely-tested endpoints** ‚Äî mitigated by the coverage
  ledger forcing at least one validated response per documented route.
- **kin-openapi strictness surprises** (e.g. `additionalProperties`,
  nullable) ‚Äî resolve by fixing the spec to match reality, never by loosening
  validation; handlers are the source of truth in P0.3.
- **Repo-root Docker context size** ‚Äî mitigated by root `.dockerignore`;
  verify final context size in the plan's Docker task.
- **`schema.d.ts` merge conflicts** ‚Äî single generated file, regenerate on
  conflict; documented in `panel/AGENTS.md`.
