# Полный аудит Idento: безопасность, баги, качество, обновление зависимостей

- **Дата:** 2026-07-09
- **Статус:** утверждено (дизайн согласован с владельцем проекта)
- **Тип:** процессный спек — аудит и серия волн исправлений

## Цель

Провести полный аудит всех подсистем Idento, составить приоритизированный отчёт
о найденных проблемах и исправить их последовательными волнами: сначала
безопасность и обновление зависимостей, затем баги, затем качество кода.

## Охват

Все шесть подсистем репозитория:

| Подсистема | Стек | Примечание |
|---|---|---|
| `backend/` | Go 1.25, Echo, pgx, golang-jwt | ядро API, аутентификация, БД |
| `agent/` | Go, go.bug.st/serial, rs/cors | принтер-агент |
| `web/` | React 18, Radix UI, axios, Tailwind | основное веб-приложение |
| `landing/` | Next.js 16, React 19, next-intl | не закоммичен — сначала baseline |
| `desktop/` | Tauri 2 + React 18 | киоск-приложение |
| `mobile/` | Kotlin Multiplatform (Android + iOS) | Gradle-каталоги версий |

## Ключевые решения (утверждены)

1. **Охват:** все подсистемы сразу; работа разбита на последовательные фазы.
2. **Глубина аудита:** безопасность + корректность (баги) + качество кода.
3. **Политика обновления зависимостей — умеренная:** все minor/patch и все
   security-фиксы обязательны; мажорные версии — только если текущая версия
   уязвима или больше не поддерживается. Мажорные миграции «ради свежести»
   (например, React 18 → 19 в `web/` и `desktop/`) не выполняются.
4. **Критерий готовности:** сканеры чистые + сборка проходит + существующие
   тесты зелёные; каждая находка исправлена или задокументирована с обоснованием.
5. **`landing/`:** сначала коммитится как есть (baseline), исправления — поверх,
   отдельными коммитами.
6. **Актуальность версий** проверяется через Context7 (документация библиотек),
   а не по памяти модели.

## Архитектура процесса

### Фаза 0 — фиксация базы

- Добавить `mobile/.watchman-cookie-*` в `.gitignore` (служебный файл Watchman).
- Закоммитить `landing/` как есть — baseline для читаемых диффов аудита.

### Фаза 1 — аудит

Три источника находок:

1. **Автоматические сканеры:**
   - Go: `govulncheck ./...`, `go list -m -u all` (backend, agent);
   - JS: `npm audit`, `npm outdated` (web, landing, desktop);
   - Mobile: анализ версий Gradle-зависимостей + проверка известных CVE
     ключевых библиотек.
2. **Проверка актуальности через Context7** для ключевых библиотек:
   Echo, pgx, golang-jwt, React, Next.js, next-intl, Tauri, Radix, axios,
   framer-motion и др. — текущие безопасные версии и заметки о миграции.
3. **Ручной обзор кода параллельными агентами** — на каждую подсистему по трём
   измерениям:
   - *безопасность:* аутентификация/авторизация, инъекции (SQL и пр.), секреты
     в коде, CORS, валидация входных данных, хранение паролей/токенов;
   - *корректность:* логические ошибки, race conditions, необработанные ошибки,
     краевые случаи, целостность данных;
   - *качество:* дублирование, разросшиеся файлы, слабые границы модулей,
     отсутствие тестов в критичных местах.

   Каждая находка проходит адверсариальную проверку независимым агентом;
   неподтверждённые находки отсеиваются или помечаются как «предположение».

**Результат:** отчёт `docs/audit/2026-07-09-audit-report.md`. Формат находки:
подсистема / файл:строка / описание / влияние / серьёзность / рекомендация.

**Шкала серьёзности:**

- **Critical** — эксплуатируемая уязвимость или потеря/порча данных;
- **High** — уязвимость с ограничивающими условиями или баг, ломающий ключевой
  сценарий (чек-ин, печать, логин);
- **Medium** — баг или слабость с обходным путём, устаревшая уязвимая
  зависимость без известного эксплойта;
- **Low** — качество кода, tech debt, некритичные устаревания.

**Гейт:** отчёт показывается владельцу проекта до начала любых исправлений;
владелец может вычеркнуть находки или изменить приоритеты.

### Фаза 2 — волна «Безопасность + зависимости»

- Обновление зависимостей по умеренной политике, **малыми партиями** со сборкой
  и тестами после каждой партии — регрессия видна сразу, откат тривиален.
- Кодовые security-фиксы: Critical и High — обязательно; Medium — в этой волне,
  если правка небольшая; для трудоёмких Medium решение (чинить сейчас или в
  backlog) принимает владелец на гейте отчёта; Low — в backlog.
- Оформление: отдельная ветка и PR на группу — `audit/security-go`,
  `audit/security-js`, `audit/security-mobile`.

### Фаза 3 — волна «Баги»

Исправление находок категории «корректность». Где возможно — сначала тест,
воспроизводящий баг, затем фикс (TDD). Своя ветка/PR.

### Фаза 4 — волна «Качество»

Только адресные улучшения из отчёта, подтверждённые владельцем: распил
разросшихся файлов, устранение дублирования, тесты для дырявых мест.
Никакого попутного рефакторинга вне списка.

## Верификация (каждая волна)

- **Go:** `go build ./...` + `go test ./...` + `govulncheck` без критичных находок.
- **JS:** `npm run build` + тесты проекта (в `landing/` — Playwright) +
  `npm audit` без high/critical.
- **Mobile:** сборка Gradle (Android) обязательна; сборка iOS — по возможности
  локального тулчейна (best-effort, отсутствие тулчейна фиксируется в отчёте).
- Итог по отчёту: каждая находка в статусе «исправлено» либо «отложено +
  обоснование». Молчаливых потерь нет.

## Не-цели

- Новые фичи и редизайн UI.
- Мажорные миграции без security-причины.
- Правки вне списка находок отчёта. Всё найденное по ходу, но вне охвата,
  заносится в раздел «Backlog» отчёта.

## Риски

| Риск | Смягчение |
|---|---|
| Обновление зависимости ломает сборку | малые партии, сборка+тесты после каждой, откат партии |
| Ложные срабатывания ручного обзора | адверсариальная проверка находок вторым агентом |
| iOS-сборка недоступна локально | best-effort; факт фиксируется в отчёте, не блокирует волну |
| Отчёт устареет до конца волн | волны идут сразу за отчётом, без длинных пауз |

## План работ

1. Спек (этот документ) → ревью владельцем.
2. Implementation-план для Фаз 0–1 (writing-plans) → исполнение → отчёт.
3. Гейт: владелец утверждает отчёт и приоритеты.
4. Планы и исполнение Фаз 2–4 (содержание зависит от находок отчёта).
