# On-Prem Phase 2 — P2.3 Distribution Bundle & Docs — Design

- **Date:** 2026-07-12
- **Status:** approved by user, pending implementation plan
- **Depends on:** P2.1 (merged, PR #40, bootstrap flow), P2.2 (merged, PR #42, frontend mode awareness). Phase 0's `docker-compose.prod.yml`, `backend/Dockerfile`, `web/Dockerfile`, and `.github/workflows/release.yml` (backend image + binaries publishing) already exist and are extended here, not replaced.
- **Roadmap item:** `docs/DUAL_DISTRIBUTION_REWORK.md` § Phase 2, P2.3.
- **Owner decision (2026-07-11, already recorded):** no Terraform/Ansible anywhere in Phase 2 — the on-prem deliverable stays tarball + docker-compose + docs, extended with an `install.sh` preflight.

## Goal

Ship a versioned release artifact, `idento-onprem-vX.Y.Z.tar.gz`, that takes a fresh Linux host with only Docker installed to a working check-in system by following `INSTALL.md` alone — no source checkout, no manual SQL, no build toolchain. Upgrades are `docker compose pull && docker compose up -d`.

## Key decisions (confirmed with user)

1. **Runtime config injection for the frontend's API URL, not build-time baking.** The web frontend currently bakes `VITE_API_URL` into its JS bundle at Docker *build* time (`web/Dockerfile`'s `ARG`/`ENV` + `npm run build`), which is why `docker-compose.prod.yml` must `build:` the web image locally with each deployment's own URL as a build arg — incompatible with the roadmap's stated `docker compose pull` upgrade path, since a single published image can't have a customer-specific URL baked in at *our* build time. Fixed via the nginx-unprivileged base image's built-in template mechanism (see Design §1) — zero custom entrypoint script, two new lines in the Dockerfile plus one new template file.
2. **A new top-level `onprem/` directory holds the bundle's source files** (`docker-compose.yml`, curated `.env.example`, `install.sh`, `backup.sh`, `restore.sh`, `INSTALL.md`), packaged verbatim into the release tarball. Kept separate from `docker-compose.prod.yml` (our own build-from-source deploy, unchanged in shape) since the two serve different audiences and now differ in one more way: the bundle's compose file references published `ghcr.io/...` images, `docker-compose.prod.yml` still builds from source.
3. **The bundle's `.env.example` is curated, not the root one.** The root `.env.example` mixes dev-only, SaaS-only, and on-prem-only guidance; a customer following `INSTALL.md` should see only what applies to them (Postgres credentials, `JWT_SECRET`, `CORS_ALLOWED_ORIGINS`, `PUBLIC_API_URL`, `VERSION`, `IDENTO_ADMIN_*`, `IDENTO_ORG_NAME`).
4. **`install.sh` auto-generates only pure secrets** (`JWT_SECRET`, `POSTGRES_PASSWORD` via `openssl rand -hex 32`), never customer-identifying values (`IDENTO_ADMIN_EMAIL`/`PASSWORD`, `PUBLIC_API_URL`) — consistent with P2.1's existing stance that bootstrap credentials are the operator's own choice, not invented for them. The script does not run `docker compose up` itself; the operator reviews the generated `.env`, fills in the placeholders, then starts the stack themselves per `INSTALL.md`.
5. **Backup/restore wraps `pg_dump -Fc`/`pg_restore`** against the bundle's `db` compose service, run via `docker compose exec` — no new backup infrastructure, no cloud storage integration (out of scope, matches the design's existing on-prem-is-simple stance).

## Design

### 1. Frontend runtime config injection

**New file `web/env.js.template`:**
```
window.__ENV__ = { API_URL: "${PUBLIC_API_URL}" };
```

**`web/Dockerfile`** — add two lines to the final (nginx) stage:
```dockerfile
FROM nginxinc/nginx-unprivileged:1.27-alpine
COPY --from=build /src/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY env.js.template /etc/nginx/templates/env.js.template
ENV NGINX_ENVSUBST_OUTPUT_DIR=/usr/share/nginx/html
EXPOSE 8080
```
No `ENTRYPOINT` override needed: the base image's default entrypoint chain already runs `/docker-entrypoint.d/20-envsubst-on-templates.sh` before starting nginx, which finds `*.template` files under `/etc/nginx/templates` (the default `NGINX_ENVSUBST_TEMPLATE_DIR`) and writes each one, with `${VAR}` substituted from the container's actual environment, to the corresponding path under `NGINX_ENVSUBST_OUTPUT_DIR`. Since `env.js.template` is the only template file shipped, this produces exactly one output: `/usr/share/nginx/html/env.js`, substituted from the `PUBLIC_API_URL` environment variable set on the container.

The build-time `ARG VITE_API_URL` / `ENV VITE_API_URL` lines in the build stage are **removed** — no longer required for any deployment path. (Local `npm run dev`/`vite build` outside Docker is unaffected; `import.meta.env.VITE_API_URL` still works as a `.env.local`-driven override for that case, per decision below.)

**`web/index.html`** — add before the module script:
```html
<script src="/env.js"></script>
<script type="module" src="/src/main.tsx"></script>
```

**`web/public/env.js`** (new, committed default — served as-is by `vite dev`/`vite preview` and as the fallback baked into any build that skips the Docker templating step):
```js
window.__ENV__ = {};
```

**`web/src/vite-env.d.ts`** — add a `Window` interface augmentation, following this file's existing convention of declaring build/runtime globals (it already declares `__APP_VERSION__` the same way):
```ts
declare global {
  interface Window {
    __ENV__?: { API_URL?: string };
  }
}
```
Without this, `window.__ENV__` does not type-check (`Property '__ENV__' does not exist on type 'Window & typeof globalThis'`) — there is no existing global-augmentation file elsewhere in `web/src` to piggyback on.

**`web/src/lib/api.ts`** — change the `baseURL` line from:
```ts
baseURL: import.meta.env.VITE_API_URL || 'http://localhost:8008',
```
to:
```ts
baseURL: window.__ENV__?.API_URL || import.meta.env.VITE_API_URL || 'http://localhost:8008',
```
Precedence: runtime-injected value (Docker/production) → build-time Vite env var (local dev override, unchanged existing behavior) → hardcoded localhost default (matches today exactly when neither is set).

**`docker-compose.prod.yml`** — the `web` service's `build.args.VITE_API_URL` is removed; a runtime `environment: PUBLIC_API_URL: ${PUBLIC_API_URL:?set PUBLIC_API_URL in .env}` entry is added instead (the `.env.example`'s existing `PUBLIC_API_URL` var is now consumed as a runtime container env var here too, not just by the new bundle — one mechanism, not two).

### 2. `onprem/` bundle source directory (new)

```
onprem/
  docker-compose.yml
  .env.example
  install.sh
  backup.sh
  restore.sh
  INSTALL.md
```

**`onprem/docker-compose.yml`** — same shape as `docker-compose.prod.yml`'s `db` service (unchanged), but `backend`/`web` reference published images instead of `build:`:
```yaml
services:
  db:
    image: postgres:16-alpine
    restart: always
    environment:
      POSTGRES_USER: ${POSTGRES_USER:-idento}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?set POSTGRES_PASSWORD in .env}
      POSTGRES_DB: ${POSTGRES_DB:-idento_db}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-idento} -d ${POSTGRES_DB:-idento_db}"]
      interval: 5s
      timeout: 3s
      retries: 12

  backend:
    image: ghcr.io/thevladbog/idento-backend:${VERSION:-latest}
    restart: always
    environment:
      DATABASE_URL: postgres://${POSTGRES_USER:-idento}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB:-idento_db}?sslmode=disable
      JWT_SECRET: ${JWT_SECRET:?set JWT_SECRET in .env}
      CORS_ALLOWED_ORIGINS: ${CORS_ALLOWED_ORIGINS:?set CORS_ALLOWED_ORIGINS in .env}
      DEPLOYMENT_MODE: onprem
      IDENTO_ADMIN_EMAIL: ${IDENTO_ADMIN_EMAIL:?set IDENTO_ADMIN_EMAIL in .env}
      IDENTO_ADMIN_PASSWORD: ${IDENTO_ADMIN_PASSWORD:?set IDENTO_ADMIN_PASSWORD in .env}
      IDENTO_ORG_NAME: ${IDENTO_ORG_NAME:-My Organization}
    ports:
      - "8008:8008"
    depends_on:
      db:
        condition: service_healthy

  web:
    image: ghcr.io/thevladbog/idento-web:${VERSION:-latest}
    restart: always
    environment:
      PUBLIC_API_URL: ${PUBLIC_API_URL:?set PUBLIC_API_URL in .env (e.g. http://your-host:8008)}
    ports:
      - "80:8080"
    depends_on:
      - backend

volumes:
  postgres_data:
```
`DEPLOYMENT_MODE: onprem` is hardcoded (not a `${...}` var) — the bundle only ever deploys on-prem installs; there is no reason for an operator to override it, and hardcoding removes one way to misconfigure a fresh install into unexpectedly running in `saas` mode.

**`onprem/.env.example`** — curated subset (Postgres creds, `JWT_SECRET`, `CORS_ALLOWED_ORIGINS`, `PUBLIC_API_URL`, `VERSION`, `IDENTO_ADMIN_EMAIL`/`PASSWORD`, `IDENTO_ORG_NAME`) with on-prem-specific comments — no `DATABASE_URL`/`PORT`/`DEPLOYMENT_MODE` (all fixed or computed in `docker-compose.yml` above), no dev-server or SaaS-mode guidance.

**`onprem/install.sh`** (bash, POSIX-portable where reasonable):
1. Check `docker` is on `PATH` and `docker compose version` succeeds (Compose v2 plugin, not the standalone v1 `docker-compose` binary) — exit with a clear message naming what's missing if not.
2. Check ports 80 and 8008 are free (`(command -v lsof && lsof -i :80,:8008) || true`-style best-effort check — a soft warning, not a hard failure, since some hosts won't have `lsof` and firewalls/binding differ).
3. Check available disk space on the current filesystem is above a small floor (e.g. 5 GB) via `df`.
4. If `.env` doesn't already exist, copy `.env.example` to `.env`, then in-place substitute `JWT_SECRET=` and `POSTGRES_PASSWORD=` with `openssl rand -hex 32`-generated values (`sed`-based substitution on the freshly copied file). If `.env` already exists, do not overwrite it — print a message and exit 0 (idempotent, safe to re-run).
5. Print a summary of what still needs manual editing (`IDENTO_ADMIN_EMAIL`, `IDENTO_ADMIN_PASSWORD`, `PUBLIC_API_URL`, `VERSION`) and the next command to run (`docker compose up -d`).

**`onprem/backup.sh`** — `docker compose exec -T db pg_dump -U "${POSTGRES_USER:-idento}" -Fc "${POSTGRES_DB:-idento_db}" > "backup-$(date +%Y%m%d-%H%M%S).dump"` (reads `POSTGRES_USER`/`POSTGRES_DB` from the same `.env` via `docker compose`'s own env-file loading — the script itself doesn't need to parse `.env`, it just shells out to `docker compose exec`, which already has the project's env context).

**`onprem/restore.sh`** — takes a dump-file path argument, confirms with the operator (destructive), then `docker compose exec -T db pg_restore -U "${POSTGRES_USER:-idento}" -d "${POSTGRES_DB:-idento_db}" --clean --if-exists` fed the dump file over stdin.

**`onprem/INSTALL.md`** — sections: Prerequisites (Docker Engine + Compose v2, recommended ~2 vCPU/2 GB RAM/10 GB disk as a starting estimate — not a hard minimum, just operator guidance), Install (extract tarball → `./install.sh` → edit `.env` → `docker compose up -d` → confirm via browser login with the bootstrap admin credentials from `.env`), Upgrade (`docker compose pull && docker compose up -d` — migrations auto-run at backend startup, already true since Phase 0), Backup & Restore (usage of the two scripts), Troubleshooting (a short pointer to `docker compose logs backend`).

### 3. Release workflow (`.github/workflows/release.yml`)

**New `web-image` job** (parallel to the existing `backend-image` job, same shape — multi-platform build+push, no `VITE_API_URL` build-arg since it's no longer needed):
```yaml
web-image:
  runs-on: ubuntu-latest
  permissions:
    contents: read
    packages: write
  steps:
    - uses: actions/checkout@v4
      with:
        persist-credentials: false
    - uses: docker/setup-qemu-action@v3
    - uses: docker/setup-buildx-action@v3
    - uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    - uses: docker/build-push-action@v6
      with:
        context: ./web
        platforms: linux/amd64,linux/arm64
        push: true
        tags: |
          ghcr.io/${{ github.repository }}-web:${{ github.ref_name }}
          ghcr.io/${{ github.repository }}-web:latest
```

**New `onprem-bundle` job** (parallel to `binaries`):
```yaml
onprem-bundle:
  runs-on: ubuntu-latest
  permissions:
    contents: read
  steps:
    - uses: actions/checkout@v4
      with:
        persist-credentials: false
    - name: Package bundle
      run: |
        mkdir -p dist
        tar -czf "dist/idento-onprem-${{ github.ref_name }}.tar.gz" -C onprem .
    - uses: actions/upload-artifact@v4
      with:
        name: idento-onprem-bundle
        path: dist/idento-onprem-${{ github.ref_name }}.tar.gz
```
No `${VERSION}`/tag substitution into `onprem/docker-compose.yml` at packaging time — the compose file already reads `VERSION` from `.env` (defaulting to `latest`), and `onprem/.env.example`'s comment tells the operator to set it to the tag they downloaded. This keeps the packaging step a plain, content-free `tar`, with no templating step that could silently drift from what's actually in the tarball.

**`release` job** — `needs:` gains `web-image` and `onprem-bundle`; `dist/*` (from `download-artifact` with `merge-multiple: true`, already the pattern) now also picks up the bundle tarball alongside the existing per-arch binaries.

## Testing

- **Frontend (Vitest):** `api.ts`'s `baseURL` resolution — `window.__ENV__.API_URL` set → used; unset/empty → falls through to `import.meta.env.VITE_API_URL`; both unset → falls through to the hardcoded default. Matches the existing test file location convention (`web/src/lib/__tests__/api.test.ts`, new — no test file exists for `api.ts` today).
- **`install.sh` / `backup.sh` / `restore.sh`:** no unit-test framework for shell scripts in this repo; verified via the live pass below instead (matches how `docker-compose.prod.yml` itself has never had automated tests — infra scripts are live-verified, application code is unit-tested).
- **Manual/live verification** (established pattern, before merge): build both images locally (`docker build ./backend`, `docker build ./web`), tag them to match what `onprem/docker-compose.yml` expects, run `./install.sh` against a scratch copy of the `onprem/` directory, edit the generated `.env`, `docker compose up -d`, confirm `/api/instance` responds, log in with the bootstrap admin, create an event and check in a test attendee (the actual "working check-in system" the roadmap's Accept criterion names) — then run `backup.sh`, tear down the stack (`docker compose down -v`, destroying the volume), bring it back up on a fresh volume, run `restore.sh` with the backup, and confirm the same event/attendee data is back. Also confirm `docker compose pull && docker compose up -d` doesn't error against the just-published `:latest` tags (or the actual release tag once one exists).

## Out of scope

- Terraform, Ansible, Helm/Kubernetes packaging — explicit owner decision, already recorded.
- Cloud backup storage (S3, etc.) — `backup.sh` writes a local file; where the operator sends it afterward is their own concern.
- A first-run web UI/setup wizard — env-var-driven only, per P2.1's already-approved design.
- License key validation/activation — `GET /api/instance`'s `license` field stays `null`, per the original dual-distribution design's explicit "contractual, not technical" stance on-prem licensing.
- Automated (non-shell-script) testing of `install.sh`/`backup.sh`/`restore.sh` — live-verified per Testing above, matching this repo's existing convention for infra scripts.
- Changing `docker-compose.prod.yml`'s `web` service from `build:` to a published `image:` — out of scope; only its `VITE_API_URL`→`PUBLIC_API_URL` mechanism changes (build-arg → runtime env var), matching the new Dockerfile capability, but the build-vs-image choice for our own deployment topology is a separate decision not part of this batch.

## Risks

| Risk | Mitigation |
|---|---|
| `install.sh`'s port/disk-space checks are best-effort (some hosts lack `lsof`, filesystem layouts vary) and could give a false "all clear" | Accepted — these are advisory pre-flight checks, not a guarantee; `docker compose up -d` itself will fail loudly on a real port conflict, which `INSTALL.md`'s troubleshooting section should point to. |
| A customer's Docker Compose is the legacy standalone v1 (`docker-compose`, hyphenated binary) rather than the v2 plugin (`docker compose`) | `install.sh` explicitly checks for `docker compose version` (v2) and fails with a clear message rather than silently trying the wrong invocation — v1 is EOL upstream and out of scope to support. |
| Publishing a `latest` web/backend image tag alongside versioned tags could let an operator accidentally track a moving target instead of pinning a release | Matches the existing `backend-image` job's already-established pattern (it already publishes both `:${tag}` and `:latest`) — not a new risk introduced by this batch, and `onprem/.env.example`'s `VERSION` comment tells the operator to pin the tag they downloaded. |
