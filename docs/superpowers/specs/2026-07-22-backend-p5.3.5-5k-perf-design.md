# Backend P5.3.5 — 5k-Attendee Performance — Design

**Status:** Approved by user 2026-07-22.

## 1. Background

P5.3.5 is the final sub-cycle of P5.3 (the last phase of the panel-rewrite initiative), originally scoped as "5k-attendee performance." Research (2026-07-22) found the panel's frontend is already well-bounded at scale by design, so — matching the pattern already seen in P5.3.4's i18n work — the original framing doesn't point at where the real gaps are:

- **Frontend**: `panel/`'s attendee list is genuinely server-paginated (`useAttendeesPage` always sends `page`/`per_page`, default 50, backend caps at 200), CSV import is chunked (500 rows/request, sequential, per-chunk retry), CSV export only ever covers the current page's selection (never an unbounded "export everything" fetch), the check-in station's recent-scans rail is server-capped at 50, and the workspace overview's stat tiles are pure `COUNT(*)` aggregates. No unbounded client-side fetch or render was found anywhere.
- **Backend**: `backend/internal/store/pg_store_attendees_page_integration_test.go`'s `TestSeeded5kAttendees_ScaleExitCriterion` already seeds a real 5,000-attendee event (via `pgxpool`/`CopyFrom` against a genuine Postgres, not the mocked store every other test uses) and asserts pagination/filter/search *correctness* at that scale. But: (a) it's gated behind `TEST_DATABASE_URL` and — per its own doc comment — never runs in CI, since no job provisions Postgres for it; (b) it asserts zero timing/duration bounds, only correctness (right rows, right counts, right ordering) — nothing would catch a query getting slow; (c) the list query's `ORDER BY last_name, first_name, id` has no matching index (`idx_attendees_event_updated` covers the `event_id` equality filter but not the sort), and this has never been measured against real data.

**Decision (user-approved):** scope P5.3.5 to the backend, where the actual gaps are. No frontend perf tooling — the frontend surfaces that touch attendee-scale data are already bounded by architecture, not by luck, so there's nothing there to instrument or guard.

## 2. Scope

**In scope:**
1. Wire `TestSeeded5kAttendees_ScaleExitCriterion` into CI so it actually runs on every PR, instead of silently skipping forever.
2. Extend that same test with real timing/duration assertions on the 4 query shapes it already exercises (unfiltered page, status filter, zone filter, search) — reusing its existing 5k-row seed, not re-seeding.
3. Investigate the list query's index/sort mismatch via `EXPLAIN ANALYZE` against the real seeded 5k-row dataset. Add a covering index (new migration) **only if the evidence shows it meaningfully helps** — this is a measure-first, fix-if-justified step, not a "add an index because it sounds right" step.

**Out of scope:**
- Any frontend perf tooling, budget, or instrumentation (per Section 1's finding — nothing there is unbounded or at measurable risk).
- The search filter's `ILIKE '%...%'` lacking a trigram/GIN index — flagged during research as a "grows further" concern, not something evidence points at mattering at 5,000 rows specifically. Not investigated in this sub-cycle; a candidate for a future note if attendee counts grow well past 5k in practice.
- Any new "export all attendees" feature — doesn't exist today (export is currently scoped to the current page's selection only), wasn't requested, and isn't a prerequisite for anything in this sub-cycle.
- Any change to `panel/`, `packages/ui`, `desktop/`, `mobile/`, or `web/`.

## 3. Wiring the test into CI

**Where:** `.github/workflows/validate.yml`'s `test-go` job (`Test Go (Coverage & Race Detection)`) — the job that already runs `go test -race -coverprofile=coverage.out -covermode=atomic ./...` across the whole backend, and is a required check (`needs: test-go` gates the downstream `report` job). The scale test lives inside `backend/internal/store`, within that same `./...` sweep — once `TEST_DATABASE_URL` is set and a real Postgres is reachable, the existing `go test` invocation picks it up automatically. No new job, no duplicated test invocation.

**How:** add two steps to `test-go`, before the "Test backend with coverage" step, mirroring `ci.yml`'s `e2e-panel` job's already-proven pattern exactly (same `docker compose up -d db` service, same readiness-poll shape, same credentials):
- Start Postgres: `docker compose up -d db`
- Wait for Postgres: poll `docker exec idento_db pg_isready -U idento -d idento_db` (same loop shape already used in `e2e-panel`)

Then set `TEST_DATABASE_URL: "postgres://idento:idento_password@localhost:5438/idento_db?sslmode=disable"` (identical connection string to `e2e-panel`'s own `DATABASE_URL`) as an env var on the "Test backend with coverage" step. The scale test calls `s.RunMigrations()` itself before seeding, so it works against a genuinely fresh, empty Postgres with no separate migration step needed in CI.

**Note on cost:** this test seeds 5,000 rows via `CopyFrom` and runs 6 real-Postgres queries — it adds real wall-clock time to `test-go` (a few seconds, not minutes, based on `CopyFrom`'s bulk-insert design), but this is a required, already-slow-ish job (full backend test suite with race detection); the addition is proportionate.

## 4. Timing assertions

Extend `TestSeeded5kAttendees_ScaleExitCriterion`'s existing subtests (don't duplicate the 5k-row seed — timing assertions wrap the same query calls the correctness subtests already make, or add sibling subtests using the same seeded `eventID`/`zoneID`) with wall-clock duration checks on:
- The unfiltered paginated query (`GetAttendeesPage`, no filters)
- The `status=checked_in` filtered query
- The zone-filtered query
- The search query

**The exact duration bound is not decided in this spec.** It will be derived empirically during implementation: run each query shape against the real 5k-row seeded dataset, record the actual observed wall-clock time, then set a bound with generous margin (e.g., several times the observed value, or a clean round number comfortably above it) — not a number chosen in advance without evidence. The goal is a regression guard against a query becoming meaningfully slower than it is today, not an arbitrary SLA.

## 5. Index investigation

Run `EXPLAIN ANALYZE` for the exact list query (`GetAttendeesPage`'s SQL, `ORDER BY a.last_name, a.first_name, a.id LIMIT $n OFFSET $n`) against the real 5,000-row seeded dataset from the scale test (reuse the same seed rather than hand-rolling a separate one-off dataset). Read the actual query plan:
- If Postgres is doing an expensive explicit sort step (not covered by an index) that meaningfully affects the query's cost/timing at this scale, add a new migration for a covering index (e.g. `(event_id, last_name, first_name, id)`) and re-run `EXPLAIN ANALYZE` to confirm the plan actually improves.
- If the planner already handles 5,000 rows cheaply (e.g. the sort is negligible relative to total query cost, or an index scan already serves it well enough), don't add a speculative index — document the `EXPLAIN ANALYZE` finding in the task's own report/ledger entry instead, so the reasoning is on record without adding unjustified schema surface.

This is deliberately evidence-first: the fix (or the decision not to fix) follows from what `EXPLAIN ANALYZE` actually shows, not from the index gap merely existing in the abstract.

## 6. Testing / validation plan

- `TEST_DATABASE_URL=... go test ./internal/store/ -run TestSeeded5kAttendees_ScaleExitCriterion -v` passes locally against the real docker-compose Postgres, including the new timing assertions.
- `cd backend && go test -race ./...` passes locally with `TEST_DATABASE_URL` set (confirms the test runs and passes as part of the full suite, not just in isolation).
- The updated `validate.yml` is valid YAML and, once pushed, the `test-go` job's own CI run shows the scale test actually executing (not skipped) with a passing result — verified on the real PR, not just locally.
- If an index migration is added: `go test ./internal/store/... ` (migration-integrity tests) plus a fresh `EXPLAIN ANALYZE` re-run confirming the plan improved.

## 7. Global constraints

- Branch: `backend/p5.3.5-5k-perf`, base = `main` `294a59f`.
- Scope: `backend/` and `.github/workflows/validate.yml` only. Do not touch `panel/`, `packages/ui`, `desktop/`, `mobile/`, `web/`.
- No speculative index — Section 5's evidence-first rule is binding.
- No fabricated timing bounds — Section 4's "derive from real measurement" rule is binding.
- Reuse `ci.yml`'s `e2e-panel` job's exact Postgres-provisioning pattern (service name, credentials, readiness-poll shape) for consistency — don't introduce a second, different way of standing up Postgres in CI.
