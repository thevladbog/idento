# Web P5.3.4 — EN/RU i18n Correctness Sweep — Design

**Status:** Approved by user 2026-07-22.

## 1. Background

P5.3.4 is the last sub-cycle of the P5.3 phase (the final phase of the panel-rewrite initiative), originally scoped as a "web/ EN/RU parity sweep." `web/` is the separate admin console SPA (distinct from `panel/`, the operator-facing app already rewritten with a dedicated key-parity test in P0–P4).

Research (2026-07-22) found the literal "parity" framing doesn't match what's actually wrong:

- `web/src/i18n.ts` holds all EN/RU strings inline inside a single `i18next` `.init({ resources: {...} })` call — 818 keys in the EN block, 818 in the RU block, and a direct diff of the two key sets is **already an exact match**. There is no missing-translation problem today.
- Roughly 600 of those 818 keys (~73%) are unreferenced anywhere in the codebase — a leftover from before P5.1 stripped `web/`'s pages down to just the admin console (events/badges/checkin/equipment/zones screens moved to `panel/`). These are inert (never rendered, never wrong), not a bug.
- The real, user-visible bugs are: **11 call sites reference a `t()` key that doesn't exist in EITHER language** (i18next silently renders the raw key string, identically broken in EN and RU — e.g. a user sees the literal text `plan` instead of "Plan"/"Тариф"), and **6 call sites hardcode English text that bypasses i18n entirely** (e.g. a literal `/mo` suffix, never translated to `/мес`).
- `web/` has a live, reachable language switcher (`language-toggle.tsx`, rendered on `Login.tsx` and every super-admin screen via `SuperAdminLayout.tsx`) — so this is a real, user-facing correctness bug, not spec-work for an unreachable feature.
- No automated test exists today that would have caught any of this in `web/` (unlike `panel/`, which has `keyParity.test.ts`).

**Decision (user-approved):** fix the real bugs only. Pruning the ~600 dead keys is deliberately **out of scope** — it's a separate, larger, more mechanical cleanup with its own risk (a key that looks unused could still be referenced via a dynamic template-string pattern, e.g. `` t(`tenantStatus_${status}`) ``) and doesn't fix anything currently broken. A future cleanup task can pick it up independently.

## 2. Scope

**In scope:**
- Fix all 11 phantom-key call sites (Section 3).
- Fix all 6 hardcoded-literal call sites (Section 3).
- Add a regression test that statically verifies every literal `t("key")` call site in `web/src` resolves to a real key in `i18n.ts`'s resources — the check that would have caught every bug in Section 3.

**Out of scope:**
- Pruning the ~600 unreferenced i18n keys.
- Restructuring `i18n.ts` into split `en.json`/`ru.json` files (matching `panel/`'s shape) — `web/`'s existing single-file-inline-resources structure is unchanged; only a minimal export is added for testability (Section 4).
- Any translation-quality review of the existing 818×2 keys beyond the specific bugs listed here.
- Any change to `panel/` or other apps.

## 3. The bugs and their fixes

All new/renamed keys go into both the EN and RU blocks of `web/src/i18n.ts`, alphabetically positioned near their nearest existing neighbor (matching the file's loose existing grouping-by-comment convention) unless a more specific location is obviously better (e.g. `trial`/`cancelled` belong next to the sibling `active`/`expired` keys they're rendered alongside).

### 3a. Phantom keys — rename call site to an existing orphaned key (4)

| Call site | Current (broken) | Fix | Existing key's EN / RU value |
|---|---|---|---|
| `AllUsers.tsx:106` | `t('createdAt')` | `t('created_at')` | "Created" / "Создано" |
| `OrganizationDetail.tsx:356` | `t('website')` | `t('organizationWebsite')` | "Website" / "Веб-сайт" |
| `OrganizationDetail.tsx:360` | `t('contactEmail')` | `t('organizationContact')` | "Contact Email" / "Контактный email" |

(Note: `created_at` and `created` are both valid "Created" column-header keys already in the file — `created_at` is chosen since it's the exact name the broken call site already used with only a naming-convention mismatch, `createdAt`→`created_at`, minimizing the diff. Not consolidated with `created` — that's a dead-key-adjacent cleanup, out of scope per Section 2.)

### 3b. Phantom keys — add a new key (7)

| Call site(s) | New key | EN | RU |
|---|---|---|---|
| `AuditLog.tsx:136` | `action` | "Action" | "Действие" |
| `Organizations.tsx:234`, `OrganizationDetail.tsx:382` | `plan` | "Plan" | "Тариф" |
| `OrganizationDetail.tsx:404` | `trial` | "Trial" | "Пробный период" |
| `OrganizationDetail.tsx:406` | `cancelled` | "Cancelled" | "Отменена" |
| `SubscriptionPlans.tsx:303` | `description` | "Description" | "Описание" |
| `SubscriptionPlans.tsx:238` | `features` | "Features" | "Возможности" |
| `SubscriptionPlans.tsx:173` | `failedToSavePlan` | "Failed to save plan" | "Не удалось сохранить план" |

(`trial`/`cancelled` are deliberately short, generic names — not `subscriptionStatusTrial` etc. — to match the naming convention of the sibling `active`/`expired` keys already used in the exact same `<Select>`. `failedToSavePlan` is deliberately NOT named bare `failedToSave`, since `failedToSaveSettings`/`failedToSaveAccessRules` establish a feature-suffixed convention for this error-message family.)

### 3c. Hardcoded literals bypassing i18n (6 call sites, 3 new keys)

| Call site(s) | Literal | New key | EN | RU |
|---|---|---|---|---|
| `SubscriptionPlans.tsx:219,222`, `OrganizationDetail.tsx:390` | `/mo` | `perMonth` | "/mo" | "/мес" |
| `SubscriptionPlans.tsx:222` | `/year` | `perYear` | "/year" | "/год" |
| `Organizations.tsx:258,282`, `OrganizationDetail.tsx:357,361` | `'N/A'` | `notAvailable` | "—" | "—" |

`notAvailable` renders an em dash, not literally "N/A"/"Н/Д" — matching the existing sibling convention already used elsewhere in `Organizations.tsx` (lines 292–295) for the identical "missing value" case in the same table. This makes the fix a genuine consistency improvement, not just an i18n wrapper around the status quo.

## 4. The regression test

**File:** `web/src/i18n-key-usage.test.ts` (new).

**Approach:**
1. `web/src/i18n.ts` currently defines `resources` inline as an argument to `.init({...})`. Extract it into a named export — `export const resources = { en: { translation: {...} }, ru: { translation: {...} } };` then `i18n.use(...).init({ ...otherOptions, resources });` — behavior-identical (same object, same shape, same init call), just addressable by a test without triggering `i18n.init()`'s side effects (language detection, React binding).
2. The test walks every `.ts`/`.tsx` file under `web/src` (excluding `i18n.ts` itself and test files) and extracts every literal `t("key")` / `t('key')` call argument via a regex scan of the file's source text — mirroring the text-scan approach `packages/ui/src/contrast.test.ts` already uses for `theme.css` in this same initiative. Dynamic calls (template strings, variables, e.g. `` t(`tenantStatus_${s}`) ``) are structurally unmatchable by a literal-string regex and are skipped — not a false negative, since there's no static key name to check; if a genuinely bad dynamic key ever ships, it's a live-testing/runtime concern, not this test's job.
3. For every extracted literal key, assert it exists in `resources.en.translation` (and, since key sets are already in exact sync, also assert it exists in `resources.ru.translation` — cheap and catches any future divergence as a side effect, without requiring a second dedicated parity test).

**Why not port `panel/`'s `keyParity.test.ts` as-is:** that test only diffs the EN key set against the RU key set. It would not have caught any of the 17 bugs in Section 3, all of which are "call site references a key that exists in neither language" — a different failure mode entirely. This test targets the actual bug class found.

## 5. Testing / validation plan

- `cd web && npm run test` — full suite green, plus the new test passing (and failing first against the pre-fix call sites, confirmed via TDD RED→GREEN during implementation).
- `cd web && npm run typecheck && npm run lint && npm run build` — all clean.
- Manual/live confirmation not required beyond the automated test — these are text-rendering bugs with no interaction-state or timing component, unlike `panel/`'s P5.3.3 a11y work.

## 6. Global constraints

- Branch: `web/p5.3.4-i18n-parity`, base = `main` `221e06b`.
- Scope: `web/src/i18n.ts` + the 5 named component files + the new test file only. Do not touch `panel/`, `packages/ui`, `desktop/`, `mobile/`, or `backend/`.
- All new/renamed key values are transcribed verbatim from Section 3 — no inventing alternate wording.
- Every fix must be independently verified against the actual current file content before editing (line numbers in this spec are a snapshot from 2026-07-22 research and may have drifted by implementation time — grep for the literal broken call, don't trust line numbers blindly).
