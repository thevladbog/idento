# On-Prem Phase 2 — P2.1 Bootstrap Flow — Design

- **Date:** 2026-07-11
- **Status:** approved by user, pending implementation plan
- **Depends on:** Phase 0/1 (merged) — `config.DeploymentMode` (`saas`|`onprem`, default `onprem`), `EnsureSeedData` (seeds the on-prem "Unlimited" plan and marks it default), `Store.ProvisionTenantWithAdmin` (P1.10's transactional tenant+subscription+admin-user+membership provisioning, currently only called from `Register`).
- **Roadmap item:** `docs/DUAL_DISTRIBUTION_REWORK.md` § Phase 2, P2.1.
- **Also fixes:** an item this design's exploration surfaced — `config.AdminEmail`/`AdminPassword` are already parsed (`backend/internal/config/config.go:25-26`, comment: "used in Phase 2") but nothing consumes them yet.

## Goal

On a fresh on-prem install (`DEPLOYMENT_MODE=onprem`, empty database), the backend automatically provisions one organization and one admin user from `IDENTO_ADMIN_EMAIL`/`IDENTO_ADMIN_PASSWORD`, so the operator reaches a working login with no manual SQL. The flow is permanently inert once any user exists in the database — it never runs again, on any later restart, even if the env vars are still set.

## Key decisions (confirmed with user)

1. **Reuse `ProvisionTenantWithAdmin` as-is.** It already does tenant + subscription (to whichever plan is `is_default`, which `EnsureSeedData` already sets to the on-prem "Unlimited" plan) + admin user + membership, transactionally. Bootstrap becomes a thin caller, not new persistence logic — zero new SQL, zero mode-branching inside the provisioning path itself.
2. **Gate: "any user exists" — not "any tenant exists."** A `SELECT EXISTS(SELECT 1 FROM users)` is the idempotency check. This is the correct fresh-install signal and is robust even if `users` ever gets a row through some path other than tenant provisioning.
3. **Missing/invalid admin env vars on a genuinely empty database is a fatal startup error**, not a silent skip or a degraded "running but unreachable" state. The operator gets one clear failure at boot (`log.Fatalf`), pointed at `INSTALL.md`, instead of a service that passes health checks but nobody can log into. Confirmed with the user over the alternative (start anyway, bootstrap lazily on next restart) — the recommended, fail-fast option was chosen.
4. **A new optional `IDENTO_ORG_NAME`** (default `"My Organization"`) names the bootstrap tenant. Everything else reuses existing config plumbing.
5. **Validation of `IDENTO_ADMIN_*` happens at bootstrap time, inside `onprem` mode, when the database is confirmed empty** — not in `config.Load()`, which runs before any database connection exists and cannot know whether bootstrap will even be needed. `saas` mode never touches these vars.
6. **No password policy invented for bootstrap.** `Register`'s validation is "non-empty" only; bootstrap matches that exactly rather than introducing an on-prem-only strength requirement unilaterally.

## Design

### New store method: `Store.HasAnyUsers(ctx) (bool, error)`

```go
func (s *PGStore) HasAnyUsers(ctx context.Context) (bool, error) {
	var exists bool
	err := s.db.QueryRow(ctx, `SELECT EXISTS(SELECT 1 FROM users)`).Scan(&exists)
	return exists, err
}
```

Added to the `Store` interface alongside the other top-level existence/count helpers.

### New function: `bootstrapOnPremAdmin` (package `main`, or a small new `backend/internal/bootstrap` package — implementer's call, matching whichever existing convention `main.go` already follows for similarly-sized startup steps)

Called from `main.go`, immediately after `EnsureSeedData`, **only when `cfg.DeploymentMode == config.ModeOnPrem`**:

```go
func bootstrapOnPremAdmin(ctx context.Context, s store.Store, cfg *config.Config) error {
	hasUsers, err := s.HasAnyUsers(ctx)
	if err != nil {
		return fmt.Errorf("check for existing users: %w", err)
	}
	if hasUsers {
		return nil // already bootstrapped (or a user exists through some other path) — inert
	}
	if cfg.AdminEmail == "" || cfg.AdminPassword == "" {
		return fmt.Errorf("first run with an empty database requires IDENTO_ADMIN_EMAIL and IDENTO_ADMIN_PASSWORD to be set — see INSTALL.md")
	}
	orgName := cfg.AdminOrgName
	if orgName == "" {
		orgName = "My Organization"
	}
	_, _, err = s.ProvisionTenantWithAdmin(ctx, orgName, cfg.AdminEmail, cfg.AdminPassword)
	if err != nil {
		return fmt.Errorf("bootstrap admin: %w", err)
	}
	log.Printf("bootstrap: created organization %q with admin %s", orgName, cfg.AdminEmail)
	return nil
}
```

`main.go` call site:

```go
if cfg.DeploymentMode == config.ModeOnPrem {
	if err := bootstrapOnPremAdmin(context.Background(), pgStore, cfg); err != nil {
		log.Fatalf("Bootstrap failed: %v", err)
	}
}
```

`saas` mode never calls this function — the env vars stay fully inert there, matching today's behavior.

### Config addition

`backend/internal/config/config.go`: add `AdminOrgName string`, read from `IDENTO_ORG_NAME`, no validation (empty is valid — the bootstrap function applies the `"My Organization"` default). No changes to `Load()`'s validation/fatal-error set — `AdminEmail`/`AdminPassword`/`AdminOrgName` stay unvalidated at config-load time, exactly as `AdminEmail`/`AdminPassword` already are today.

### Data flow

```
main.go startup (onprem mode only):
  RunMigrations → EnsureSeedData(seeds "Unlimited" plan, marks default)
    → bootstrapOnPremAdmin
        → HasAnyUsers?
            yes → no-op, continue startup
            no  → IDENTO_ADMIN_EMAIL/PASSWORD set?
                    no  → log.Fatalf (process exits, container restarts and hits the same fatal error until fixed)
                    yes → ProvisionTenantWithAdmin(orgName, email, password)
                            → tenant + subscription(default plan) + admin user + membership, one tx
                          → log success, continue startup
```

### Error handling

- `HasAnyUsers` DB error (connection lost, etc.) → fatal, same severity class as a migration failure — the process should not silently continue in an unknown state.
- Missing/empty admin env vars on empty DB → fatal, per decision #3.
- `ProvisionTenantWithAdmin` failure (e.g. DB constraint violation) → fatal, wrapped with context; this mirrors how `RunMigrations`/`EnsureSeedData` failures are already handled two lines above in `main.go`.
- No handling needed for "email already exists" — impossible on a confirmed-empty `users` table.

## Testing

- **Store (Go):** `HasAnyUsers` returns `false` on an empty table, `true` after any user insert (any tenant, any role).
- **Bootstrap function (Go, store-mocked or against a test DB):**
  - Empty DB + valid env vars + `onprem` → tenant, subscription (linked to the on-prem default "Unlimited" plan), admin user, and membership all created; returned error is `nil`.
  - Empty DB + missing/empty `IDENTO_ADMIN_EMAIL` or `IDENTO_ADMIN_PASSWORD` → non-nil error, no rows written.
  - Non-empty `users` table (pre-existing user from any source) + any env var state → no-op, `nil` error, no new tenant/user created — asserts idempotency across repeated calls (simulating restarts).
  - `IDENTO_ORG_NAME` unset → created tenant's name is `"My Organization"`; set → created tenant's name matches.
- **Not exercised in `saas` mode** — the function is simply never called there; no test needed to prove a no-op that isn't reachable.
- **Manual/live verification** (established pattern, before merge): fresh `docker compose` Postgres, `DEPLOYMENT_MODE=onprem` + admin env vars set, start the backend from empty, confirm `/api/instance` responds, log in via the web UI with the bootstrap credentials, confirm the org and its "Unlimited" plan/subscription are visible; restart the backend and confirm no second organization is created; then a second fresh-DB run with the admin env vars unset, confirming the process exits non-zero with the documented error message.

## Out of scope

- `IDENTO_ORG_NAME` is the only new config surface; no other bootstrap customization (e.g. picking a non-default plan, seeding sample data).
- No first-run web UI/wizard — env-var-driven only, per the approved distribution design.
- No password strength policy — matches `Register`'s existing (absence of) validation.
- No handling for multi-replica races on the same empty database — documented as an accepted risk (see below), not engineered around, since on-prem ships as a single-replica `docker-compose` deployment.
- P2.2 (frontend mode-awareness), P2.3 (distribution bundle/docs), P2.4 (client compatibility pass) are separate, later batches.

## Risks

| Risk | Mitigation |
|---|---|
| Two replicas racing `bootstrapOnPremAdmin` against the same empty database could both pass the `HasAnyUsers` check before either commits, creating two organizations with two admins instead of one | Accepted, documented: on-prem's supported topology is single-replica `docker-compose` (P2.3's deliverable). Not worth transaction-level locking for a topology this product doesn't ship. |
| A `saas`-mode database that later gets pointed at with `DEPLOYMENT_MODE=onprem` (misconfiguration) and already has users → `HasAnyUsers` correctly no-ops, but if it were instead empty (e.g. a wrong, fresh `DATABASE_URL`), bootstrap would silently create an on-prem-style admin against what the operator intended as a SaaS database | Out of scope for this item — mode/database-pointing mismatches are an operator error class the roadmap doesn't otherwise guard against today (no db-level mode tag exists). Noted here for awareness, not fixed. |
| Fatal-on-missing-vars means a `docker compose up` with a forgotten `.env` crash-loops instead of starting in a "waiting for admin setup" state | Accepted per user's explicit choice (decision #3) — the clear, immediate failure is preferable to a running-but-unusable service. `INSTALL.md` (P2.3) will document this explicitly so it reads as expected behavior, not a bug report. |
