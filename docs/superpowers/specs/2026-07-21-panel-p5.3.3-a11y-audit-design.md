# Panel P5.3.3 — WCAG-AA accessibility audit — Design

Third of five P5.3 sub-cycles (the final phase of the panel rewrite
initiative). Parent decomposition: the P5.3 scoping brainstorm's 5-way split
— P5.3.1 docs (done), P5.3.2 e2e (done), P5.3.3 a11y (this one), P5.3.4 EN/RU
parity for `web/`, P5.3.5 5k-attendee perf. Establishes accessibility tooling
from scratch (none exists today) and fixes every real WCAG 2.2 AA violation
the tooling surfaces on a representative set of `panel/` screens.

## 1. Scope

**`panel/` only** (user-approved 2026-07-21). `AGENTS.md`/`panel/AGENTS.md`
freeze `web/` for feature work (critical fixes only, no a11y carve-out); this
cycle does not touch `web/`. Level: **WCAG 2.2 AA**.

Explicitly out of scope: `web/` console, `desktop/` kiosk (has its own
already-a11y-conscious `@idento/ui/kiosk` subtree, built separately in K1),
`mobile/`. AAA-level criteria are noted where trivially satisfied as a side
effect but are not a goal.

## 2. Starting posture (confirmed during brainstorm — not a blank slate)

`@idento/ui` already has an above-average baseline: 14 of 23 panel-facing
primitives sit directly on a Radix primitive (Select, Dialog, Sheet,
Dropdown, Checkbox, RadioGroup, Switch, Tooltip, Avatar, Separator — strong
built-in focus-trap/ARIA/keyboard-nav for free); the hand-rolled remainder
already shows real a11y engineering (`StatusPill`'s `sr-only` text +
`aria-hidden` decoratives, `Progress`'s manual `role="progressbar"` +
`aria-value*`, `DatePicker`'s locale-aware ARIA labels via
`react-day-picker/locale`, `BadgeCanvas`'s keyboard-nudge with an explicit
WCAG 1.4.1-adjacent citation, tested in `BadgeCanvas.test.tsx`). The panel's
own Vitest suite already queries via `getByRole`/`getByLabelText` in 66% of
test files (55/83) — role/label-based queries are the dominant idiom, not
the exception, so most existing tests already require accessible markup to
pass.

The known, concrete gaps found during scoping (§3) are the audit's starting
fix list; the automated tooling established in §4 will surface anything else.

## 3. Known findings (confirmed during brainstorm, to be fixed regardless of what automated tooling later reports)

### 3a. Color contrast failures (WCAG 1.4.3, 4.5:1 normal text) — empirically computed against the real token hex values in `packages/ui/src/theme.css`

| Pair | Ratio | Threshold |
|---|---|---|
| Light `--primary` (`#00935e`) vs `--primary-foreground` (`#ffffff`) — default Button | 3.94:1 | Fails 4.5:1 |
| Dark `--primary` (`#36a16f`) vs `--primary-foreground` (`#ffffff`) — default Button, dark mode | 3.24:1 | Fails 4.5:1 |
| Light `--warning` (`#d97706`) vs `--warning-foreground` (`#ffffff`) | 3.19:1 | Fails 4.5:1 |

These sit on the **default Button variant** — the single most-reused
interactive primitive in the app — so the blast radius is every primary
action button, not an edge case.

**Fix approach (user-approved: fix token values now, not just document):**
darken each failing background just enough to clear 4.5:1 while preserving
white foreground text (keeps the existing brand-green/amber hue family and
button visual language — this is a token-value correction, not a color
redesign). Minimally-darkened candidates already computed (WCAG relative
luminance formula, iterative darkening in 1% steps until ≥4.5:1):

- Light `--primary`: `#00935e` → **`#008756`** (4.57:1)
- Dark `--primary`: `#36a16f` → **`#2c845b`** (4.61:1)
- Light `--warning`: `#d97706` → **`#b26205`** (4.52:1)

**Real constraint found, not yet resolved — binding on implementation:**
`--primary-hover` (light `#00714a`, dark `#2c8a5e`) is supposed to read as
visibly darker than `--primary` on hover. The dark-mode darkened candidate
above (`#2c845b`) is nearly indistinguishable from the EXISTING dark
`--primary-hover` (`#2c8a5e`) — darkening primary without also touching
hover would collapse that affordance. **Final hex values must be chosen
together** (primary + primary-hover, both themes) so that after the fix:
(a) every primary/primary-foreground and warning/warning-foreground pair
clears 4.5:1, AND (b) `--primary-hover`'s relative luminance stays
meaningfully below the new `--primary`'s (preserve or widen the current gap,
don't let it collapse toward zero). Use the same iterative-darkening method
above; the candidate hexes here are a starting point, not the mandated final
values — verified by the programmatic contrast test in §3b, not by eye.

Also add a small safety margin above the bare 4.5:1 minimum (target ≥4.55:1
in the final values) to absorb any rounding-method difference between this
spec's manual calculation and whatever axe-core/the eventual contrast test
computes internally.

### 3b. New: a programmatic, exhaustive contrast-checking test (not just the 3 known pairs)

The 3 failures above were found by manually checking a handful of pairs the
scoping research picked — not an exhaustive sweep. Add a new test (extending
`packages/ui/src/theme.test.ts` or a new sibling file) that: parses every
`--foreground`/`--background`-style semantic pair actually defined in
`theme.css` (background/foreground pairs: `background`/`foreground`,
`card`/`card-foreground`, `popover`/`popover-foreground`,
`primary`/`primary-foreground`, `secondary`/`secondary-foreground`,
`muted`/`muted-foreground`, `accent`/`accent-foreground`,
`destructive`/`destructive-foreground`, `success`/`success-foreground`,
`warning`/`warning-foreground`, `info`/`info-foreground`), computes the WCAG
contrast ratio for each pair in BOTH `:root` and `.dark`, and asserts ≥4.5:1
for every one. This is the actual regression guard — it must fail on the 3
known-bad pairs before the fix and pass after, and it now protects every
OTHER pair (already passing) from silently regressing later.

### 3c. `VerdictCard` has no live-region announcement (WCAG 4.1.3, status messages)

`panel/src/features/checkin/VerdictCard.tsx`'s root `<div
data-testid="checkin-verdict-card" data-verdict={verdict}>` (the check-in
station's primary result surface — "Allowed" / "Already checked in" /
"Access denied") has no `role`/`aria-live`. A screen-reader-using operator
gets no automatic announcement when a scan resolves — for a time-critical,
high-frequency operator flow, this is a real usability gap, not a
theoretical one. The exact same pattern already exists correctly elsewhere
in this codebase: `packages/ui/src/kiosk/verdict-screen.tsx`'s root
`<section role="status" data-verdict={verdict} ...>` — `role="status"` alone
is sufficient (implies `aria-live="polite"` per the ARIA spec; no separate
`aria-live` attribute needed). Fix: add `role="status"` to `VerdictCard`'s
verdict-state root div, mirroring the kiosk component exactly. (The
`resolving`-state div, shown above it in the component, stays as-is — a
"resolving" transient loading state is not itself the content that needs
announcing; the settled verdict is.)

### 3d. `Skeleton` has no loading-state semantics (WCAG 4.1.3)

`packages/ui/src/components/skeleton.tsx` (currently 5 lines, a bare
`animate-pulse` div) has no `role="status"`/`aria-busy`. Fix: add
`role="status"` + a visually-hidden accessible name (e.g. `aria-label`
sourced from a caller-overridable prop, defaulting to a generic "Loading"
string — this component has no i18n dependency today, matching
`NumberInput`'s established `@idento/ui`-carries-no-i18n-dependency
convention, so default to an English string overridable via prop, not a
hardcoded translation key).

### 3e. Verified-compliant, not a gap — no code change

- **Badge canvas resize handles** (`BadgeCanvas.tsx`'s four corner
  `ResizeHandle` divs, `onPointerDown`-only) — WCAG 2.5.7 (dragging
  movements, new in 2.2) requires "a means to accomplish the function using
  a single pointer without a dragging movement," not that the SAME control
  support both. `PropertiesPane.tsx`'s X/Y/Width/Height `NumberField` inputs
  already provide exactly that alternative, routed through the identical
  `clampSize`/`clampPosition` logic the pointer-drag path uses. This
  satisfies 2.5.7's letter and spirit today — audit confirms compliance,
  does not add new keyboard-drag-handle interaction code (would be
  gold-plating beyond what the criterion requires).
- **`Dialog`'s `hideClose` prop** (used once, in the CSV import wizard's
  step 3 — an in-flight, uninterruptible operation) is a deliberate,
  already-PR-reviewed product decision, documented in the component's own
  code comment. Not a gap to silently "fix" — confirming it stays
  intentional is itself part of the audit (§6).

## 4. Tooling — three layers, all established this cycle

Each is cheap to add given existing infra (P5.3.2 already stood up
Playwright; the Vitest suite already uses Testing-Library idioms
pervasively) and each catches a different class of problem:

1. **`eslint-plugin-jsx-a11y`** — added to `panel/eslint.config.js`'s
   existing `tseslint.config(...)` flat-config array (same shape as the
   already-present `no-restricted-syntax` native-control ban). Static,
   source-level: catches missing `alt`, missing form labels, redundant
   roles, invalid ARIA attribute usage — in every PR via the existing lint
   gate (`npm run lint -w panel`, already CI-blocking). Cannot see runtime
   contrast, real layout, or Radix's actual rendered ARIA output.
2. **`vitest-axe`** (or `jest-axe` — pick whichever has cleaner Vitest 4.x
   compatibility at implementation time) — wired into `panel/vitest.config.ts`'s
   existing `jsdom` environment. Adds `toHaveNoViolations()`-style assertions
   layered onto EXISTING component tests (no new render-setup needed, since
   the bulk of `*.test.tsx` files already render via Testing Library).
   Catches DOM-shape a11y issues (missing labels, invalid nesting, ARIA
   misuse) at the component level, still jsdom-limited — no real contrast or
   layout.
3. **`@axe-core/playwright`** — added to the existing `panel/e2e/` suite
   (P5.3.2's infra). The only layer that runs against a REAL rendered
   Chromium page: real computed contrast, real focus order, Radix's actual
   runtime ARIA. New e2e spec `panel/e2e/a11y.spec.ts` reuses the exact
   `seedCheckinEvent`-style auth-bypass pattern (`page.addInitScript` +
   token injection) already established in `checkin.spec.ts` to reach
   authenticated screens without going through the login UI. Runs
   `AxeBuilder(page).withTags(["wcag2a", "wcag2aa", "wcag21aa", "wcag22aa"]).analyze()`
   and asserts zero violations per screen. Inherits `e2e-panel`'s existing
   soft (non-blocking) CI gate for free — no new CI-gating decision needed
   for this cycle.

### 4a. Representative screen set for the axe-core sweep (not a full route crawl)

Testing every registered route with realistic seeded data would be its own
multi-week initiative. Instead, pick screens that between them exercise
every major interactive pattern once — since the pattern (e.g. "a Radix
Dialog") is what's actually being verified, not each individual consumer of
it:

- `/login` — form, unauthenticated entry point.
- `/` (home/dashboard) — landing surface, first authenticated screen.
- `/events/$eventId` (workspace overview) — the readiness rail, primary nav.
- `/events/$eventId/badge` (badge editor) — canvas, keyboard-nudge, complex
  multi-pane layout, dialogs (GuardDialog).
- `/events/$eventId/checkin` (check-in station, reusing P5.3.2's seed
  helper) — live-region verdict (post-3c fix), scan input, manual search.
- `/events/$eventId/settings` (a forms-heavy settings screen) — validation,
  error states, multiple form controls.
- `/events/$eventId/attendees` (the attendees table) — data table, bulk
  actions, dropdown menus.

Each screen reached via the same seeded-event + token-injection setup
pattern already proven in `checkin.spec.ts`.

## 5. WCAG 2.2 AA checklist items axe-core may not catch — spot-checked manually as part of this cycle

Automated tooling (axe-core, jsx-a11y) has known blind spots. These are
checked by hand against the representative screen set in §4a, using the
`@idento/ui` primitive-level implementation as the source of truth (fixing
at the primitive level fixes every consumer at once):

- **2.5.8 Target size (new in 2.2, 24×24px minimum)** — spot-check
  `@idento/ui`'s smallest interactive elements (icon-only buttons, the badge
  editor's toolbar, `StatusPill`'s dismiss affordances if any) against the
  24×24 CSS px floor.
- **2.4.7 Focus visible** — confirm `@idento/ui` has a global
  `:focus-visible` treatment and that it clears 3:1 contrast against every
  background it can land on (light + dark).
- **2.4.11 Focus not obscured (new in 2.2)** — check `WorkspaceRail`'s
  sticky/fixed elements don't hide a keyboard-focused element behind them.
- **2.3.3 / prefers-reduced-motion** — confirm `animate-pulse`
  (`Skeleton`)/`animate-spin` (`Loader2` usages) and Dialog/Sheet's
  slide-in transitions respect `prefers-reduced-motion: reduce` (Tailwind's
  `motion-reduce:` variant, or a global CSS media-query override in
  `theme.css`).
- **3.1.1 Page language** — confirm `panel/index.html`'s `<html lang>`
  matches the active i18n locale and updates on a language switch (not just
  hardcoded once at build time).
- **1.4.4 Resize text / RU string-length risk** — brainstorm-time research
  found real EN→RU length ratios up to 4.33x on short action labels (e.g.
  `badgeAddBox`: "Box"→"Прямоугольник"), mitigated today by fluid
  (non-fixed-width) button layouts. Take 2-3 representative RU-locale
  screenshots (the badge editor's Elements "+Add" dropdown, a settings rail)
  to confirm no clipping — a spot-check given the codebase-wide `w-[Npx]`
  audit already done at scoping time found no fixed-width-plus-truncate
  combination on static translated labels.

## 6. Testing (of this cycle's own deliverable)

- The new `theme.test.ts` contrast assertions (§3b) must fail before the
  token fix and pass after — this is the actual proof the fix is complete
  and correct, not eyeballing hex values.
- `panel/e2e/a11y.spec.ts` (axe-core sweep, §4a) run against a live local
  stack (same dev-mode CI environment P5.3.2 established:
  `docker compose up -d db`, `go run` backend with the onprem bootstrap env
  vars, panel dev server) — zero violations across all 7 representative
  screens is the pass bar.
- Existing `vitest-axe` assertions added to a representative sample of
  EXISTING component tests (not every one — pick tests already covering
  Dialog/Select/Checkbox/RadioGroup/DatePicker consumers plus `VerdictCard`
  and `Skeleton` directly, since those are this cycle's own fix targets).
- Full regression guard: `npm run test -w panel` (Vitest, including the new
  jsx-a11y-satisfying markup), `npm run lint -w panel` (new
  `eslint-plugin-jsx-a11y` rules must not fire on existing code — if they
  do, fix the flagged code, don't suppress the rule), `npm run typecheck -w
  panel`, `npm run build -w panel`, backend `go test ./...` (unaffected,
  regression guard only).
