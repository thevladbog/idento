> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Port the codebase's only working Cyrillic ZPL pipeline (web's client-side generator) into `panel/`, wire the badge editor's ZPL-preview and test-print, integrate event fonts with real Cyrillic-coverage flags, and unlock the P2-deferred print affordances (drawer Reprint, bulk Print badges) through the local print agent ‚Äî per spec ¬ß7 as corrected by the reconciliations below.

## Plan-time reconciliation (spec ¬ß7 vs verified code, 2026-07-16 ‚Äî these override spec prose; flagged items are spec corrections)

1. **[SPEC CORRECTION ‚Äî ¬ß7.1] Both preview tabs derive from the CLIENT generator, not `POST /badge-zpl`.** The server generator (`backend/internal/zpl`) is the Latin-only sibling used by the desktop/web CHECK-IN kiosks (`desktop/src/pages/CheckinEvent.tsx:323`, `web/src/pages/CheckinFullscreen.tsx:215`) ‚Äî it has zero raster capability. The only pipeline that prints Cyrillic/custom fonts today is web's client-side `web/src/utils/zpl.ts` + `zpl-image-text.ts` (pure TS, zero deps, browser Canvas). Panel PORTS it; "the truth" = the exact bytes panel will send to the printer = the ported generator's output. The kiosk server path is untouched. The spec's `badge-preview` PNG endpoint is dropped (nothing exists to reuse ‚Äî the spec's own fallback clause governs).
2. **[SPEC CORRECTION ‚Äî ¬ß7.2] "Fonts on this printer" cannot be live.** `GET /printers/{name}/fonts` (agent/main.go:713-795) returns STATIC hardcoded lists; the real font query is printed on a physical label by the printer, never returned over the network. The preview modal's font panel shows: built-in ZPL fonts (static list, known NO Cyrillic ‚Äî flagged as such) and the event's uploaded fonts with REAL coverage flags (#3). The board's live printer-font list is dropped as designed-but-unimplementable.
3. **Coverage flags need the ONE new dependency:** `opentype.js` parses the uploaded font bytes (`GET /api/fonts/{id}/file`) client-side; coverage = every char of the sample set `–ê–Ø–Å–ñ–©—ã—ë—è` maps to a nonzero glyph index. web NEVER checked coverage ‚Äî its raster path silently draws browser-fallback glyphs into the printed bitmap (wrong badge, no signal); the flag exists to prevent exactly that.
4. **Agent connectivity (verified):** direct browser‚Üíagent, NO backend proxy, NO token for browsers (Origin-allowlist path only ‚Äî `agent/internal/httpauth`). web hardcodes `http://localhost:12345`; panel adds `window.__ENV__.AGENT_URL` (default `http://localhost:12345`) via `http.ts`-style helper + `public/env.js` + `env.js.template`. **Deployment note (docs, Task 11):** a deployed panel origin must be added to the agent's `allowed_origins`/`AGENT_ALLOWED_ORIGINS`; dev Vite ports 5173/5174 are in the agent's default allowlist.
5. **Agent `/print` is a transport-level ack** (`{"status":"printed"}` after `SendRaw` ‚Äî main.go:610); no job-status endpoint exists. Success copy = "sent to printer".
6. **[SPEC ADAPTATION] `printed_count` has NO write path anywhere** (no handler field, no endpoint, no client bump ‚Äî the table's Printed pill is decorative today). Task 7 adds the minimal `POST /api/attendees/{attendee_id}/printed` increment endpoint so the pill becomes real after panel prints. The spec's "server-side print journal is out of scope" clause targets audit/dedupe journals, not this existing counter; flagged for the user regardless.
7. **Parity-first port:** the ported generator preserves web's exact behavior INCLUDING its limitations ‚Äî raster-path text (Cyrillic/customFont) silently drops `rotation`/`valign`/`^FB` wrap/`maxLines` and renders one unwrapped line; monochrome threshold 127, no dithering; uncompressed `^GFA` hex. Fixing those is a functional change deferred to backlog; the port must not silently "improve" output.
8. **Escaping convention:** web's (`\`-escape of `\ ^ ~`, no `^FH`) ‚Äî diverges from backend's hex+`^FH`; ported verbatim because it is what prints correctly today. Documented in code.
9. **[DELIBERATE FIX vs web] Font readiness is AWAITED before generation.** web fires `loadEventFonts` un-awaited ‚Äî printing before fonts load silently rasterizes fallback glyphs. Panel's generation path awaits the fonts-ready state (Task 2's hook exposes it); test-pinned.
10. **jsdom has no canvas** (verified: no polyfill, `getContext` unavailable). The generator core takes an injectable `rasterizeText` function; pure helpers (`convertToMonochrome`, `bitmapToZPLHex` ‚Äî plain math on arrays) are fully unit-tested with synthetic bitmaps; the real canvas rasterizer is a thin browser-only module with structural tests only.
11. **`customFont` becomes a first-class typed field** on panel's `BadgeElement` (today it survives only as an extras-passthrough). Typed-field promotion deliberately changes serialize semantics for this ONE key (typed fields overwrite the original ‚Äî customFont edits now propagate; test-pinned). The Properties font selector gains an "Event fonts" group that sets `customFont` (raster path); picking a native ZPL code clears it.
12. **QR/barcode stay native** (`^BQN`/`^BCN` ‚Äî never rasterized, per board and web parity). The Rendered tab approximates them (existing `qrcode` lib for QR; striped block for barcode) with per-element-class honesty captions: raster-text elements show the TRUE print bitmaps; native elements are labeled approximations.
13. **Selection-scoped truth for generation inputs:** the editor's preview/test-print generate from `serializeTemplateDoc(state.doc, originalRawRef.current)` (the live doc incl. extras); the drawer/bulk flows generate from the SERVER's raw template (`GET /badge-template` ‚Üí `template` object as-is) + `attendeeToPreviewData(attendee)` ‚Äî matching `badge_zpl.go`'s data map exactly.
14. **Agent spec/code drift found and left alone** (undocumented `/print-pdf`, `printers/{name}/fonts` response-shape drift, undocumented auth, web's `testScanner` 404s) ‚Äî out of P3.2 scope; filed as a follow-up chip by the controller.
