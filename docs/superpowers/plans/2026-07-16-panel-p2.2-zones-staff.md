## Reconciliation decisions (spec/board vs backend verified on main, 2026-07-16)

1. **Zone `color` does not exist on the model** (`EventZone` = id, event_id, name, zone_type, order_index, open_time, close_time, is_registration_zone, requires_registration, is_active, settings, timestamps — `backend/openapi.yaml` /api/events/{event_id}/zones POST). `settings` is a writable free-form object (`additionalProperties: true`) → persist the swatch as a palette KEY in `settings.color` (`"green" | "amber" | "blue" | "slate"`), mapped to token classes in code. Zones created elsewhere (old web/) have no `settings.color` → deterministic fallback from the zone id (decorative only, stable across renders).
2. **"Entrance zone" ≡ `is_registration_zone: true`.** No other entrance concept exists. Subtitle shown for such zones; the header caption keeps the board's "entrance zone" vocabulary.
3. **Access-type text:** rules are default-allow when none exist (`evaluateZoneAccessRules`, `pg_store_zones.go:655`) → `access_rules_count === 0` ⇒ "All attendees"; `> 0` ⇒ "By rule". The board's third state "Manual list only" is NOT a modeled zone mode (individual `attendee_zone_access` overrides coexist with rules on any zone) → dropped; manual membership stays managed from the attendee drawer chips (spec §7 already says so).
4. **People count per zone: no data source** (`with_stats` returns `total_checkins`, `today_checkins`, `assigned_staff`, `access_rules_count` — none is a population count) → omitted per the spec's own honesty rule. Same for the rule builder's **live match count** (no count-by-category capability anywhere) → omitted.
5. **Rule shape is category-only:** `ZoneAccessRule` = `{category, allowed, time_from, time_to}`. The sentence UI's field selector is a static "Category" chip; each OR clause = one `{category, allowed: true}` rule. The value control is a **free-text input** (no distinct-categories endpoint; deriving values would require paging all attendees — rejected at 5k scale).
6. **`PUT /api/zones/{zone_id}/access-rules` is delete-and-replace** (returns only `{message}`). Rules the sentence UI can't represent (`allowed: false` or time-windowed, creatable via old web/) must be rendered read-only in the expanded row and passed through VERBATIM in the PUT array — saving must never silently destroy them.
7. **`PUT /api/zones/{id}` is a full replace — omitted fields are blanked** (documented in openapi.yaml). Every zone edit sends the complete merged body built from the currently-fetched zone.
8. **No backend guard on zone delete** (bare `DELETE FROM event_zones`; `attendees.registration_zone_id` FK has no cascade, so deleting a registration zone that attendees reference can fail server-side) → typed-confirm always (spec), dialog stays open and surfaces the server error honestly.
9. **Staff = `User[]`** (`GET /api/events/{event_id}/staff` returns full users: id, email, role, has_qr_token, qr_token_created_at, …). **No `name`, no `station` field** → cards show email + role subtitle; the board's station subtitle is omitted (never fabricate).
10. **"Signed in / Not yet" dot: NO data source** — verified definitively (no sessions table, no last_login/last_seen anywhere in migrations or handlers) → omitted per the spec's own guard.
11. **The QR token string is retrievable ONLY from `POST /api/users/{id}/qr-token`** (admin-only; response `{qr_token, user_id, email}`); the staff list exposes only `has_qr_token` + `qr_token_created_at`. Token format `QR_<uuid>_<hex>`; login side is `POST /auth/login-qr {qr_token}`. Consequences: a QR can be rendered/printed only right after (re)generation; "Print card" for an already-issued token = regenerate-with-confirm (invalidates the previously printed card — this generalizes the board's lost-card flow, so the dedicated "reported lost" state is dropped: no data source for it either).
12. **Tokens expire 30 days after generation and are per-USER (tenant-wide), not event-scoped** → board copy "expires when event ends" is replaced with honest copy ("Issued <date> · valid 30 days").
13. **"Print all QR cards" = regenerate every staff member's token** (sequential loop, per-item failure handling, one tier-1 confirm stating that all previously printed cards stop working), admin-only. In-session tokens are reused instead of re-regenerating.
14. **Zone-scope endpoints return ids only** (`StaffZoneAssignment` = id, user_id, zone_id, assigned_at, assigned_by) → client-side join with the event zones list (old web/ precedent).
15. **Role gates verified:** `getUsers` admin|manager (`users.go:29`), `createUser` admin-only + body `{email, password, role: staff|manager}` (`users.go:57-74`), `generateQRToken` admin-only (`users.go:129`), assign/unassign event staff admin|manager. Panel reads the role via `getCurrentTenant()?.role` (`panel/src/shared/api/session.ts`; `OrganizationPage.tsx:65` precedent). Gate affordances accordingly; still surface server 403s honestly. `assignStaffToZone`'s gate is unverified — implementer greps the handler and mirrors it.
16. **`field_schema` is `string[]`** (plain field names — `openapi.yaml` Event schema; PATCH body `schema.d.ts:2838`), not `{name,label,type}` objects → the Attendee-fields card manages names only. Backend matches CSV/custom-field names **case-insensitively** (`strings.ToLower`, `bulk_import.go`) → card validation must normalize the same way (P2.1 lesson).
17. **`POST /api/events/{event_id}/attendees/generate-codes` is backfill-only** (`if attendee.Code == ""`, `attendee_codes.go:33-42`; response `{status, updated_count, message}`) → ships as a plain non-destructive row (no typed confirm) with honest copy.
18. **Already fixed pre-merge in P2.1 — NOT in this phase's scope:** the import-wizard file-pick/encoding-change race and silent add-zone failures (both landed in PR #65's Codex round). Task 11 re-verifies the assign-zone attempt-vs-success readout (`assignFailedCount` landed in the CodeRabbit round) before touching it.
