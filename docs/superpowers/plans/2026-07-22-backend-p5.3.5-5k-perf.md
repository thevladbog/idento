# Backend P5.3.5 — 5k-Attendee Performance — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire the existing 5,000-attendee scale-correctness test into CI, add real timing assertions to it, and fix a genuine ~100x production performance bug (stale planner statistics after bulk insert) found via `EXPLAIN ANALYZE` during planning — both in the test's own baseline and in the real bulk-import code path.

**Architecture:** `backend/internal/store/pg_store_attendees_page_integration_test.go`'s `TestSeeded5kAttendees_ScaleExitCriterion` already seeds 5,000 real attendees against a genuine Postgres and checks pagination/filter/search correctness; it never runs in CI and asserts no timing. This plan (1) provisions Postgres for `validate.yml`'s `test-go` job so the test actually runs, (2) fixes the test's own baseline (its zone-filtered query was empirically found to run ~100x slower than necessary right after the bulk seed, due to stale planner statistics — not a missing index) and adds timing assertions once that's fixed, and (3) applies the same real-world fix to the production bulk-import code path so a real 5k-attendee event doesn't hit the identical stale-statistics window.

**Tech Stack:** Go, `pgx`/`pgxpool`, Postgres (via the existing `docker-compose.yml` `db` service), GitHub Actions.

## Global Constraints

- Branch `backend/p5.3.5-5k-perf`, base = `main` `294a59f`. `git branch --show-current` before every commit.
- **Scope: `backend/` and `.github/workflows/validate.yml` only.** Do not touch `panel/`, `packages/ui`, `desktop/`, `mobile/`, `web/`.
- **No new index, no new migration.** Real `EXPLAIN ANALYZE` evidence (gathered during planning, see the design spec's Section 5) showed the existing indexes are already sufficient — the bug was stale planner statistics after a bulk insert, fixed by calling `ANALYZE`, not by adding schema.
- **Timing bound: 100ms**, uniform across all 4 new timing assertions. Derived from real measurement during planning (observed steady-state times: ~5–7ms unfiltered, ~2ms status-filtered, ~2–4ms zone-filtered once statistics are fresh, ~11–18ms search) — 100ms gives roughly 5–20x margin over every observed value, while still being comfortably below the ~300ms the bug produced before the fix, so it would catch a real regression back to that bug class.
- Reuse `.github/workflows/ci.yml`'s `e2e-panel` job's exact Postgres-provisioning pattern (service name `db`, container name `idento_db`, credentials, readiness-poll shape) for `validate.yml`'s `test-go` job — don't invent a second way of standing up Postgres in CI.
- The new `Store.AnalyzeAttendeesTable` method's `fakeStore` implementation (`internal/handler/testsupport_test.go`) must be nil-safe (`return nil` when the test-hook field isn't set) — unlike every other `fakeStore` method, which intentionally panics on a nil field. This is deliberate: `ANALYZE` is a best-effort optimization, not correctness-critical, and none of the 4 existing tests that exercise `BulkCreateAttendees` should need to change just to keep compiling.

// AnalyzeAttendeesTable runs ANALYZE on the attendees table -- see the
// Store interface doc comment for why this exists. ANALYZE has no
// per-event scoping (it updates table-wide planner statistics), so this

func (s *PGStore) AnalyzeAttendeesTable(ctx context.Context) error {
