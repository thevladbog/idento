# P3.1 — Badge editor & save model — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the badge template's dedicated versioned storage (migration + GET/PUT with a real 409), and the three-pane editor at `/events/$eventId/badge` — elements pane, canvas with drag/resize, properties inspector, the four-state save model with conflict banner, the dirty guard, and attendee-backed preview — per spec §3–§6/§8.

**Architecture:** Backend-first (Tasks 1–3: migration/store → endpoints → fallback reads), then panel data layer (Task 4), then the editor from the inside out (Task 5 document model → Task 6 page/route/unlocks → Tasks 7–9 panes/canvas → Task 10 save model → Task 11 dirty guard → Task 12 preview), then the final sweep (Task 13). The editor's document state is a plain `useReducer` model — no state library exists in this repo and none is added.

**Tech Stack:** Go/Echo + kin-openapi contract harness; React 19, TanStack Router (`useBlocker`) + Query, `$api`, MSW, `@idento/ui`. **No new dependencies** (drag/resize is hand-rolled pointer events on a fixed artboard — YAGNI over dnd-kit).

**Spec:** `docs/superpowers/specs/2026-07-16-panel-p3-badge-editor-design.md` §3–§6, §8–§9.

## Plan-time reconciliation (spec vs code, verified 2026-07-16 — these facts override spec prose where they conflict; flagged items are spec corrections)

1. **Template format ground truth** (`backend/internal/zpl/zpl.go:14-37`): `Config {width_mm float64, height_mm float64, dpi int}` (dpi NOT enforced to 203/300; ≤0 falls back to 203) + `elements: BadgeElement[]` with fields `id, type, x, y, width?, height?, fontSize?, text?, source?, align?, valign?, rotation?, fontFamily?, bold?, maxLines?`. Types: `text | qrcode | barcode | line | box` (unknown types are SILENTLY DROPPED at generation — `zpl.go:320`). `x/y/width/height` are **mm**; `fontSize` is **points**. `bold` is a dead field (never read) — the editor does not expose it.
2. **Binding is the `source` field, not inline tokens.** `source` (attendee field name, e.g. `first_name`) replaces the WHOLE element text when it resolves non-empty; `text` is the static fallback (`zpl.go:172-177`). No in-string interpolation exists. The board's `{first_name}` chips are a UI display convention for `source` — the editor renders `{source}` visually but stores the plain field name. Bindable names: `id, first_name, last_name, email, company, position, code` + the event's `field_schema` entries (`badge_zpl.go attendeeToData`).
3. **`ParseBadgeTemplate` validates structure only** (JSON type mismatches — e.g. `x` as string, `elements` not an array). No enum/required/range checks. The PUT's 400 branch covers exactly that; the editor's own UI validation is the real guard.
4. **PUT stores the raw JSON verbatim — never round-trip through Go structs.** The shipped web editor writes a SUPERSET format (`customFont` field, `web/src/utils/zpl.ts:28`) that the Go struct doesn't carry; `json.Unmarshal` drops unknown keys, so re-marshaling the parsed struct would silently destroy web's data. Validation parses a copy; the column stores the request's `template` bytes as-is.
5. **[SPEC CORRECTION — §2/§7.1] The truetype-raster Cyrillic path is NOT in `backend/internal/zpl`** — it lives in the WEB CLIENT (`web/src/utils/zpl.ts:107`: non-Latin text or custom fonts render to an image client-side). Backend `badge_zpl.go` emits native-ZPL text only (`^A`+`^FD`, `^CI28`). P3.1 touches NO print path; the raster/fonts story belongs entirely to P3.2 (its plan will decide where the raster lives). The editor canvas is an editing aid — no fidelity claims.
6. **[SPEC CORRECTION — §4 fonts] Event fonts and Cyrillic-coverage flags are OUT of P3.1.** Uploaded event fonts are completely disconnected from ZPL generation (no font-store call anywhere in `zpl.go`/`badge_zpl.go`); no cmap-parsing capability exists anywhere in the repo (would need a new dependency). Per the spec's own honesty clause, the P3.1 font selector offers ONLY the ZPL native fonts the format supports — codes `0` (scalable) and `A/B/C/D/E` bitmap sizes (`sanitizeZPLFont`, `zpl.go:97-132`; the shipped web editor offers exactly `0|A|B|D|E`) — with size-hint labels and NO coverage flags. Event fonts + coverage checks move wholesale to P3.2's font flow.
7. **[SPEC CORRECTION — §8] `readiness.go` DOES need a backend change.** It reads `event.CustomFields["badgeTemplate"]` directly (`readiness.go:56-84`); since the new PUT never writes the legacy key, readiness would go permanently stale for post-P3.1 templates. Fix: column-first read with legacy fallback — the same treatment `badge_zpl.go` gets (spec §3.1 already mandates the latter).
8. **Migration copies only object-typed legacy values** (`jsonb_typeof(custom_fields->'badgeTemplate') = 'object'`). String-typed values are a shipped inconsistency (readiness says done, `POST /badge-zpl` 400s on them via a JSON double-encode artifact) — copying them would re-break the new endpoints; they're left in the legacy key untouched (fallback reads still see them exactly as today; nothing regresses).
9. **No optimistic-concurrency precedent exists in the backend** — the versioned PUT is a new pattern: `UPDATE events SET badge_template=$1, badge_template_version=badge_template_version+1, updated_at=now() WHERE id=$2 AND badge_template_version=$3` + rows-affected check → 0 rows = 409 with the current version re-read.
10. **`useBlocker` (@tanstack/react-router 1.170.18)**: `useBlocker({shouldBlockFn, enableBeforeUnload, withResolver: true})` → `{status: 'blocked'|'idle', proceed, reset, next}`. ONE hook covers both the router block and tab-close (`enableBeforeUnload` accepts a boolean-returning fn). Escape is a separate keydown handler. No `beforeunload` usage exists in panel yet.
11. **No drag/resize/canvas/Select/Slider/NumberInput anywhere** (panel, packages/ui, deps). Drag/resize = hand-rolled pointer events (`setPointerCapture`); numeric inputs = `<input type="number">` with `Input`'s classes; dropdown-selects = native `<select>` styled with token classes (a11y-first, no new Radix dep).
12. **Fonts UI in panel (`FontsCard.tsx`) is untouched by P3.1** (its inline query key stays local — nothing here consumes fonts anymore per #6).
13. **`badgeZpl` op already exists** (`POST /api/events/{id}/badge-zpl` body `{attendee_id}` → `{zpl}`) — P3.1 does not call it (P3.2's preview does); listed to prevent an implementer from "helpfully" wiring it early.
14. Board 4a's "Zone strip" element = a bound TEXT element (there is no zone attendee field in `attendeeToData`; the board sample binds a category-like custom field). The add-menu offers the format's real types (text/qrcode/barcode/line/box) — nothing invented.
